汇编层面理解函数调用

发表于2026-06-15|更新于2026-06-16|函数调用原理
|浏览量:

下面从汇编指令底层动作 + 栈内存结构图两个维度，把函数调用的完整过程讲透。先从规整的 32 位 x86 入门理解栈帧原理，再过渡到你当前常用的 64 位 x64 调用约定，全部配对应内存布局图。

核心前提：栈内存从高地址向低地址生长，push 压栈时栈指针减小，pop 出栈时栈指针增大。EBP/RBP 指向栈帧底部（固定不动），ESP/RSP 指向栈顶（实时变化）。

函数调用的完整生命周期（通用）

任何架构的函数调用，都严格遵循 6 个阶段，对应不同的汇编动作：

  1. 传参阶段：调用者按约定把参数放到寄存器或栈上
  2. call 指令：自动把返回地址压栈，然后跳转到函数入口
  3. 建立栈帧：保存旧栈底，初始化当前函数的栈帧
  4. 函数体执行：分配局部变量、执行逻辑
  5. 恢复栈帧：回收局部变量，恢复上一层的栈底
  6. ret 返回：弹出栈顶的返回地址，跳回调用处
  7. 栈清理：按约定清理栈上的参数

3. 关键指令底层行为

指令 底层做了什么
call Add 两步：① push 下一条指令地址（返回地址压栈）② jmp Add 跳转到函数
ret 8 两步：① pop eip 弹出返回地址，跳回去 ② esp += 8 自动清理栈上的参数
leave 等价于 mov esp, ebp; pop ebp，一键恢复栈帧，很多编译器会用它替代两行

逆向识别技巧：看到 push ebp / mov ebp, esp 固定开头，就是标准函数栈帧；[ebp+8] 开始是参数，[ebp-xxx] 是局部变量。

X64（64 位）函数调用（Microsoft x64 约定）

这是你当前逆向练习的核心约定，和 32 位差异极大：前 4 个参数放寄存器，必须预留 32 字节影子空间，栈 16 字节对齐。

以 int64_t TestFunc(int a, int b, int c, int d, int e) 为例，调用 TestFunc(1,2,3,4,5)。

2. 栈帧结构图（函数执行到中间时）

3. 核心重点

  1. 参数传递：前 4 个整型 / 指针参数依次放 RCX、RDX、R8、R9，第 5 个及以后从右往左压栈。
  2. 影子空间：调用者必须预留 32 字节，哪怕函数只有 1 个参数也要留；被调用函数可以把寄存器参数存到这里，方便调试和变参处理。
  3. 栈清理：全部由调用者负责，ret 只负责返回，不带清理数值；调用者在函数返回后用 add rsp, xxx 统一回收。
  4. 返回值：普通返回值放 RAX，128 位结构体放 RAX+RDX。
  5. 16 字节对齐：执行 call 指令前，RSP 必须是 16 的倍数，否则调用系统 API 会直接崩溃。

逆向识别技巧：x64 里看到给 RCX/RDX/R8/R9 赋值后紧跟 call，就是在传参调用函数；第 5 个参数一定在 RSP+偏移 的栈上。

核心对比总结

对比项 X86 stdcall X64 Microsoft 约定
参数传递 全部压栈，从右往左 前 4 个放寄存器，其余压栈
栈清理 被调用者 ret n 清理 调用者 add rsp, xxx 清理
影子空间 无 必须预留 32 字节
栈帧指针 Debug/Release 都常用 Release 版常省略，直接用 RSP 寻址
返回值 EAX RAX

调试器里怎么对应着看

在 x64dbg 中验证函数调用的标准方法：

  1. 在函数入口下断点，断下后看右上寄存器面板：RCX/RDX/R8/R9 就是前 4 个参数。
  2. 看右下栈面板：栈顶第一行是返回地址，再往下 32 字节影子空间，再往下是第 5 个及以后的参数。
  3. 函数执行完返回后，看 RAX 寄存器，里面就是函数返回值。

难点核心详解

1. 栈的核心规则（背下来）

栈就是一块从高地址向低地址增长的内存，ESP（栈指针寄存器）永远指向栈的最顶端（最后一个有效数据）。

  • push 寄存器

    ：做两件事

    1. esp = esp - 4（栈顶往下挪 4 字节）
    2. 把寄存器的值写入 esp 指向的内存

  • pop 寄存器

    ：做两件事

    1. 把 esp 指向的值读给寄存器
    2. esp = esp + 4（栈顶往上挪 4 字节）

通俗理解：栈就像一摞盘子，放盘子（push）就往下摞，栈顶往下走；拿盘子（pop）就往上拿，栈顶往上走。ESP 永远指着最上面那个盘子。

2. call 和 ret 的本质

很多人以为 call/ret 只是跳转，其实它们本质是跳转 + 操作栈：

  • call 目标地址

    ：

    1. 把下一条指令的地址（返回地址）push 压栈
    2. jmp 跳转到目标函数

  • ret

    ：

    1. 把栈顶的值弹出来，赋值给 EIP（指令指针）
    2. 程序就跳回调用处，继续往下执行

  • ret n

    （带数字的 ret）：

    1. 先弹栈恢复返回地址
    2. 再额外把 esp 加 n，相当于顺便回收了 n 字节的栈空间

    ✅ 这就是「被调用者清理栈」的核心！函数返回的时候顺手把参数栈清了。

一、32 位函数栈帧全流程（__cdecl vs __stdcall 对比）

我们用同一个函数做例子，把两种约定的完整执行流走一遍，你一眼就能看出区别。

例子函数原型：int add(int a, int b); 功能：返回 a+b

调用方代码：int res = add(1, 2);

先讲通用部分：函数栈帧的建立与销毁

为什么要有 push ebp / mov ebp, esp 这两步？

因为函数执行过程中，ESP 会随时变化（比如 push 东西、调用子函数），用 ESP 访问参数、局部变量，偏移量一直在变，很容易错。

所以我们用 EBP 做固定的基准指针：

  • [ebp + 正数] 永远是参数、返回地址

  • [ebp - 负数]

push 2 ; 参数b=2，先压（从右往左传参）
push 1 ; 参数a=1，后压
call add ; 1. 把返回地址压栈 2. 跳转到add函数
add esp, 8 ; ← 【关键】调用者自己把8字节参数栈回收

###  关键细节（面试 / 逆向必记）

1. **影子空间是调用方留的**，不是函数自己开的。函数可以直接用 `[rbp+16]` 这种偏移读写影子空间。
2. **栈必须 16 字节对齐**：call 指令执行前，RSP 必须是 16 的倍数。所以你经常会看到 sub rsp 后面的数字很奇怪，那就是在做对齐。
3. **64 位是调用者清理栈**：调用方开辟的参数栈 + 影子空间，函数返回后，调用方自己 `add rsp, n` 回收。对应 32 位的 __cdecl。
4. **逆向找第 5 个参数**：不是 `rsp+8`，是 `rsp + 8（返回地址） + 32（影子空间） = rsp+40`。很多新手在这里找错参数。

------

## __thiscall 为什么特殊？

32 位 C++ 成员函数默认用 __thiscall，本质和 __stdcall 几乎一样，只有一个区别：

- **this 指针放在 ECX 寄存器里，不压栈**
- 其他参数从右往左压栈
- 被调用者用 `ret n` 清栈

所以逆向时，只要看到函数里频繁用 `[ecx+偏移]` 访问数据，基本就是类成员函数，ECX 就是 this 指针。

## 一句话总结

1. 栈帧就是用 EBP 做基准，把参数、返回地址、旧 ebp、局部变量规整地放在栈上，方便稳定访问。
2. 调用者清理栈（__cdecl）：call 后面跟着 `add esp, n`，支持可变参数。
3. 被调用者清理栈（__stdcall/__thiscall）：函数尾是 `ret n`，代码体积更小，Windows API 默认。
4. 64 位影子空间：调用方必须给前 4 个寄存器参数预留 32 字节栈空间，供被调用函数溢出存储参数，位置在返回地址上方。
