PE 文件结构

PE（Portable Executable，可移植可执行文件）是 Windows 平台的标准可执行文件格式，.exe/.dll/.sys/.ocx 全部遵循该结构。它本质是 **「头信息 + 节表 + 节数据」的分层结构 **，磁盘文件和内存镜像一一对应，只是对齐粒度不同。

PE 结构是逆向分析的根基：静态分析看节区、导入表、导出表，脱壳修复导入表、IAT Hook、找 OEP，全部建立在对 PE 结构的理解之上。

一、PE 整体分层结构（磁盘文件顺序・文字结构图）

从文件开头到结尾，严格按以下顺序排列，层级清晰：

核心逻辑：头是「说明书」，记录文件的所有元信息；节表是「目录索引」，告诉每个节在哪里、有多大；节区是「真实内容」，存代码、数据、资源。

系统加载 PE 时，先读头获取信息，再按节表把每个节映射到虚拟内存里。

二、逐层拆解：关键字段 + 逆向意义

1. DOS 头（IMAGE_DOS_HEADER）

  • 大小：64 字节

  • 最核心字段：

    1. e_magic

0x5A4D

MZ


（最后 4 字节）：NT 头在文件中的偏移（FOA）。

- 逆向意义：找 NT 头全靠它，`PE头地址 = 文件起始地址 + e_lfanew`。

  • 其余字段都是为了兼容 DOS 系统，Windows 下完全不用管。

2. DOS 存根（DOS Stub）

  • 一段 16 位 DOS 代码，在纯 DOS 环境下运行时会弹出「This program cannot be run in DOS mode」提示。
  • Windows 下完全没用，逆向可以直接忽略。

3. NT 头（IMAGE_NT_HEADERS）

PE 文件的核心头部，分三部分：签名、文件头、可选头。

（1）PE 签名（Signature）

  • 4 字节，固定值 0x00004550，对应 ASCII PE\0\0。
  • 逆向意义：验证 e_lfanew 指向的是不是真的 PE 头。

（2）文件头（IMAGE_FILE_HEADER）

20 字节，描述文件的基础属性，逆向高频字段：

字段 大小 含义 + 逆向意义
Machine 2 字节 目标 CPU 架构：0x14C=x86 32 位，0x8664=x64 64 位
NumberOfSections 2 字节 节的数量。加壳后节数量通常会变少或变多
SizeOfOptionalHeader 2 字节 可选头的大小：32 位 = 224 字节，64 位 = 240 字节
Characteristics 2 字节 文件属性标志位：0x0002= 可执行文件0x2000=DLL 文件0x0001= 只重定位信息

（3）可选头（IMAGE_OPTIONAL_HEADER）

名字叫「可选」，实则是 PE 最核心的头，记录加载、内存布局的全部关键信息。32 位叫 PE32，64 位叫 PE32+，字段基本一致，只是指针宽度不同。

逆向必背的核心字段：

字段 含义 + 逆向意义
Magic 标识 PE 版本：0x10B=PE32 (32 位)，0x20B=PE32+(64 位)
AddressOfEntryPoint 程序入口点的 RVA，也就是 OEP（原始入口点）。加壳后这个值指向壳的入口，脱壳就是找真实的原始 OEP
ImageBase 程序默认加载的虚拟基址。32 位 exe 默认0x00400000，64 位默认0x0000000140000000。DLL 默认0x10000000。如果加载时这个地址被占了，就需要重定位
SectionAlignment 内存中的节对齐粒度，默认 4KB（0x1000），和页大小一致
FileAlignment 磁盘文件中的节对齐粒度，默认 512 字节（0x200）
SizeOfImage 整个 PE 在内存中的总大小（所有节 + 头按内存对齐后的总大小）
SizeOfHeaders 所有头 + 节表在磁盘上的总大小，按文件对齐
DataDirectory[16] 16 项数据目录表，每项 8 字节（4 字节 RVA + 4 字节大小），记录所有核心表的位置

16 项数据目录表（高频索引必须记）：

索引 名称 作用
0 导出表 DLL 的导出函数清单
1 导入表 依赖的 DLL 和 API 清单
2 资源表 图标、菜单、对话框等资源
5 重定位表 基址重定位修正项
6 调试信息 PDB 调试符号路径，Release 版通常为空

逆向意义：想找导入表、导出表、重定位表，直接去数据目录表对应索引拿 RVA 就行。

4. 节表（IMAGE_SECTION_HEADER）

节表是一个数组，每个元素对应一个节区，每个元素 40 字节，描述节区的全部信息。

逆向必背字段：

字段 含义 + 逆向意义
Name[8] 节名，比如 .text，最多 8 字节，ASCII 字符串
VirtualSize 节在内存中的实际大小（未对齐）
VirtualAddress 节在内存中的起始 RVA（相对于 ImageBase 的偏移）
SizeOfRawData 节在磁盘文件中的大小（按文件对齐后的大小）
PointerToRawData 节在磁盘文件中的起始偏移（FOA）
Characteristics 节的属性标志位，核心权限位：0x00000020= 代码段0x00000040= 已初始化数据0x20000000= 可执行0x40000000= 可读0x80000000= 可写

最核心的两个字段：VirtualAddress（内存偏移）和 PointerToRawData（磁盘偏移），RVA 和 FOA 的转换全靠这两个值。

三、核心概念深度解析：VA / RVA / FOA + 对齐

这是 PE 最容易懵的点，也是逆向计算地址的基础，必须彻底搞懂。

1. 三个地址的定义

  • VA（Virtual Address，虚拟地址）：程序加载到内存后，完整的虚拟地址，也就是调试器里看到的地址。

    公式：

  • RVA（Relative Virtual Address，相对虚拟地址）：相对于加载基址 ImageBase 的偏移量，内存中使用。

  • FOA（File Offset Address，文件偏移）：数据在磁盘 exe 文件里的字节偏移，010Editor 里看到的地址。

2. 为什么会有两种偏移？对齐的本质

PE 在磁盘和内存里是两套对齐规则：

  • 磁盘对齐（FileAlignment）：默认 512 字节，为了适配磁盘扇区大小，节省磁盘空间。
  • 内存对齐（SectionAlignment）：默认 4096 字节（4KB），和内存页大小一致，因为虚拟内存是按页管理的。

同一个节，在磁盘里按 512 字节对齐，在内存里按 4KB 对齐，所以同一个数据的 RVA 和 FOA 不一样。

四、三大核心表深度拆解

导入表、导出表、重定位表是 PE 最核心的三个表，和 DLL、Hook、脱壳、重定位直接联动。

1. 导入表（Import Table）

本质

记录程序依赖了哪些 DLL、调用了哪些 API，是动态链接的核心结构。对应数据目录表索引 1。

两个核心子表：INT 和 IAT

  • INT（Import Name Table，导入名称表）

    ：保存函数名字符串的 RVA 列表，相当于「购物清单」。

    • 磁盘和内存内容完全一致，永远不变。

  • IAT（Import Address Table，导入地址表）：

    • 磁盘里：和 INT 内容一模一样，存的也是函数名 RVA
    • 加载后：系统遍历 INT，根据函数名去 DLL 导出表里找真实地址，把真实 VA 地址填进 IAT 里
    • 程序调用 API 时：call [IAT表项地址]，间接跳转到 API 真实地址

逆向联动知识点

  1. IAT Hook 原理：直接修改内存里 IAT 表项的函数地址，替换成我们的 Hook 函数地址。程序再调用该 API 时，就会跳转到我们的代码。

  2. 脱壳为什么要修复导入表：

    修复导入表的本质：根据内存里已经填好的 IAT，反向重建出标准的导入表结构，写回磁盘文件。

  3. 显式链接隐藏导入：

2. 导出表（Export Table）

本质

DLL 对外暴露的函数清单，记录哪些函数可以被外部调用。对应数据目录表索引 0。exe 一般没有导出表，DLL 必有。

核心结构

导出表包含三个关键数组：

  1. 地址表：存每个导出函数的 RVA
  2. 名称表：存函数名字符串的 RVA
  3. 序号表：名称和地址的对应索引
  • 导出方式两种：按名称导出、按序号导出。按序号导出更隐蔽，体积更小。

逆向联动知识点

  1. GetProcAddress 的原理：就是查导出表，根据函数名 / 序号找到函数的 RVA，加上模块基址就是真实地址。
  2. EAT Hook 原理：修改导出表里的函数地址，别人调用这个 DLL 的导出函数时就会被劫持。
  3. 逆向分析 DLL：第一步就是看导出表，快速知道这个 DLL 提供了哪些功能。

3. 基址重定位表（Relocation Table）

本质

PE 默认会加载到 ImageBase 指定的地址，如果这个地址被其他模块占了，系统就会把它加载到别的地址。此时所有「绝对地址」都会失效，重定位表就是用来修正这些绝对地址的。对应数据目录表索引 5。

工作原理

  1. 编译时，编译器把所有需要修正的绝对地址整理成重定位表，存在 PE 里。
  2. 加载时，如果实际基址！= 默认 ImageBase，系统就遍历重定位表，把每个绝对地址都加上「实际基址 - 默认基址」的偏移量。
  3. 修正完程序就能正常运行。

逆向联动知识点

  1. DLL 必须有重定位表，因为 DLL 加载地址不固定；exe 默认有，开启 ASLR 后一定会被系统随机基址，必须重定位。
  2. 壳脱壳后通常需要修复重定位表，否则换个加载地址程序就崩溃。
  3. 加壳程序通常会自己处理重定位，不需要系统的重定位表，所以加壳后重定位表可能被删掉。

五、常见节区全解 + 加壳后特征

1. 标准节区详解

节名 内容 默认权限 逆向意义
.text 可执行代码指令，所有函数的机器码 可读可执行 反汇编、分析逻辑的主要区域
.data 已初始化的全局变量、静态变量 可读可写 全局配置、全局变量都在这里
.rdata 只读数据：字符串常量、虚函数表、常量 只读 搜字符串、找虚表都在这里
.idata 导入表 只读 看依赖的 DLL 和 API
.edata 导出表 只读 DLL 的公开接口
.reloc 基址重定位表 只读 基址变动时修正地址
.rsrc 资源：图标、菜单、对话框、字符串表 只读 提取资源、找对话框 ID
.bss 未初始化的全局变量 可读可写 磁盘不占空间，加载时系统自动分配并清零

2. 加壳后的节区特征（一眼识别加壳）

  1. 节名异常：出现 .upx0、.packed、.vmp0 这类非标准节名
  2. 入口点不在 .text：OEP 指向壳自己的节，不是代码段
  3. 导入表极少：只有 kernel32.dll 的几个 API，其余全靠动态加载
  4. 权限异常：多个节都是 可读+可写+可执行 全权限，正常程序只有代码段可执行
  5. 节数量变化：可能变少（合并节），也可能变多（新增壳节）

六、PE 加载到内存的完整流程

结合虚拟内存、动态链接知识，串起整个过程：

  1. 系统读取 PE 头，获取 ImageBase、节数量、节表信息
  2. 在进程虚拟地址空间里，按 SizeOfImage 分配一块连续内存
  3. 按节表把每个节从磁盘映射到对应的虚拟地址，设置对应页的读写执行权限
  4. 处理基址重定位：如果实际加载地址！= ImageBase，遍历重定位表修正所有绝对地址
  5. 处理导入表：加载所有依赖的 DLL，遍历 INT 查找函数地址，填充 IAT 表
  6. 初始化 TLS、堆、栈等运行环境
  7. 跳转到 AddressOfEntryPoint（OEP），开始执行程序

七、逆向实战中的 PE 结构应用

  1. 查壳判难度：看节名、节权限、导入表，快速判断有没有壳、壳的类型
  2. 找 OEP：正常程序 OEP 在 .text 段，加壳后指向壳节，脱壳就是找跳回 .text 的位置
  3. 静态定位功能：看导入表判断程序功能方向；搜 .rdata 的字符串回溯业务逻辑；看资源找对话框和控件 ID
  4. 脱壳修复：Dump 内存镜像后，修复导入表、修复重定位、修复节对齐，生成可运行的脱壳文件
  5. IAT Hook：找到 IAT 表的内存地址，修改对应表项的函数地址，完成劫持

八、问题合集

Q：讲一下 PE 文件的结构

答：PE 是 Windows 的标准可执行文件格式，整体分层结构，从磁盘开头依次是 DOS 头、DOS 存根、NT 头、节表、各个节区。

DOS 头开头是 MZ 魔数，通过 e_lfanew 字段指向 NT 头；NT 头包含 PE 签名、文件头、可选头和数据目录表，文件头描述架构、节数量、文件属性，可选头是核心，记录了入口点 OEP、默认加载基址、对齐粒度，16 项数据目录表则指向导出表、导入表、重定位表这些核心结构。

节表是数组，每个元素描述一个节区的名字、大小、磁盘和内存偏移、权限；后面就是真实的节区内容，比如存放代码的 .text、存放数据的 .data、存放导入表的 .idata 等。

PE 在磁盘和内存中是一一映射的，只是对齐粒度不同，磁盘默认 512 字节对齐，内存默认 4KB 页对齐，所以同一个数据的文件偏移和相对虚拟地址可以通过节表的字段互相转换。

Q：导入表和 IAT 是什么关系？脱壳为什么要修复导入表？

答：导入表包含 INT 导入名称表和 IAT 导入地址表。INT 保存函数名清单，全程不变；IAT 在磁盘里和 INT 内容一致，程序加载时系统会根据 INT 查找每个 API 的真实地址，填充到 IAT 里，之后程序调用 API 就通过 IAT 间接跳转。

加壳程序运行时，壳代码会自己手动加载 DLL、填充 IAT，但不会修复磁盘上的导入表结构。直接 Dump 出来的文件，磁盘导入表是损坏的，下次运行系统无法解析依赖，程序会直接崩溃。所以脱壳必须根据内存里已经填好的 IAT，反向重建标准的导入表结构写回磁盘，也就是修复导入表。

加壳 PE 的肉眼识别特征

无论用哪种工具，看到以下特征即可快速判定程序加壳：

  1. 节名异常：不是标准的.text/.data/.rdata，出现.upx/.packed/.vmp0 等壳专属名称
  2. 节权限异常：多个节同时具备「可读 + 可写 + 可执行」全权限，正常程序只有代码段可执行
  3. 导入表极简：只有 kernel32.dll 的少数基础 API（如 LoadLibraryA、GetProcAddress），其余依赖全部隐藏
  4. 入口点异常：OEP 不在.text 节，指向壳自己的节区
  5. 大小倒挂：节的 VirtualSize 远大于 SizeOfRawData，说明运行时会在内存中解压出大量代码

PE 结构是纯二进制布局，直接看原始文件全是十六进制乱码，必须靠「工具解析 + 结构映射」才能看懂。按学习和使用优先级分三种方法，全部对应你之前学的 PE 理论字段，学完既能上机实操，也能应对面试手算偏移的题型。

查看PE结构

方法一：专用可视化工具

最省心的方式，工具自动把二进制解析成结构化的树状表格，字段名、数值、含义全部标注好，不用自己算偏移。

推荐工具

  • CFF Explorer：Windows 逆向圈最通用的 PE 查看 / 编辑工具，轻量、字段完整、支持直接修改，面试和日常分析都用它
  • PE-bear：开源免费，内置查壳签名库、熵值分析，加壳、损坏的 PE 也能解析，恶意样本分析常用

    ：共 20 字节

  • SizeOfRawData / PointerToRawData：节在磁盘中的对齐后大小、起始文件偏移
  • Characteristics：节的权限标志（可读、可写、可执行）
第四层：导入表 / 导出表
  • 点击「Import Directory」：看到所有依赖的 DLL，展开每个 DLL 可看到导入的全部 API 函数名
  • 点击「Export Directory」：仅 DLL 文件有，展示所有对外导出的函数名、序号、地址

对应理论：导入表对应数据目录索引 1，导出表对应索引 0

方法二：010 Editor + PE 模板（字节级对应・学底层必用）

十六进制编辑器搭配结构模板，既能看到原始二进制字节，又能自动解析出结构化字段。点击结构字段会自动高亮对应的十六进制字节，是理解「每个字段占几字节、在文件哪个位置」的最佳方式，彻底把理论和二进制对应起来。

操作步骤

  1. 安装 010 Editor，软件自带 PE 格式模板；没有的话在顶部「Templates → Template Repository」里搜索「Windows PE」安装
  2. 拖入目标 exe 文件
  3. 点击工具栏的「运行模板」按钮（▶️），选择 Windows Binary (PE) 执行
  4. 左侧出现「Template Results」结构树，分层和 PE 理论完全一致
核心优势：字节级高亮联动
  • 点击结构树里的 e_lfanew，右侧十六进制窗口会自动高亮对应的 4 个字节
  • 点击节表里的 .text 节项，会高亮整个节表项的 40 字节
  • 能直观看到小端存储的真实样子、结构之间的衔接边界、对齐填充的空白字节

方法三：纯手动十六进制定位（不用任何模板・面试手算必会）

只用普通十六进制编辑器（如 WinHex、HxD），纯靠偏移计算一步步找到所有结构，这是面试常考的「硬实力」，也是脱壳修复损坏 PE 的基础。

完整手动定位步骤（按顺序查找）

打开十六进制编辑器载入 exe，从文件开头（偏移 0x00000000）开始：

第一步：确认 DOS 头，定位 NT 头
  1. 文件开头前 2 字节为 4D 5A（ASCII 字符 MZ），确认是合法 PE 文件
  2. 跳到 0x003C 偏移处（e_lfanew 字段，固定在 DOS 头的 0x3C 位置，占 4 字节）
  3. 读出这 4 字节的值（小端），比如 E0 00 00 00 对应实际值 0x000000E0，这就是 NT 头的文件偏移
第二步：验证 NT 头签名
  1. 跳到上面算出的 NT 头偏移（如 0xE0）
  2. 前 4 字节为 50 45 00 00（对应字符串「PE\0\0」），确认是 NT 头起始位置
第三步：读取文件头

签名后紧接着是文件头，固定 20 字节：

  • 偏移 + 0：Machine（2 字节），0x014C=32 位，0x8664=64 位
  • 偏移 + 2：NumberOfSections（2 字节），节区总数量
  • 偏移 + 16：SizeOfOptionalHeader（2 字节），可选头的字节长度
  • 偏移 + 18：Characteristics（2 字节），文件属性标志位
第四步：读取可选头

文件头之后就是可选头，大小由 SizeOfOptionalHeader 决定（32 位默认 224 字节，64 位默认 240 字节）：

  • 可选头 + 12 字节：AddressOfEntryPoint（4 字节），OEP 的 RVA
  • 可选头 + 16 字节：ImageBase（4/8 字节），默认加载基址
  • 可选头尾部：16 项数据目录，每项 8 字节（4 字节 RVA + 4 字节大小）
    • 第 0 项：导出表
    • 第 1 项：导入表
    • 第 5 项：基址重定位表
第五步：定位节表

可选头结束的位置就是节表起始位置：

  • 每个节表项固定 40 字节
  • 前 8 字节：节名（ASCII 字符串，不足 8 字节补 0）
  • 偏移 + 12：VirtualAddress（4 字节），节的内存起始 RVA
  • 偏移 + 20：PointerToRawData（4 字节），节的磁盘起始偏移
  • 偏移 + 36：Characteristics（4 字节），节的权限标志
第六步：手算 RVA 转 FOA（面试高频题）

比如已知导入表 RVA，求它在磁盘文件中的偏移：

  1. 先获取导入表 RVA，例如 0x2000
  2. 遍历节表，找到满足「节 VirtualAddress ≤ 0x2000 < 节 VirtualAddress + 节 VirtualSize」的节
  3. 代入公式：FOA = RVA - 节VirtualAddress + 节PointerToRawData
  4. 跳到算出的 FOA 偏移，就能看到导入表的原始二进制结构
