MYSQL锁机制

全局锁

mysql全局锁用于保护整个数据库实例，优先级最高

全局锁处于激活状态时，其他会话只能对数据库进行读操作

全局锁特点

全局性：全局锁会锁定整个数据库实例，而不是单个表或行

**独占性：**全局锁是互斥的，一次只能由一个会话持有。

**高优先级：**全局锁的优先级很高，其他锁无法与之竞争

使用场景

**1.数据库备份：**在进行数据库备份时，需要保证备份数据的完整性，可以使用全局锁来锁定整个数据库，确保不会有其他会话对数据库进行写操作。

**2.数据库恢复：**在进行数据库恢复操作时，为了保证恢复的完整性，可以使用全局锁来防止其他会话对数据库进行修改。

**3.维护任务：**在进行需要独占数据库资源的维护任务时，可以使用全局锁来确保任务的顺利进行

使用语法

表锁

表锁粒度太大

多个会话同时对同一个表进行读写操作时，可能会发生数据不一致或丢失问题。

共享锁

多个会话可以同时获取共享锁，并且可以同时读取表中的数据，但不能进行写操作，适用于并发读取操作 ，可以防止其他会话获取排他锁，保证读操作的一致性

排他锁

只有一个会话能够获取到排他锁，并可以进行读写操作。排他锁适用于更新和删除操作，可以防止其他会话同时对表进行读写操作

使用方法

一定要释放锁

在需要对表进行读操作的地方，可以使用共享锁

在需要对表进行写操作的地方，可以使用排他锁

使用完毕后，需要释放锁（UNLOCK TABLES）

元数据锁

元数据是描述数据库的数据结构及其属性的信息，包括表、列、索引、触发器等对象的定义和属性。

元数据锁是数据库管理系统用来保护元数据一致性的机制，它能够确保在多用户并发访问数据库时，对元数据的修改操作能够按照适当的顺序进行，从而保证数据库的正确性和稳定性，元数据锁一般由数据库管理系统自动管理，不需要用户手动操作。

在MySQL5.5中引入了MDL、当对一张表进行增删改査的时候、加MDL读锁(共享):当対表结构进行变更操作的时候、加MDL写锁（排他)。

意向锁

当我想改表结构或者其他操作时，需要加表锁，但此时我不知道表里有没有行锁，我要遍历全表检查行锁，意向锁就是我准备在表里某些行加锁，我在该表贴个标签，告诉其他想要加表锁的会话，这样就不用扫描全表，

意向共享锁（IS锁）：当事务要对一个资源进行共享访问时，可以先尝试获取意向共享锁。当事务获取到意向共享锁后，其他事务也可以获取意向共享锁，但不能获取意向排他锁。这样可以减少锁冲突，提高并发性能。

意向排他锁（IX锁）：当事务要对一个资源进行排他访问时，可以先尝试获取意向排他锁。当事务获取到意向排他锁后，其他事务不能获取意向共享锁或意向排他锁。这样可以确保只有一个事务可以同时对资源进行排他操作。

行锁

MYSQL数据库行锁是一种用于并发控制的锁机制，只有在InnoDB引擎才有，可以在多个会话同时访问同一张表不同行时提供数据一致性和并发保证。

行级锁是MySQL中最细粒度的锁，它只锁定表中的一行数据，而不是整个表或页面。这意味着其他会话可以同时访问同一张表的其他行，而不会被阻塞。

共享锁（S锁）：也称为读锁，多个会话可以同时获取S锁，并且互不干扰。多个会话获取S锁后，可以并发地读取被锁定的行数据，但无法修改或删除这些行。只有当所有S锁都被释放后，其他会话才能获取X锁进行修改操作。

排他锁（X锁）：也称为写锁，只有一个会话可以获取X锁，并且其他会话无法获取任何锁。获取X锁的会话既可以读取被锁定的行数据，也可以修改或删除这些行。只有当X锁被释放后，其他会话才能获取任何锁。

行级锁的使用可以提高并发性能，因为它可以减少锁冲突。但是，行级锁也有一些限制，比如占用更多的内存和CPU资源，以及可能导致死锁等并发问题

行锁无法解决幻读

行锁只能锁住已经存在的记录，无法阻止其他事务在间隙（两个记录之间的空白区域）插入新数据。Next-Key 锁通过锁住 “记录 + 间隙” 的整个区间，彻底解决了幻读问题。

默认情况下，InnoDB在可重复读隔离级别下，innoDB使用next-key锁进行搜索和索引扫描，防止幻读

NEXT_KEY锁为记录所+间隙锁

幻读是指同一个事务内，两次执行相同的范围查询，第二次查询返回了第一次没有的行。这是因为其他事务在两次查询之间插入了符合条件的新数据。

记录锁（Record Lock）：锁住索引上的某一条具体记录

间隙锁（Gap Lock）：锁住两个索引记录之间的间隙，防止其他事务在间隙中插入数据

间隙锁的主要作用是锁定一段数据范围之间的间隙，即锁定两个值之间的空隙。当一个事务对数据范围中的某个数据行进行锁定时，间隙锁会阻止其他事务在这个数据范围内插入新的数据行。这样可以确保在事务修改数据时，其他事务不会在相同的数据范围内插入新的数据，从而保持数据的一致性。

记录锁也分为共享锁和排他锁

因为记录锁是作用于索引上的，innoDB肯定是有索引的，记录锁总是锁定索引记录。

行锁也有一种情况会升级为表锁，就是在我们的写操作中，where条件后没有用到索引，就会升级为表锁。

间隙锁只存在于 RR 隔离级别，RC 隔离级别下没有间隙锁

间隙锁之间不会互相冲突，多个事务可以同时持有同一个间隙的锁

间隙锁的唯一作用是阻止其他事务插入间隙，不会阻止读操作

加锁机制

1. 加锁的基本单位是 Next-Key 锁

默认情况下，InnoDB 对所有访问到的索引记录都加 Next-Key 锁。

2. 只有访问到的对象才会加锁

InnoDB 只会对查询过程中实际扫描到的索引记录加锁，而不是对所有符合条件的记录加锁。

反例：如果 age 字段没有索引，那么SELECT * FROM user WHERE age > 20 FOR UPDATE会进行全表扫描，对所有记录和间隙加锁，相当于表锁。

3. 唯一索引等值查询，命中记录时，退化为行锁

当使用唯一索引进行等值查询，并且命中了一条记录时，Next-Key 锁会退化为只锁住该记录的行锁。

4. 唯一索引等值查询，未命中记录时，退化为间隙锁

当使用唯一索引进行等值查询，但没有命中任何记录时，Next-Key 锁会退化为只锁住该值所在的间隙。

5. 普通索引等值查询，无论是否命中，都加 Next-Key 锁

普通索引没有唯一性约束，即使等值查询命中了记录，也会继续向后扫描，直到遇到第一个不满足条件的记录，所以会加完整的 Next-Key 锁。

6. 范围查询会访问到不满足条件的第一个值为止

无论是唯一索引还是普通索引，范围查询都会一直扫描到第一个不满足条件的记录，并对其加锁。

7. limit 1 会提前终止扫描

如果查询语句带有limit 1，那么扫描到第一条符合条件的记录后就会停止，不会继续向后扫描，加锁范围会缩小。

误区 1：MyISAM 的读性能比 InnoDB 好

正确：在 MySQL 5.1 及以前，MyISAM 的读性能确实比 InnoDB 好。但在 MySQL 5.5 以后，InnoDB 的性能已经全面超越了 MyISAM，尤其是在并发读的情况下。

误区 2：MyISAM 的 count (*) 比 InnoDB 快很多

正确：MyISAM 的 count () 不带条件确实很快，因为它预存了表的总行数。但 InnoDB 的 count () 在有索引的情况下也很快，而且 MyISAM 的 count (*) 是不准确的，因为它不支持事务。

误区 3：Memory 引擎比 Redis 快

正确：Memory 引擎是 MySQL 的内存表，而 Redis 是专门的缓存数据库。Redis 的性能比 Memory 引擎高得多，而且支持更多的数据结构和功能。所以，缓存场景优先选择 Redis，而不是 Memory 引擎。

MYSQL索引

索引本质

如何定位一个文件？
定位一个文件→定位一个扇区
想要找到一个文件的全部，本质就是在磁盘中找到所有保存该文件数据的扇区。而我们如果能够定位任何一个扇区，那么便能找到所有扇区，因为查找方式是一样的 到此为止，定位一个文件的问题，就被我们转化成了定位一个扇区的问题

如何定位一个扇区？
CHS地址
盘面号+柱面号+扇区号
磁头（Heads）、柱面(Cylinder)、扇区(Sector)对应的编号。即可在磁盘上定位所要访问的扇区。这种磁盘数据定位方式叫做 CHS

LBA地址
实际系统软件使用的并不是 CHS （但是硬件是），而是 LBA ，一种线性地址，LBA与CHS之间的关系可以想象成虚拟地址与物理地址之间的关系。系统先将LBA传给磁盘控制器，然后磁盘控制器就会自动将 LBA 地址最后会转化成为 CHS ，交给磁盘去进行数据读取。不过，我们现在不关心转化细节，知道这个东西，让我们逻辑自洽起来即可

为什么操作系统不使用CHS地址，而使用LBA地址访问磁盘呢？
因为硬件的型号是容易更换的，如果我操作系统 不做到和硬件解耦，那硬件一但更换，操作系统也得跟着瘫痪

mysql存储数据以页为单位存储，16kB(不同于操作系统内存页，一个mysql的IO对应系统的四次IO)

用户修改数据库中1KB的数据，与整体替换该1KB数据所属的磁盘块，哪个时间更久？

一样的！哪怕你只改1个比特位，你花的时间和改16KB的数据都是一样的

MYSQL的CRUD操作需要通过计算找到对应的插入位置，只要涉及到计算就有cpu的参与，为了便于cpu参与则需要先将数据移到内存中，再特定时间一定是磁盘和内存都有的，刷盘加载涉及到IO，IO的基本单位为page，因此更高的效率就是减少系统和磁盘的IO次数

mysql与磁盘交互过程

mysql应用程序通过内部的缓冲区，以PAGE（16KB）为单位与操作系统进行交互。操作系统通过内核缓冲区以4KB为单位与磁盘进行交互
mysql中对任何数据进行修改，根本不是直接对磁盘文件进行任何修改！！ 具体步骤如下：

程序通过open系统调用打开对应的磁盘文件，底层调用磁盘驱动程序将磁盘中对应的page读取到内存中
操作系统调用系统调用read，将page中的数据读取到程序中
mysql在这个PAGE中查找对应的数据
找到之后对其进行相应修改
然后使用特定的刷新策略刷新回PAGE（操作系统调用系统调用write）
将内存中的PAGE刷新回磁盘（磁盘驱动程序）
mysql所有CURD操作，其实都是对内存中对应的PAGE做操作，然后定期刷新到磁盘中的！！

为何MySQL和磁盘进行IO交互的时候，要采用Page的方案进行交互呢？用多少，加载多少不香吗？
决定IO效率的，往往不是IO的数据大小，而是IO的次数
采用Page的方案进行交互，如果我多次查找的数据恰好都在一个PAGE中，那么总共就只需要一次IO，这就节约了大量时间，提升了效率

为什么mysql内部要自动排序？

因为自动排序有利于查找，插入数据时排序就是优化查询效率

索引文件在内存还是磁盘？

使用的时候在内存我们用户对数据库做的增删查改操作，对象全都是内存中的索引结构（其实就是一颗B+树）
索引结构的最底层就是具体的数据，这些数据包含于索引结构，也都是在内存中的，使用前会将索引文件从磁盘读到内存中，磁盘会备份一份索引文件

mysql对page的管理？

创建一个索引的本质，其实就是在mysql中创建一个B+树

mysql是做数据管理的，IO的基本单位是16KB的page，mysql要管理的数据量可能非常大，也就意味着需要大量的Page！！是不是意味着，在同一时刻，mysql内部会存在大量的page！，因此MySQL就需要对所有的Page进行统一管理

通过优化索引结构提升查找效率

如果数据的量比较大，那么我们一定需要多个Page来保存这些数据
目前多个Page彼此使用双链表链接起来，而且每个Page内部的数据也是基于链表的。那么，查找特定一条记录，无论是页表内还是页表间都只能通过线性 查找，时间复杂度是O(n)，这效率也太低了

如何优化单个页表内的查找效率？

我们的方法就是：引入页目录

给每一张PAGE都创建一个对应的页目录

什么是页目录？页目录如何提高查找效率？

页目录主要就是通过以时间换空间的思路来提高效率

PAGE的ID值如何选取？

通常我们选取这一页PAGE中值最小的主键作为这一页的ID

我们通常建立索引，本质就是直接或者间接建立B+树

B+树索引

非叶子节点page中存着下层目录，叶子节点存着数据。

查找的时候，自定向下找，只需要加载部分目录页到内存，即可完成算法的整个查找过程，大大减少了IO次数。

B+树是胖矮状态的，我们只先需要加载顶层目录页，再加载对应的一个次级目录，最后加载一个数据页，这样通过B+树的结构我们大大减少了IO次数

为什么不用其他数据结构？

InnoDB 是在 MySQL 5.5 之后成为默认的 MySQL 存储引擎，B+Tree 索引类型也是 MySQL 存储引擎采用最多的索引类型。

在创建表时，InnoDB 存储引擎会根据不同的场景选择不同的列作为索引：

如果有主键，默认会使用主键作为聚簇索引的索引键（key）；
如果没有主键，就选择第一个不包含 NULL 值的唯一列作为聚簇索引的索引键（key）；
在上面两个都没有的情况下，InnoDB 将自动生成一个隐式自增 id 列作为聚簇索引的索引键（key）；
其它索引都属于辅助索引（Secondary Index），也被称为二级索引或非聚簇索引。创建的主键索引和二级索引默认使用的是 B+Tree 索引。

自动创建索引

1. 创建主键约束时（必创建）

这是最常见的自动创建索引场景。当为表指定 PRIMARY KEY 约束时，InnoDB 会自动创建一个名为PRIMARY的聚簇索引，这也是 InnoDB 表数据的物理存储方式。

关键细节：

  • 如果创建表时没有显式指定主键，InnoDB 会按以下规则自动选择聚簇索引：
    1. 选择第一个非空唯一索引作为聚簇索引
    2. 如果没有任何非空唯一索引，会自动创建一个隐藏的 6 字节 ROWID 列作为聚簇索引（对用户不可见）
  • 主键索引是唯一的、非空的，且一个表只能有一个聚簇索引

2. 创建唯一约束时（必创建）

当为列或列组合指定UNIQUE约束时，MySQL 会自动创建对应的唯一索引来实现唯一性检查。

关键细节：

  • 唯一索引允许有多个NULL值（因为NULL != NULL）
  • 联合唯一约束会自动创建联合唯一索引
  • 唯一约束是逻辑概念，唯一索引是其物理实现

3. 创建外键约束时（默认自动创建）

这是最容易被忽略的自动创建索引场景。当创建 FOREIGN KEY 外键约束时，MySQL 会自动在外键列上创建一个普通索引，如果该列已经存在索引（或作为联合索引的前缀），则不会重复创建。

关键细节：

  • 这是 MySQL 的默认行为，目的是加速外键关联查询和级联操作
  • 自动创建的外键索引命名规则通常为表名_ibfk_n
  • 可以通过FOREIGN KEY ... INDEX显式指定索引名
  • 重要坑点：删除外键约束时，自动创建的索引不会被自动删除，需要手动删除

4. 执行查询时自动创建临时索引

MySQL 优化器在执行某些复杂查询时，会临时为派生表、临时表或连接列创建索引来加速查询执行，查询结束后这些索引会被自动销毁。

常见触发场景：

  • 执行JOIN操作时，如果连接列没有索引，MySQL 可能会为其中一个表的连接列创建临时索引
  • 执行GROUP BY或ORDER BY操作时，如果排序列没有索引，MySQL 可能会创建临时索引
  • 子查询生成的派生表，如果需要进行连接或排序操作

  • 全文索引：如果在创建表时指定FULLTEXT关键字，MySQL 会自动创建全文索引（这本质上是显式指定，不算完全 “主动”）

MySQL 永远不会为普通查询列自动创建索引，即使该列被频繁用于WHERE、ORDER BY、GROUP BY或JOIN条件。

原因：

  • 创建索引有显著的存储开销和维护开销（插入、更新、删除操作都需要维护索引）
  • MySQL 无法自动判断 “创建索引带来的查询性能提升” 是否大于 “维护索引带来的开销”
  • 索引的选择需要结合业务场景和查询模式，这是开发者的责任

自动创建索引的注意事项

  1. 自动创建的索引可以手动删除

    注意：删除主键索引会同时删除聚簇索引，导致表重建，性能开销极大。

  2. 索引名冲突问题

    如果手动创建的索引名与自动创建的索引名冲突，会报错。建议显式指定所有索引的名称。

  3. MySQL 8.0 的变化

    • 自动创建的索引可以通过ALTER TABLE ... ALTER INDEX ... INVISIBLE设置为不可见
    • 优化器会忽略不可见索引，但索引仍然会被维护
    • 可以用于测试删除某个索引对性能的影响，而无需真正删除

  4. 性能影响

    • 自动创建的临时索引会消耗额外的 CPU 和内存资源
    • 对于大表，创建临时索引可能会导致查询变慢
    • 如果发现查询经常自动创建临时索引，说明需要手动为相关列创建永久索引

辅助索引

所以如果我们想要对其他非主键属性建立索引，这种索引就叫做辅助索引

Myisam建立辅助索引的方式是什么？

对于存储引擎Myisam而言，建立辅助索引的本质就是以这个属性为比较标准，新创建一棵B+树
注意：在查询频率很高，修改频率很低的场景下，MyISAM用的比较多

Innodb建立辅助索引的方式是什么？

对于存储引擎Innodb而言，建立辅助索引的本质也是以这个属性为比较标准，新创建一棵B+树

但Innodb建立索引的方式与Myisam的区别在于：
Myisam的辅助索引对应B+树叶节点中保存的是数据页的指针 （索引键值 + 该记录对应的物理地址指针） ，但Innodb的辅助索引B+树叶节点中保存的仅仅是数据的一列主键信息 （索引键值 + 主键值），并不会把一行的所有数据都保存进叶节点

InnoDB 的非主键索引中叶子节点并没有数据，而只有对应记录的主键值。所以通过辅助（普通）索引，找到目标记录，需要两遍索引：

首先检索辅助索引获得主键
然后用主键到主索引中检索获得记录
这种根据辅助索引查询记录的方式，就叫做回表查询

为什么Innodb的辅助索引叶节点中存的不是完整的数据记录，而只存该记录的主键值呢？

因为建立B+树也是很耗空间的，Myisam这种方式虽然简单，但如果我想给所有非主键属性（假如说8个）都做辅助索引，就得建8个和主键索引一样大小的B+树，太占空间了！
因此我们要采取一种用时间换空间的策略
而采用Innodb的方案，虽然增加了查找次数（要查一个辅助索引和一个主键索引），但是却大大减少了空间消耗（假如说一行数据总共36B，主键占4B，那同样的辅助索引，Myisam的一个叶节点大小就是Innodb的9倍）

问题

InnoDB 为什么用聚簇索引？

  • 主键查询性能极高，直接返回完整数据
  • 范围查询可以连续读取数据页，IO 效率高
  • 数据和索引在一起，减少了一次 IO

为什么主键建议用自增整数？

  • 自增主键是有序的，插入数据时只会在 B + 树的末尾追加，不会导致页分裂
  • 如果用 UUID 作为主键，插入是随机的，会频繁导致页分裂，性能极差
  • 整数比字符串小，二级索引的叶子节点存储主键值时更节省空间

为什么不建议用很长的字段作为主键？

  • 二级索引的叶子节点都存储主键值，主键越长，每个二级索引占用的空间就越大
  • 回表时需要比较更长的主键值，性能也会下降

什么时候需要创建二级索引？

  • 经常用于WHERE、JOIN、ORDER BY、GROUP BY的字段
  • 区分度高的字段（比如手机号、邮箱），区分度低的字段（比如性别）不需要建索引

为什么不要创建太多二级索引？

  • 每个二级索引都需要占用存储空间
  • 插入、更新、删除操作时，需要维护所有相关的二级索引，性能开销大

误区：聚簇索引就是主键索引

  • 纠正：主键索引是聚簇索引的一种常见形式，但不是唯一形式。如果表没有显式主键，InnoDB 会选择第一个非空唯一索引作为聚簇索引；如果也没有，会自动创建一个隐藏的 6 字节 ROWID 作为聚簇索引。

误区：非聚簇索引一定比聚簇索引慢

  • 纠正：如果是覆盖索引，非聚簇索引的性能和聚簇索引一样快（1 次 IO）。

误区：MyISAM 的查询速度一定比 InnoDB 快

  • 纠正：InnoDB 的主键查询比 MyISAM 快一倍；只有在纯读且没有主键查询的场景下，MyISAM 才可能略快。

误区：InnoDB 的二级索引比 MyISAM 慢

  • 纠正：普通二级查询都是 2 次 IO，性能相当；但 InnoDB 的覆盖索引比 MyISAM 快，因为不需要读数据文件。

就是基于主键字段创建的索引，一般主键索引会在指定主键时默认创建好，也就是说，主键字段天然就具有主键索引，不用我们手动创建。

唯一索引

唯一键索引（Unique Key Index）是带有 “唯一性约束” 的辅助索引—— 在辅助索引 “加速查询” 的基础上，额外强制索引列（或列组合）的值 “不可重复”（部分数据库允许一个 NULL，如 MySQL）。它是辅助索引的特殊类型，本质是 “辅助索引 + 唯一性约束” 的结合体。

在 InnoDB 中，唯一键索引和普通辅助索引的B + 树结构完全一致，仅在 “数据写入时的校验逻辑” 上有差异。

对B+树执行插入/更新操作时，普通辅助索引是直接将 “键值 + 主键值” 插入索引的 B+ 树，无需判断键值是否重复；
而唯一键索引在插入 / 更新前，还会先在索引的 B+ 树中查询 “新键值是否已存在”。若存在：直接返回 Duplicate entry 错误，拒绝写入；若不存在：才将 “键值 + 主键值” 插入索引，保证索引键值唯一。

为什么唯一键索引查询效率高？

唯一键索引查询效率高的本质是键值唯一性”带来的精准定位能力 ：

键值不重复，使得 B+ 树查找可“一次命中”，无需范围扫描，从而减少了磁盘 I/O 次数和数据处理量；
与主键索引协同时，回表效率更高； （普通索引可能返回多个主键值，需多次回表，唯一键索引只需要一次回表）
覆盖查询场景下可直接返回结果，避免回表操作，减少额外开销。
这种效率优势在“等值查询”（=）场景中尤为明显，这也是唯一键索引适合作为“高频查询的唯一标识列”（如手机号、订单号）的核心原因。

唯一键索引与主键索引有什么区别

主键索引的特点是什么？

一个表中，最多有一个主键索引
主键索引的效率高（主键不可重复）
创建主键索引的列，它的值不能为null，且不能重复
唯一键索引的特点是什么？

一个表中，可以有多个唯一索引
查询效率高
如果在某一列建立唯一索引，必须保证这列不能有重复数据

唯一键索引的底层结构（以 InnoDB 为例）
在 InnoDB 存储引擎中，唯一键索引属于辅助索引，其 B+ 树结构与普通辅助索引一致，但会额外校验“值的唯一性”：

非叶子节点：存储“唯一键的键值 + 指向子节点的指针”，用于索引导航；
叶子节点：存储“唯一键的键值 + 对应的主键值”（而非完整行数据）。
当通过唯一键索引查询时，流程为：

在唯一键索引的 B+ 树中找到匹配的叶子节点，获取对应的主键值；
用主键值到“主键聚簇索引”中查找完整行数据（即“回表”操作，与普通辅助索引一致）。
示例：若user表主键为id，为email列建唯一键索引，则该索引的叶子节点存储(email值, id值)；查询WHERE email = 'a@xxx.com’时，先通过唯一键索引拿到id，再到主键索引取完整数据。

唯一键索引的适用场景

唯一键索引的核心价值是“在保证数据唯一性的同时加速查询”，典型适用场景包括：

存储“天然唯一”的业务字段，如用户的手机号、邮箱、身份证号（需避免重复注册/录入）；
关联表的唯一标识字段，如订单表的order_no（需确保订单号不重复，同时支持按订单号快速查询）；
多列组合唯一的场景，如课程表course中，teacher_id + course_time需唯一（同一老师同一时间不能上两节课），此时可建立“teacher_id + course_time”的组合唯一键索引

前缀索引

1. 核心定义

只对字符串类型（varchar/char/text）字段的前 N 个字符建立索引，而不是对整个字段值建立索引。

它是一种空间换时间的优化手段，本质上是牺牲部分查询能力，换取更小的索引体积和更快的索引维护速度。

2. 底层原理

MySQL 索引是 B+Tree 结构，每个索引节点存储的是索引键值。如果字段很长（比如邮箱、URL、文章摘要），完整存储会导致：

  • 索引节点能存储的键值数量大幅减少
  • B+Tree 高度增加，查询需要更多次磁盘 IO
  • 索引文件体积巨大，内存无法缓存全部索引

前缀索引只存储前 N 个字符，让每个索引节点能容纳更多键值，B+Tree 更矮更胖，查询更快。

4. 适用场景

必须用：长字符串字段（邮箱、URL、手机号、身份证号、文章标题）

推荐用：字段长度 > 20，且前 10-20 个字符区分度很高

适合查询：where email like 'xxx%'（前缀匹配）

5. 绝对不能用的场景

需要排序 / 分组：order by email、group by email（索引不包含完整字符串，无法排序）

需要覆盖索引：无法通过前缀索引直接获取完整字段值，必须回表

后缀匹配：where email like '%@gmail.com'（前缀索引完全没用）

区分度低的字段：比如所有 URL 都以 https://www. 开头，前 12 个字符完全相同

6. 常见坑

  • 前缀太短：区分度太低，导致查询时扫描大量行，性能反而下降
  • 前缀太长：失去了前缀索引节省空间的意义
  • 误以为能用于范围查询：只能用于精确匹配和前缀模糊匹配

单列索引

1. 核心定义

只对一个字段建立的索引，是最基础、最常用的索引类型。

包括：

  • 普通索引（INDEX）：最基本的索引，没有任何限制
  • 唯一索引（UNIQUE INDEX）：索引值必须唯一，但允许为 NULL
  • 主键索引（PRIMARY KEY）：特殊的唯一索引，不允许为 NULL，一张表只能有一个

2. 底层原理

B+Tree 结构，叶子节点存储：

  • InnoDB 主键索引：完整的数据行（聚簇索引）
  • InnoDB 普通 / 唯一索引：主键值（需要回表）
  • MyISAM 所有索引：数据的磁盘物理地址

3. 语法

4. 适用场景

单条件查询：where age = 18、where phone = '13800138000'

主键查询：where id = 1001（InnoDB 中最快的查询方式）

唯一约束：保证字段值唯一（比如手机号、身份证号）

5. 多个单列索引的问题（非常重要）

很多人以为：where age=18 and name='张三'，给 age 和 name 各建一个单列索引，MySQL 会同时用两个索引。

大错特错！

MySQL 确实有索引合并功能，但它有很大的局限性：

  • 只有在 and 条件下才可能触发
  • 性能远不如一个联合索引
  • 会消耗大量 CPU 进行索引合并计算
  • 大部分情况下，MySQL 会选择其中一个索引，另一个完全没用

结论：多个条件一起查询时，永远不要建多个单列索引，应该建联合索引。

6. 最佳实践

  • 主键必须用自增 INT/BIGINT，不要用 UUID（会导致严重的页分裂）
  • 唯一业务字段（手机号、身份证号）必须建唯一索引
  • 不要给低基数字段建单列索引（比如性别、状态，只有 2-3 个值）

联合索引

什么是复合索引？
类似于多个关键字匹配查询，就是你索引字段不止一个
联合索引（复合索引）是由多个字段组合而成的索引，在数据库中常以 B+树结构存储。以一个包含 (col1, col2, col3) 的联合索引为例，B+ 树节点首先按照 col1 的值进行排序，当 col1 的值相同时，再按照 col2 的值排序，若 col2 的值也相同，最后按照 col3 的值排序。

复合索引支持精准查询吗？支持范围查询吗？
复合索引对精准查询的支持较好，只要查询条件匹配最左前缀列就能有效利用索引加速查询。
对范围查询，只有当范围查询列处于复合索引的最左前缀列中，且后续列不影响索引使用的情况下，才能部分利用复合索引；如果范围查询列没有处于最左前缀，或者影响了后续列的索引使用，可能导致复合索引无法被使用。
复合索引的过程中一旦遇到范围查询，后面的字段将无法使用该复合索引进行查询（为什么）
数据库在使用联合索引进行查询时，会从最左边的字段开始，根据索引的有序性快速定位符合条件的记录范围。当遇到范围查询（如 >、<、BETWEEN 等）时，由于范围查询的结果不是一个确定的值，而是一个区间，这个区间内 col2 或者后续字段的值是无序的，数据库无法再利用索引的有序性来进一步快速定位后续字段的值，所以后面的字段无法再使用该联合索引进行查询。

什么是索引最左匹配原则？
在使用联合索引（复合索引，即由多个字段组成的索引）时，我的索引字段不止一个，那我应该先检索哪一个呢？
索引最左匹配原则指的就是，让我从最左边的字段开始检索，满足了这个字段，再检索第二个，像这样依次向右匹配，直到遇到范围查询（如 >、<、BETWEEN 等）或者条件不满足为止。

覆盖索引（Covering Index）

如果查询的所有字段都包含在联合索引中，MySQL 不需要回表，直接从索引中获取数据。

这是性能最高的查询方式，比普通查询快 10-100 倍。

联合索引字段顺序选择

三个原则，按优先级排序：

  1. 区分度高的字段放前面：比如 name 比 sex 区分度高，应该放前面
  2. 查询频率高的字段放前面：经常出现在 where 条件中的字段放前面
  3. 范围查询的字段放最后：因为范围查询后面的列无法命中索引

全文索引

什么是全文索引？
全文索引就是在一大段文字中去查找某个特定关键字
全文索引查找的对象是文本数据

全文索引和普通索引有什么区别？
（1）用途上的区别

全文索引主要用于需要在大量文本内容（如文章、博客、产品描述等）中查找包含特定关键词或短语的记录
例如，在一个新闻网站的文章数据库中，用户通过输入关键词搜索相关新闻报道，使用全文索引可以快速定位到包含这些关键词的文章。

普通索引主要用于精确匹配和范围查询。
普通索引常用于对数值型、日期型、字符串等字段进行快速查找，比如根据用户 ID 查询用户信息、根据订单日期范围查询订单记录等。在电商系统中，根据商品 ID 查找商品详情就可以使用普通索引提高查询速度。

（2）数据处理方式上的区别

全文索引会对文本内容进行分词处理，将文本拆分成一个个独立的单词或词组，并为这些词建立索引。 例如，对于句子 “我爱自然语言处理”，全文索引可能会将其分词为 “我”“爱”“自然语言处理” 等词，并分别记录它们在文档中的位置和出现频率等信息。

而普通索引直接对字段的值进行索引，不进行分词处理。
以用户表中的姓名字段为例，普通索引会将姓名作为一个整体进行索引，而不会对姓名进行拆分。

（3）索引结构的区别

全文索引通常采用倒排索引结构
倒排索引是一种从关键词到文档的映射关系，它记录了每个关键词在哪些文档中出现以及出现的位置等信息。这种结构使得在进行全文搜索时能够快速定位到包含关键词的文档。

普通索引常见的结构是B +树和哈希表
B - 树索引适用于范围查询和排序操作，它可以高效地支持大于、小于、等于等比较操作；哈希索引则基于哈希表实现，适用于精确匹配查询，能够在常数时间内完成查找操作。

mysql支持全文索引的条件是什么呢？
InnoDB 存储引擎支持全文索引，但该支持是从 MySQL 5.6 版本开始引入的。在这之前，MySQL 的全文索引功能主要由 MyISAM 存储引擎提供。
默认的全文索引仅支持英文，不支持中文。如果对中文进行全文检索，可以使用sphinx的中文版(coreseek)

全文索引采用的索引结构并不是B+树，而是倒排索引结构

索引优化

索引优化核心原则

  1. 索引是 “空间换时间”：加速查询，但会降低增删改性能（每次 DML 都要更新索引树）
  2. 索引不是越多越好：一张表索引数不要超过 10 个，单索引字段数不要超过 5 个
  3. 联合索引永远优于多个单列索引：多条件查询时，一个联合索引比 10 个单列索引都好用
  4. 覆盖索引是性能优化的终极手段：不需要回表的查询，速度提升 10-100 倍
  5. 优先优化最频繁、最慢的查询：不要为了 1% 的查询，优化 99% 的业务

怎么建索引

1. 主键索引设计（最关键）

必须用自增 INT/BIGINT

  • 自增主键是顺序插入，不会产生页分裂，性能最高
  • 主键越小越好，因为所有二级索引都携带主键值

绝对不要用 UUID / 字符串当主键

  • UUID 是无序的，插入会导致严重的页分裂，性能下降 10 倍以上
  • 字符串主键体积大，所有二级索引都会变得巨大

2. 单列索引设计

适合建单列索引的字段：

  • 经常出现在WHERE条件中的字段
  • 经常出现在ORDER BY、GROUP BY中的字段
  • 唯一业务字段（手机号、身份证号、订单号）

绝对不要建单列索引的字段：

  • 低基数字段（性别、状态、类型，只有 2-3 个值）
  • 很少被查询的字段
  • 频繁更新的字段

3. 联合索引设计（核心中的核心）

字段顺序黄金法则（按优先级排序）

  1. 区分度高的字段放前面

    ：区分度 = 不同值的数量 / 总记录数，越接近 1 越好

    • 例：name(区分度 0.9) > age(区分度 0.1) > sex(区分度 0.001)

  2. 查询频率高的字段放前面：经常单独出现在 WHERE 条件中的字段

  3. 范围查询的字段放最后：因为范围查询后面的列无法命中索引

4. 前缀索引设计

  • 只用于长字符串字段（邮箱、URL、文章标题）
  • 计算最佳前缀长度：区分度达到 0.9 以上

mysql查询优化器

MySQL 的查询优化器是基于成本的优化器（CBO），它会计算每个执行计划的成本，选择成本最低的那个。

成本主要包括：

  1. IO 成本：从磁盘读取数据的成本（占 90% 以上）
  2. CPU 成本：在内存中处理数据的成本

索引选择的优先级：

  1. 主键索引 > 唯一索引 > 普通索引
  2. 能覆盖查询的索引 > 需要回表的索引
  3. 区分度高的索引 > 区分度低的索引
  4. 扫描行数少的索引 > 扫描行数多的索引

为什么 MySQL 会选错索引？

  1. 统计信息不准确：MySQL 的统计信息是抽样统计的，不是精确值
  2. 成本模型的局限性：MySQL 的成本模型不能完全反映真实的硬件性能
  3. 数据分布不均匀：比如某个值出现的频率特别高

解决方案：

  1. 用ANALYZE TABLE更新统计信息
  2. 用FORCE INDEX强制指定索引
  3. 优化索引结构，让 MySQL 更容易选择正确的索引

MVCC机制

事务多版本并发控制，主要是为了提高数据库的并发性能

MVCC是MySQL中被广泛采用的并发控制机制，它通过版本管理来实现事务的隔离性，允许读写操作同时进行，提高数据库的并发性能和响应能力。

同一行数据平时发生读写请求时，会上锁阻塞住。但mvcc用更好的方式去处理读—写请求，做到在发生读—写请求冲突时不用加锁。

这个读是指的快照读，而不是当前读，当前读是一种加锁操作，是悲观锁。

当前读

读取数据库记录，都是当前最小的版本，会对当前读的数据进行加锁，防止其他事务修改数据。是悲观锁的一种操作。

以下操作为当前读

  • select lock in share mode (共享锁)
  • select for update (排他锁)
  • update (排他锁)
  • insert (排他锁)
  • delete (排他锁)
  • 串行化事务隔离级别

快照读

快照读实现基于多版本并发控制，即MVCC，快照读可能是之前历史版本的数据

不加select的操作（事务不是串行化）

快照读和MVCC

MVCC是维持一个数据的多个版本，使读写操作没有冲突的抽象概念，这个功能由MVCC实现。

数据库并发场景

  • 读-读：不存在任何问题，也不需要并发控制
  • 读-写：有线程安全问题，可能会造成事务隔离性问题，可能遇到脏读，幻读，不可重复读
  • 写-写：有线程安全问题，可能会存在更新丢失问题，比如第一类更新丢失，第二类更新丢失

MVCC用来解决那些并发问题？

问题 定义 发生场景
脏读 一个事务读到了另一个事务未提交的修改数据 读未提交 (RU) 级别
不可重复读 同一个事务内，两次读取同一行数据，结果不一样（中间被其他事务修改并提交） 读已提交 (RC) 级别
幻读 同一个事务内，两次相同的范围查询，返回的行数不一样（中间被其他事务插入 / 删除了符合条件的数据） 标准 SQL 的可重复读 (RR) 级别

MVCC用来解决读—写冲突的无锁并发控制，就是为事务分配单向增长的时间戳。为每个数据修改保存一个版本，版本与事务时间戳 相关联。

读操作只读取该事务开始前的数据库快照。

解决问题如下：

并发读-写时：可以做到读操作不阻塞写操作，同时写操作也不会阻塞读操作。MVCC 100% 解决了脏读

MVCC 在 InnoDB 可重复读 (RR) 级别解决了不可重复读

MVCC 单独不能解决幻读，InnoDB 是通过 MVCC + 间隙锁 (Next-Key Lock) 共同解决了幻读因此有了下面提高并发性能的组合拳：

MVCC + 悲观锁：MVCC解决读写冲突，悲观锁解决写写冲突
MVCC + 乐观锁：MVCC解决读写冲突，乐观锁解决写写冲突

Undo Log

Undo Log是mysql中的一种重要的事务日志

作用：

事务回滚：当事务需要回滚时，MySQL可以通过Undo日志中的旧值将数据还原到事务开始之前的状态，保证了事务回滚的一致性。

MVCC实现：MVCC 是InnoDB存储引擎的核心特性之一。通过使用Undo日志，MySQL可以为每个事务提供独立的事务视图，使得事务读取数据时能看到一致且符合隔离级别要求的数据版本。

在InnoDB存储引擎中，Undo日志分为两种：插入（insert）Undo日志 和 更新（update）Undo日志

**insert undo log：**插入Undo日志是指在插入操作中生成的Undo日志。由于插入操作的记录只对当前事务可见，对其他事务不可见，因此在事务提交后可以直接删除，无需进行purge操作。

**update undo log：**更新Undo日志是指在更新或删除操作中生成的Undo日志。更新Undo日志可能需要提供MVCC机制，因此不能在事务提交时就立即删除。相反，它们会在提交时放入Undo日志链表中，并等待purge线程进行最终的删除。删除操作只是设置一下老记录的 DELETED_BIT，并不真正将过时的记录删除，为了节省磁盘空间，InnoDB有专门的purge线程来清理 DELETED_BIT 为true的记录。
注意：由于查询操作（SELECT）并不会修改任何记录，所以在查询操作执行时，并不需要记录相应的 undo log 。

不同事务或者相同事务对同一记录行的修改，会使该记录行的 undo log 成为一条链表，链首就是最新的记录，链尾就是最早的旧记录

版本链

在MVCC中，对于每次更新操作，旧值会被保存到一条undo日志中，即使它是该记录的旧版本。随着更新次数的增加，所有的版本都会通过roll_pointer属性连接成一个链表，称之为版本链。

版本链的头节点代表当前记录的最新值。此外，每个版本还包含生成该版本的事务ID。

Read View

一致性视图，全称 Read View ，是用来判断版本链中的哪个版本对当前事务是可见的

Read View 说白了就是事务进行快照读操作时候生成的读视图（Read View），在该事务执行快照读的那一刻，会生成数据库系统当前的一个快照，记录并维护系统当前活跃事务的ID（每个事务开启时，都会被分配一个ID，这个ID是递增的）。

这里有一点要注意一下：Read View只针对 RC 和 RR级别

Read Uncommitted 和 Serializable 隔离级别下的事务规则不涉及基于 Read View 的可见性判断。RU 允许脏读，而 Serializable 则通过锁机制保证串行执行。因此，在这两个隔离级别下，不需要创建或使用 Read View。

可见性原则

Read View遵循一个可见性原则，将要被修改的数据的DB_TRX_ID取出来，与系统当前其他活跃事务ID去对比。

如果 DB_TRX_ID 跟 Read View 的属性做了某些比较，不符合可见性，那就通过 DB_ROLL_PTR 回滚指针去取出 Undo Log 中的 DB_TRX_ID 再比较。

即遍历链表的 DB_TRX_ID （从链首到链尾，即从最近的一次修改查起），直到找到满足特定条件的 DB_TRX_ID，那么这个 DB_TRX_ID 所在的记录就是当前事务能看见的最新老版本

Read View会维护以下字段

m_creator_trx_id：自己的 ID，自己改的自己能看到

m_up_limit_id：过去的边界，小于它的都能看到

m_low_limit_id：未来的边界，大于等于它的都看不到

m_ids：正在进行中的事务列表，里面的都看不到

可见性判断

如果被访问版本的 DB_TRX_ID 属性值与 Read View 中的 m_creator_trx_id 值相同，表示当前事务正在访问自己所修改的记录，因此该版本可以被当前事务访问。

如果被访问版本的 DB_TRX_ID 属性值小于 Read View 中的 m_up_limit_id 值，说明生成该版本的事务在当前事务生成 Read View 之前已经提交，因此该版本可以被当前事务访问。

如果被访问版本的 DB_TRX_ID 属性值大于或等于 Read View 中的 m_low_limit_id 值，说明生成该版本的事务在当前事务生成 Read View 之后才提交，因此该版本不能被当前事务访问。

如果被访问版本的 DB_TRX_ID 属性值位于 Read View 的 m_up_limit_id 和 m_low_limit_id 之间（包括边界），则需要进一步检查 DB_TRX_ID 是否在m_ids 列表中。如果在列表中，说明在创建ReadView时生成该版本的事务仍处于活跃状态，因此该版本不能被访问；如果不在列表中，说明在创建 Read View 时生成该版本的事务已经提交，因此该版本可以被访问。

RC 和 RR 下的 Read View
RC 和 RR 下生成 Read View 的时机是有所差异的：

RC：每次 SELECT 数据前都生成一个ReadView。
RR：只在第一次读取数据时生成一个ReadView，后面会复用第一次生成的。
正因为RC 和 RR生成 Read View 的时机不同，导致两个级别下看到的数据会不一致

对于版本链中的任意一个数据版本（trx_id是该版本的事务 ID）：

  1. 如果 trx_id == m_creator_trx_id → 可见（自己改的）

  2. 如果 trx_id < m_up_limit_id → 可见（过去的事务，已经提交）

  3. 如果 trx_id >= m_low_limit_id → 不可见（未来的事务，还没启动）

  4. 否则（m_up_limit_id <= trx_id < m_low_limit_id）

    ：

    • 如果 trx_id 在 m_ids 中 → 不可见（正在进行中的事务，未提交）
    • 如果 trx_id 不在 m_ids 中 → 可见（已经提交的事务）

一句话总结：

只能看到：自己改的 + 拍照前已经提交的

看不到：拍照时还在进行的 + 拍照后才启动的

RC和RR隔离级别的区别为什么 RC 会出现不可重复读？

还是上面的例子：

  1. 事务 F 第一次 SELECT，生成 Read View1，看不到事务 B (6) 的修改
  2. 事务 B 提交了
  3. 事务 F 第二次 SELECT：
    • RC 级别：生成新的 Read View2，m_ids变成[7,9]，事务 B (6) 不在m_ids中了 → 能看到事务 B 的修改 → 不可重复读
    • RR 级别：复用 Read View1，m_ids还是[6,7,9] → 还是看不到事务 B 的修改 → 可重复读
隔离级别 Read View 生成时机 四个字段的变化 结果
读已提交 (RC) 每次执行 SELECT 都会生成一个新的 Read View 每次都会重新计算m_up_limit_id、m_low_limit_id、m_ids 能看到其他事务刚提交的修改 → 不可重复读
可重复读 (RR) 事务中第一次执行 SELECT 时生成，整个事务期间复用 四个字段永远不变 永远只能看到第一次拍照时的内容 → 可重复读

MVCC+间隙锁

当一个事务既有当前读又有快照读时两者配合解决幻读

解决幻读

MVCC 负责解决「快照读」的幻读，间隙锁（Next-Key Lock）负责解决「当前读」的幻读。

为什么单独用 MVCC 不行？

MVCC 只能解决快照读（普通 SELECT）的幻读，但解决不了当前读的幻读。

  • 快照读：读取数据的历史版本，不加锁
  • 当前读：读取数据的最新版本，加锁（SELECT ... FOR UPDATE、INSERT、UPDATE、DELETE）

如果一个事务中既有快照读又有当前读，或者全是当前读，单独靠 MVCC 就会出现幻读

为什么单独用间隙锁不行？

如果所有读都用间隙锁，就会变成读写互斥，完全失去了 MVCC “读写不阻塞” 的高并发优势。

  • 读加共享锁，写加排他锁
  • 读阻塞写，写阻塞读
  • 性能会下降到和串行化级别差不多

所以，InnoDB 采用了最优组合：

  • 普通读用 MVCC，不加锁，性能高
  • 写操作和锁定读用间隙锁，保证一致性

误区：间隙锁只锁索引，不锁数据

正确：间隙锁是基于索引的，没有索引的列不会加间隙锁，只会加表锁。

  • 所以，一定要给查询条件的列建索引
  • 否则不仅会全表扫描，还会锁住整个表，性能极差

误区：间隙锁只会在范围查询时生效

正确：精确匹配查询也会加间隙锁。

  • 比如 SELECT * FROM user WHERE age=20 FOR UPDATE
  • 会加 (15, 20] 和 (20, 30] 两个间隙锁
  • 防止其他事务插入 age=20 的数据

误区：间隙锁会导致死锁

正确：间隙锁是导致 InnoDB 死锁的常见原因之一。

  • 死锁是两个或多个事务互相持有对方需要的锁，且都不释放自己的锁，导致无限等待的现象。InnoDB 会自动检测死锁并回滚代价最小的事务，但我们需要主动排查原因、优化代码和索引，从根本上避免死锁。因为间隙锁是共享的，多个事务可以同时持有同一个间隙的共享锁
  • 当两个事务都持有同一个间隙的共享锁，然后都试图插入数据时，就会发生死锁

误区：所有数据库的 RR 级别都解决了幻读

正确：只有 InnoDB 的 RR 级别解决了幻读。

  • 标准 SQL 的 RR 级别只解决了不可重复读，没有解决幻读
  • 其他数据库（如 Oracle、PostgreSQL）的 RR 级别仍然存在幻读

mysql死锁

死锁是两个或多个事务互相持有对方需要的锁，且都不释放自己的锁，导致无限等待的现象。InnoDB 会自动检测死锁并回滚代价最小的事务，但我们需要主动排查原因、优化代码和索引，从根本上避免死锁。

只有 InnoDB 会产生死锁：MyISAM 只有表锁，不会出现循环等待

InnoDB 会自动检测死锁：默认开启死锁检测，检测到后会回滚代价最小的事务（通常是更新行数少的那个）

死锁无法完全避免：只能通过优化降低发生概率

死锁的通用解决步骤

当生产环境发生死锁时，按照以下步骤处理：

  1. 查看死锁日志：用 SHOW ENGINE INNODB STATUS 找到最近一次死锁的详细信息
  2. 定位死锁的两个事务：看它们分别在执行什么 SQL
  3. 分析锁的持有和等待关系：找出循环等待的原因
  4. 根据场景选择解决方案：
    • 如果是更新顺序问题 → 统一更新顺序
    • 如果是间隙锁问题 → 降低隔离级别到 RC
    • 如果是先查后改问题 → 直接用 UPDATE
    • 如果是没有索引 → 建索引
    • 如果是长事务 → 拆分事务
  5. 验证优化效果：观察是否还会发生死锁

预防

统一所有事务的更新顺序：这是解决死锁最有效的方法

优先使用读已提交 (RC) 隔离级别：避免间隙锁导致的死锁，同时性能更好

避免长事务：事务越短，持有锁的时间越短，死锁概率越低

给所有查询条件建索引：避免全表扫描和全表锁

尽量用小事务：只在事务中包含必要的数据库操作

避免在事务中做耗时操作：不要在事务中调用外部接口、文件 IO 等

尽量用 UPDATE 代替 SELECT ... FOR UPDATE + UPDATE：直接加排他锁，避免锁升级

控制并发粒度：不要让太多事务同时更新同一批数据
