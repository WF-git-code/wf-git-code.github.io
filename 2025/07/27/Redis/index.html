SET命令

Redis 中的 Set 类型具有与 Java 中的 HashSet 类似的特性，它们都是基于哈希表实现的集合结构，具备以下特征：

无序性：集合中的元素没有特定的顺序，即不像列表（List）类型那样按照插入顺序存储元素。
唯一性：集合中不允许重复的元素，每个元素在集合中是唯一的。
快速查找：集合内部使用哈希表来存储元素，因此查找、插入、删除操作的时间复杂度都是 O(1)。
支持集合操作：Redis 的 Set 类型支持多种集合操作，如交集、并集、差集等，这使得可以方便地对多个集合进行组合和计算。

SOETEDSET命令

Redis的SortedSet是一个可排序的set集合，SortedSet中的每一个元素都带有一个score属性，可以基于score属性对元素排序，底层的实现是一个跳表（SkipList）加 hash表。

可排序性：Sorted Set 中的每个元素都关联一个分数（score），Redis 根据这个分数对集合中的元素进行排序。分数可以是浮点数，允许元素按照从低到高或从高到低的顺序排列。
元素唯一性：与普通 Set 类型一样，Sorted Set 中的元素是唯一的，不允许重复。
快速查询：Redis 使用跳表（SkipList）和哈希表来实现 Sorted Set，这使得对元素的插入、删除和获取操作的时间复杂度为 O(log(N))，其中 N 是集合中的元素数量。这比传统的平衡二叉树要快，并且在元素数量较大时仍能保持高效。

应用场景：

排行榜 优先级队列 范围查询 带权重的数据集合

所有的排名默认都是升序，如果要降序则在命令的Z后面添加REV即可

HyperLogLog算法

HyperLogLog 是一种用于估计集合中唯一元素数量（基数）的算法，它通过使用固定的内存空间来实现对大数据集合的基数估计，而不需要准确存储每个元素。

HyperLogLog 的原理

HyperLogLog 使用了一种称为基数估计器（cardinality estimator）的数据结构，其核心思想是：

哈希函数：首先，将每个元素通过哈希函数映射到一个固定位数的二进制串中。
前导零计数：接着，统计这些哈希值的前导零的数量。前导零是指从左边开始数，连续的零的数量。
估计基数：根据前导零的最大数量，估算集合中不同元素的数量。

为什么需要分布式锁？

分布式锁是用于在分布式系统中实现同步和互斥访问的机制。分布式系统中，多个节点同时访问共享资源可能会导致数据不一致或竞争条件发生。分布式锁提供了一种保护共享资源的方式，确保任意时刻只有一个节点可以访问该资源

分布式锁就是为了解决分布式环境下多个进程之间的互斥访问问题，保证同一时间只有一个进程可以操作共享资源。

分布式锁的核心要求

  1. 互斥性：同一时间只能有一个客户端持有锁
  2. 高可用：锁服务本身不能单点故障
  3. 可重入性：同一个客户端可以多次获取同一把锁
  4. 锁超时：防止客户端宕机导致锁永远无法释放
  5. 公平性：可选，按照请求顺序获取锁
  6. 高性能：加锁和释放锁的操作要快

这样实现存在问题

不可重入性：****

基础 SETNX 锁的逻辑是：只要 key 存在，就认为锁被持有，不管是谁持有。

  • 同一个线程已经持有锁，再次调用lock()时，SETNX会返回 0，加锁失败
  • 这会导致同一个线程自己把自己锁住，甚至死锁

解决：

用 Redis 的 Hash 结构代替 String 结构：

  • key：锁的名称
  • field：线程唯一标识
  • value：重入次数
加锁逻辑
  1. 如果锁不存在（EXISTS key == 0）：创建 Hash，设置 field 为线程标识，value 为 1，加过期时间 → 加锁成功
  2. 如果锁存在，且当前线程的 field 存在（HEXISTS key thread_id == 1）：重入次数 + 1，刷新过期时间 → 加锁成功
  3. 否则：加锁失败
解锁逻辑
  1. 如果当前线程的 field 不存在：解锁失败（不是自己的锁）
  2. 重入次数 - 1
  3. 如果重入次数 > 0：刷新过期时间 → 解锁完成（锁还没释放）
  4. 如果重入次数 == 0：删除整个 key → 锁完全释放

无法续期：

基础 SETNX 锁的过期时间是固定的，如果业务执行时间超过了过期时间：

  1. 锁会自动释放
  2. 其他线程可以加锁成功
  3. 原来的线程还在执行临界区代码
  4. 最终导致两个线程同时持有锁，并发操作共享资源

解决方案：看门狗自动续期机制

启动一个后台守护线程（看门狗）：

  1. 加锁成功后，启动看门狗
  2. 每隔过期时间的 1/3（比如过期时间 30 秒，每隔 10 秒），给锁续期到原始过期时间
  3. 只要业务线程还在运行，看门狗就会一直续期
  4. 业务线程执行完解锁，或者业务线程崩溃，看门狗会自动停止
  5. 看门狗停止后，锁会在过期时间后自动释放

集群环境锁丢失：

Redis 主从复制是异步的：

  1. 线程 A 在主节点加锁成功
  2. 主节点还没来得及把锁同步到从节点，就宕机了
  3. 从节点升级为新的主节点
  4. 新的主节点上没有这个锁
  5. 线程 B 可以在新的主节点上加锁成功
  6. 最终导致两个线程同时持有锁

解决方案

方案 1：使用 Redis WAIT 命令（推荐，简单有效）

Redis 3.0 以后支持WAIT命令，可以等待主节点将写操作同步到指定数量的从节点后再返回。

  • 加锁成功后，执行WAIT num_slaves timeout
  • 等待至少num_slaves个从节点同步完成
  • 只有同步成功，才算加锁真正成功
方案 2：红锁算法（RedLock，强一致性场景）

红锁是 Redis 官方提出的分布式锁算法，用于解决集群环境下的锁丢失问题：

  1. 部署N 个完全独立的 Redis 节点（N 一般为奇数，比如 5 个）
  2. 客户端同时向所有 N 个节点请求加锁
  3. 只有在超过半数（N/2+1）的节点上加锁成功，才算加锁成功
  4. 如果加锁失败，向所有节点释放锁

优点：只要不超过半数节点宕机，锁就不会丢失

缺点：实现复杂，性能有一定损失

Redis集群

主从复制

主从复制解决高并发问题

主从复制指将一台redis服务器的数据，复制到其他redis服务器

数据复制是单向的，只能由主节点—》从节点

单台redis并发能力是有限的，因此需要读写分离

一个主节点可以有多个从节点（或没有从节点），但一个从节点只能有一个主节点

主节点负责写数据，从节点负责读数据

A.B.C三台电脑搭建主从架构

①三台电脑都安装redis，将A电脑redis配置文件复制并覆盖到另外两个节点的redis配置文件，保证redis状态的同步，然后启动三台redis

②开启主从关系：要配置主从可以使用replicaof或者slaveof命令

永久配置：在从节点的Redis配置文件（通常是redis.conf）中添加一行配置：slaveof 。其中，后两个参数分别是主节点的ip地址和Redis端口号。重启redis服务

临时配置：使用redis-cli客户端连接到redis服务，执行slaveof命令（重启后失效）：slaveof 。

验证：登录主库查看从库连接状态，在从库查看复制状态

同步原理

全量同步

Replication_ID:是数据集的标记（40位的十六进制随机字符串），ID一致则说明是同一数据集，每一个master都有唯一的replid,salve会继承master节点的replid

offset：偏移量，数据的进度条，offset是一个整数，单位是字节，标识数据在复制流中的位置

主库每执行一个写命令，就会把命令写入复制缓冲区，同时增加自己的偏移量，从库每执行一个主库命令也会增加自己的偏移量，主从之间的offset差值就是主从延迟的字节数

复制缓冲区（repl_backlog）：offset是和复制缓冲区紧密配合的：

复制缓冲区是主库固定大小的一个环形缓冲区，默认1MB，它保存了主库最近执行的所有写命令

repl_backlog_first_byte_offset：缓冲区中第一个字节的 offset

repl_backlog_histlen：缓冲区中当前保存的数据长度

当从库断连重连时，主库会检查从库发送的 offset 是否在[repl_backlog_first_byte_offset, master_repl_offset]这个范围内。如果在，就只发送 offset 之后的命令（增量复制）；如果不在，说明缓冲区已经被覆盖，只能触发全量复制。

master判断发现slave发送来的Replication Id与自己的不一致，说明这是一个全新的slave，就知道要做全量同步了。此外，master会将自己的Replication Id和offset都发送给这个slave，因此从，经过第一次全量同步之后，slave的Replication Id就和master的一样了。

增量同步

通过BGSAVE是一个比较消耗性能的操作，除了第一次做全量同步，其他大多数时候都是做增量同步

增量同步就是只更新slave和master存在差异的部分数据（与offset有关）

优点：解决了系统的高并发读的问题。
缺点：无法保证系统的高可用，所以哨兵模式出现了。

哨兵模式

主从同步解决了高并发读的问题，但无法保证高可用，所以哨兵模式出现了

作用：

**监控：**Sentinel 会不断检查您的master和slave是否按预期工作。
**自动故障恢复：**如果master故障，Sentinel会将一个slave提升为master。当故障实例恢复后也以新的master为主。
**通知：**Sentinel充当Redis客户端的服务发现来源，当集群发生故障转移时，会将最新信息推送给 Redis的客户端。

哨兵的监控

哨兵如何进行监控（心跳机制）

每个一秒向集群的每个实例发送Ping命令

主管下线：如果某哨兵节点发现某实例未在规定时间响应，则认为该实例主管下线

客观下线：若超过指定数量（quorum）的sentinel都认为该实例主观下线，则该实例客观下线。quorum值最好超过Sentinel实例数量的一半。

选主规则

一旦发现主节点客观下线了。哨兵会推举新的主节点，选主规则如下：

判断主与从节点断开时间长短，如超过指定值就排除该从节点
然后判断从节点的slave-priority值，越小优先级越高
如果slave-prority一样，则判断slave节点的offset值，越大（说明从节点数据与主节点越相近） 优先级越高

总结下来就是，先排除断开过久的，然后看提前设置的优先值，最后看数据丢失的情况。

脑裂

如果此时原本的主节点（暂时称为A）因为网络问题，使得主从节点处在不同的网络分区,哨兵监测不到主节点（没有回应心跳），那么哨兵便会进行选举出一个新的主节点（暂时称为B），这样就存在了两个主节点，像是大脑分两列了一样。等A节点网络恢复之后才会由主节点降为从节点。这个过程称为脑裂。

当然如果是哨兵监测不到从节点，则会去除这个从节点

从节点A向主节点同步数据时，由于发现数据不一致，就会删除自己原来的数据，进行同步，造成数据丢失的问题

解决

解决方案有两种，对应着redis的两个配置参数：

min-slaves-to-write：设置主节点在至少有指定数量的从节点确认写操作的情况下才执行写操作
min-salves-max-lag：设置从节点的最大延迟（以秒为单位），如果从节点的延迟超过这个值，则该从节点不会被计入 min-slaves-to-write 的计数中
举个例子：当 min-slaves-to-write 设置为 2，min-slaves-max-lag 设置为 10 秒时，主节点只有在至少有 2 个从节点延迟不超过 10 秒的情况下才会接受写操作。

这两个参数就使得发生脑裂的时候，如果某个主节点跟随的从节点数量不够或延迟较大，就无法被写入，这样就能避免脑裂导致的数据不一致。

建议集群部署奇数个节点，例如集群数为 5，那么可以设置 min-slaves-to-write 为 3，min-slaves-max-lag 为 5-10 秒。

分片集群

集群中有多个master，每个master保存不同数据，且每个master都可以有多个slave节点。这样就解决了海量数据存储，高并发读写的问题。相当于把主从模式概括进来了。
不再需要哨兵，直接master之间通过ping监测彼此健康状态。只要超过一定数量的master节点认为某个master节点宕机了，那么那个节点就客观下线了。相当于变形的哨兵模式。
客户端请求可以访问集群任意节点，经过一定的路由规则，最终都会被转发到正确节点。

路由规则

Redis 分片集群引入了哈希槽的概念，[Redis 集群](https://so.csdn.net/so/search?q=Redis 集群&spm=1001.2101.3001.7020)有 16384 个哈希槽，每个 key通过 CRC16 校验后对 16384 取模来决定放置哪个槽，集群的每个节点负责一部分 hash 槽。这样能保证客户端请求不冲突地正确转发到redis的某个master节点上。

优点：解决了系统的海量数据存储、高可用、高并发读写的问题。
缺点：集群维护很麻烦，而且集群之间的通信和心跳检测消耗大量的网络带宽，无法使用lua脚本和事务。

分片导致数据分散在多台机器，单机的事务/lua原子执行无法跨机器生效

1. Redis集群有哪些方案？

Redis 主流集群方案共三种：

  • 主从复制集群：基础集群架构，实现读写分离，提升并发能力，无故障自动转移能力。
  • 哨兵（Sentinel）集群：基于主从架构，增加哨兵节点，实现故障监控、自动故障转移，保障高可用。
  • 分片集群（Cluster集群）：多主多从架构，解决海量数据存储、高并发读写问题，支持数据分片分布式存储。

2. 简单介绍一下Redis主从同步

单节点Redis并发能力、数据容灾能力有限，因此可搭建一主多从的主从集群，核心实现读写分离，大幅提升并发处理能力。

架构分工：主节点（Master）负责处理写请求，从节点（Slave）负责处理读请求；主节点写入数据后，会自动将数据同步到所有从节点，保证主从数据一致性。

3. 详细说一下Redis主从同步的数据流程

Redis主从同步分为全量同步和增量同步两个阶段，适配不同同步场景：

（1）全量同步（首次同步）

从节点第一次连接主节点时触发，完整同步主节点所有数据，流程如下：

  1. 从节点向主节点发起数据同步请求，携带自身的复制ID（replication id）和数据偏移量（offset）。
  2. 主节点对比复制ID，判断与从节点不属于同一同步链路，判定为首次同步，将自身的复制ID和偏移量同步给从节点。
  3. 主节点执行 bgsave 命令，生成RDB持久化文件，发送给从节点。
  4. 从节点清空本地原有数据，加载并执行RDB文件，完成基础数据同步。
  5. RDB生成、传输期间，主节点新产生的读写命令会记录在**repl_backlog（复制缓冲区日志）**中，RDB同步完成后，主节点将缓冲区日志发送给从节点，同步增量命令，保证主从数据完全一致。
（2）增量同步（日常/重启同步）

适用于从节点重启、网络短暂中断后的后续同步场景，无需全量拷贝数据：

  1. 重启后的从节点再次连接主节点，发起同步请求并携带本地offset偏移量。
  2. 主节点识别为非首次同步，根据从节点的offset，从复制缓冲区日志中截取偏移量之后的所有新增命令。
  3. 主节点将增量命令发送给从节点，从节点执行命令，快速同步缺失数据，恢复数据一致性。

4. 如何保证Redis的高并发、高可用？

通过主从集群 + 哨兵模式组合架构，同时实现高并发和高可用：

  1. 保障高并发：搭建一主多从主从架构，实现读写分离，主节点扛写请求、从节点分摊读请求，突破单节点并发上限。
  2. 保障高可用：引入哨兵节点，实现三大核心能力：
    1. 监控：实时监测主、从节点的健康状态；
    2. 自动故障转移：若主节点故障，哨兵会通过选举机制，将健康的从节点升级为新主节点；故障旧主节点恢复后，自动降级为新主的从节点；
    3. 服务发现：实时向客户端推送集群最新节点信息，保证客户端正常访问。

该架构稳定、运维简单，是企业项目中保证Redis高并发、高可用的主流方案。

5. 项目中Redis用的是单点还是集群？哪种集群架构？

项目中使用的是1主1从 + 哨兵集群架构，不轻易使用分片集群。

选型理由：

  1. 单节点Redis内存控制在10G以内，满足常规业务需求；若内存不足，会为不同业务服务配置独立的主从+哨兵节点，横向拆分压力。
  2. 分片集群运维成本高，节点间心跳检测、数据同步会消耗大量网络带宽；且分片集群存在功能限制，不支持Lua脚本、多键事务，容易引发业务问题。

6. 什么是Redis集群脑裂？如何解决？

（1）脑裂问题原理

集群出现网络分区时，旧主节点与哨兵、从节点网络断开，哨兵无法检测到旧主节点心跳，判定主节点故障，通过选举将一个从节点升级为新主节点。

此时集群会同时存在旧主、新主两个主节点，即集群脑裂。客户端可能继续向旧主节点写入数据，新主节点无法同步该部分数据。待网络恢复后，哨兵会将旧主降级为从节点，旧主会清空本地数据、同步新主数据，最终导致数据丢失。

（2）解决方案

通过修改Redis核心配置，规避脑裂导致的大量数据丢失问题：

  1. 配置最小从节点同步数量，主节点必须满足指定数量的从节点正常同步，才接收写请求；
  2. 限制主从数据同步延迟时间，设置最大同步延迟不超过5秒，未达到同步一致性要求的主节点，直接拒绝客户端写请求。

该配置可有效避免网络分区期间，孤立旧主节点接收大量写数据，极大减少数据丢失量。

7. Redis分片集群的核心作用是什么？

Redis分片集群（Cluster集群）核心解决两大痛点：海量数据存储和超高并发读写，是大数据量、高流量业务的首选架构：

  1. 数据分片存储：集群包含多个主节点，每个主节点独立存储一部分数据，打破单节点内存上限，支持海量数据分布式存储。
  2. 提升并发能力：每个主节点可搭配多个从节点，多主多从架构可同时处理读写请求，并发能力远高于哨兵架构。
  3. 集群自愈监控：各主节点通过ping心跳机制互相监测健康状态，具备故障感知能力；客户端可访问集群任意节点，请求会自动转发至对应数据节点。

8. Redis分片集群的数据存储和读取原理

分片集群通过哈希槽机制实现数据分布式存储与精准读取，核心原理如下：

  1. Redis集群预设 16384个固定哈希槽，集群内每个主节点会绑定一段专属的哈希槽区间。
  2. 数据存储时：对Key执行CRC16算法校验，将结果对16384取模，计算出该Key对应的哈希槽编号，最终将Key存入对应槽位绑定的主节点。
  3. 数据读取时：客户端根据相同算法计算Key的哈希槽，定位到对应的集群节点，读取数据；若访问集群任意随机节点，该节点会自动转发请求至目标节点，完成数据查询。

缓存穿透、击穿、雪崩

1. 什么是缓存穿透？如何解决？

（1）定义与产生原因

缓存穿透是指查询一个缓存和数据库中都不存在的数据，导致每次请求都绕过缓存，直接打到数据库，给数据库造成巨大压力。

根本原因：缓存层没有起到 “挡箭牌” 的作用，不存在的数据无法被缓存，每次查询都会穿透到数据库。

常见场景：

  • 恶意攻击：黑客构造大量不存在的 ID（如负数、超大数）发起请求
  • 业务 bug：前端 / 后端参数校验不严格，传入非法参数
  • 数据被物理删除：数据库中数据已删除，但缓存未同步清理，后续查询仍会穿透

（2）解决方案（按优先级排序）

  1. 接口层参数校验（第一道防线）
    • 对所有入参做合法性校验，如 ID 范围、格式、长度等
    • 示例：用户 ID 必须是正整数，直接拦截负数和 0 的请求
  2. 缓存空值 / 默认值（最简单通用）
    • 当数据库查询结果为空时，也将空值（如null、空字符串）写入缓存，并设置较短的过期时间（如 30 秒 - 5 分钟）
    • 优点：实现简单，成本低
    • 缺点：会占用少量内存，可能存在短暂的数据不一致（如数据库新增了该数据）
  3. 布隆过滤器（Bloom Filter）（高并发防恶意攻击首选）
    • 原理：将数据库中所有存在的 key 预先存入布隆过滤器，查询时先经过布隆过滤器
    • 若布隆过滤器判断 key 不存在，则直接返回，不查询缓存和数据库
    • 优点：内存占用极小，性能极高，能有效拦截 99% 以上的非法请求
    • 缺点：存在误判率（可能把不存在的 key 判断为存在），不支持删除操作
  4. 黑名单限流（兜底方案）
    • 对短时间内发起大量非法请求的 IP 进行限流或拉黑
    • 结合网关层（如 Nginx、Spring Cloud Gateway）实现

2. 什么是缓存击穿？如何解决？

（1）定义与产生原因

缓存击穿是指一个热点 key 在过期的瞬间，有大量并发请求同时查询该 key，导致所有请求都打到数据库，数据库压力骤增。

根本原因：热点 key 过期，且缓存重建时间较长，大量并发请求同时触发缓存重建。

关键区别：和缓存穿透不同，击穿的 key 是真实存在的，只是缓存过期了；和雪崩不同，击穿是单个热点 key 失效。

常见场景：

  • 秒杀活动的商品详情、库存信息
  • 热门新闻、热搜榜单
  • 高频访问的系统配置

（2）解决方案（按适用场景排序）

  1. 热点 key 永不过期（最简单粗暴，推荐）
    • 不给热点 key 设置过期时间，由后台异步线程定期更新缓存
    • 优点：实现简单，不会出现击穿问题
    • 缺点：占用内存，需要额外的异步更新逻辑
  2. 互斥锁（Mutex Lock）（通用方案）
    • 当缓存失效时，不是所有请求都去查询数据库，而是只有第一个请求能获取到分布式锁，去查询数据库并重建缓存
    • 其他请求获取锁失败后，等待一段时间（如 100ms），重试查询缓存
    • 优点：实现简单，能保证只有一个请求重建缓存
    • 缺点：存在死锁风险，高并发下会有线程阻塞，性能略有下降
  3. 逻辑过期（高并发极致性能方案）
    • 不设置缓存的物理过期时间，而是在 value 中额外存储一个逻辑过期时间
    • 当查询到逻辑过期时，立即返回旧数据，同时异步启动一个线程去重建缓存
    • 优点：性能极高，不会有线程阻塞，所有请求都能快速返回
    • 缺点：实现复杂，需要维护逻辑过期时间，会存在短暂的数据不一致

3. 什么是缓存雪崩？如何解决？

（1）定义与产生原因

缓存雪崩是指大量缓存 key 同时过期，或者缓存服务整体宕机，导致所有请求都打到数据库，数据库压力瞬间达到峰值，甚至宕机。

根本原因：缓存层整体不可用或大面积失效，失去了对流量的承载能力。

两种类型：

  • 过期雪崩：大量 key 设置了相同的过期时间，到期时同时失效
  • 服务雪崩：Redis 集群宕机，整个缓存层不可用

常见场景：

  • 定时任务批量更新缓存，所有 key 的过期时间相同
  • 促销活动结束后，大量活动相关的 key 同时过期
  • Redis 主节点宕机，从节点切换不及时或数据丢失

（2）解决方案（按优先级排序）

  1. 过期时间加随机值（解决过期雪崩首选）
    • 给所有缓存 key 的过期时间加上一个随机偏移量（如 1-5 分钟）
    • 避免大量 key 在同一时间点过期，将缓存失效的时间分散开
    • 示例：expireTime = baseTime + random.nextInt(300)
  2. Redis 集群高可用（解决服务雪崩根本方案）
    • 搭建 Redis 主从 + 哨兵集群，实现自动故障转移
    • 生产环境建议使用 Redis Cluster 分片集群，避免单节点故障影响整个集群
    • 做好 Redis 的监控和告警，提前发现节点异常
  3. 服务熔断与降级（兜底方案）
    • 当数据库压力达到阈值时，触发熔断机制，暂时关闭非核心接口
    • 对核心接口返回默认值、降级数据或提示信息
    • 结合 Hystrix、Sentinel 等熔断组件实现
  4. 多级缓存架构
    • 搭建 “本地缓存（Caffeine）+ Redis 缓存” 的多级缓存架构
    • Redis 宕机时，本地缓存可以承接一部分流量，避免所有请求直接打到数据库
  5. 限流
    • 在网关层或服务层对请求进行限流，保护数据库不被打垮
    • 采用令牌桶、漏桶等限流算法，控制每秒的请求量

5. 实际项目中的最佳实践

  1. 组合使用多种方案：不要依赖单一方案，例如：接口参数校验 + 缓存空值 + 布隆过滤器 防穿透；热点 key 永不过期 + 互斥锁 防击穿；过期时间加随机值 + Redis 高可用 防雪崩。
  2. 提前识别热点 key：通过监控 Redis 的访问量，提前发现热点 key，对其进行特殊处理（如永不过期、本地缓存）。

  4. 做好容灾演练：定期进行 Redis 宕机、数据库压力测试等容灾演练，验证系统的容错能力。

通过写实复制的方式，fork之后子进程共享主进程内存数据，主进程执行读操作时，访问共享内存，主进程执行写操作则会拷贝一份数据，执行写操作。（只复制页表特换自己的页表，不复制物理内存数据，达到异步持久化效果）

事务

1. 什么是 Redis 事务？它的核心设计目标是什么？

Redis 事务是一组 Redis 命令的集合，将多个命令打包，作为一个整体按顺序执行，保证：

  • 事务内的所有命令要么全部执行，要么全部不执行（原子性的有限实现）
  • 事务执行过程中不会被其他客户端的命令打断，所有命令串行执行

Redis 事务的核心设计目标是保证多个命令的串行原子执行，而非传统关系型数据库的强 ACID 事务，它牺牲了部分特性换取极致性能。

2. Redis 事务的常用命令和完整执行流程是什么？

（1）核心命令

命令 作用
MULTI 开启事务，标记事务开始
任意 Redis 命令 事务开启后，所有命令不会立即执行，而是进入事务队列排队
EXEC 提交事务，按顺序执行事务队列中的所有命令，返回所有命令的执行结果
DISCARD 放弃事务，清空事务队列，退出事务状态
WATCH key [key...] 乐观锁，监控一个或多个 key，若事务执行前这些 key 被其他客户端修改，事务将被打断
UNWATCH 取消对所有 key 的监控

（2）完整执行流程

  1. 开启事务：客户端执行MULTI命令，Redis 将客户端标记为事务状态

  2. 命令入队：客户端发送的所有读写命令不会立即执行，Redis 会校验命令语法，语法正确则加入事务队列，返回QUEUED；语法错误则直接报错，事务后续无法提交

  3. 提交 / 放弃事务

    ：

    • 执行EXEC：Redis 按入队顺序依次执行队列中的所有命令，返回所有命令的执行结果列表，事务结束
    • 执行DISCARD：清空事务队列，客户端退出事务状态，事务结束

3. Redis 事务是否满足 ACID 特性？请分别说明

Redis 事务仅部分满足 ACID 特性，与传统关系型数据库有本质区别：

（1）原子性（Atomicity）：有限支持

  • ✅ 若事务执行前（命令入队阶段）出现语法错误，整个事务会被拒绝，所有命令都不执行
  • ✅ 若事务执行EXEC时，Redis 进程崩溃 / 断电，未执行的命令不再执行，已执行的命令会持久化（若开启 RDB/AOF）
  • ❌ 若事务执行过程中，某条命令出现运行时错误（如对字符串执行 LPUSH），Redis 会跳过错误命令，继续执行后续命令，不支持回滚

（2）一致性（Consistency）：满足

事务执行前后，Redis 的数据始终符合其自身的约束规则（如数据类型约束、过期时间等），不会出现数据损坏或不一致的情况。

（3）隔离性（Isolation）：完全满足

Redis 是单线程执行模型，事务执行过程中不会被其他客户端的任何命令打断，所有命令串行执行，天然具备串行化隔离级别。

（4）持久性（Durability）：取决于持久化配置

  • 若未开启任何持久化（RDB/AOF），事务执行后数据仅存在于内存，Redis 重启后数据全部丢失，不满足持久性
  • 若开启 RDB：默认配置下，RDB 快照间隔较长，事务执行后可能未触发快照，存在数据丢失风险
  • 若开启 AOF 且appendfsync=always：每条命令执行后立即刷盘，事务完全满足持久性，但性能会大幅下降

4. WATCH 命令的作用和实现原理是什么？

（1）作用

实现乐观锁机制，解决并发场景下的竞态条件问题。例如：实现秒杀库存扣减时，防止多个客户端同时修改同一个库存 key 导致超卖。

（2）实现原理

  1. Redis 内部维护一个watched_keys字典，记录每个被监控的 key 以及监控它的客户端
  2. 当某个 key 被修改（执行写命令）时，Redis 会遍历watched_keys，将所有监控该 key 的客户端标记为事务脏状态
  3. 当标记为脏状态的客户端执行EXEC提交事务时，Redis 会直接拒绝事务执行，返回nil，表示事务失败

（3）注意事项

  • WATCH的监控有效期仅到本次事务结束（执行EXEC或DISCARD后自动取消监控）
  • 若在WATCH之后执行MULTI之前，key 被修改，事务同样会失败
  • 不支持嵌套 WATCH，多次执行WATCH会累加监控的 key

5. Redis 事务为什么不支持回滚？

Redis 官方明确不支持事务回滚，核心原因有三点：

  1. 性能优先：回滚机制需要维护复杂的事务日志和回滚逻辑，会大幅降低 Redis 的性能，违背了 Redis"极致性能" 的设计目标
  2. 错误可预判：Redis 事务的错误绝大多数是语法错误（入队阶段即可发现）和开发人员逻辑错误（如对错误的数据类型执行命令），这些错误在开发阶段就能避免，不需要运行时回滚
  3. 简化实现：不支持回滚让 Redis 事务的实现变得极其简单，代码量少、稳定性高，符合 Redis"简单高效" 的设计哲学

6. Redis 事务有哪些局限性？实际项目中如何替代？

（1）核心局限性

  1. 不支持回滚：运行时错误无法回滚，可能导致部分命令执行、部分未执行，数据不一致
  2. 无法处理复杂逻辑：事务内只能执行简单的 Redis 命令，不支持条件判断、循环等复杂逻辑
  3. 不支持跨节点事务：在 Redis 分片集群中，事务只能作用于同一个节点上的 key，无法跨节点执行
  4. WATCH 乐观锁效率低：高并发场景下，大量事务会因 key 被修改而失败，重试成本高

（2）主流替代方案：Lua 脚本

Redis 2.6 及以上版本支持执行 Lua 脚本，Lua 脚本是 Redis 事务的最佳替代方案，具备以下优势：

  • ✅ 强原子性：整个 Lua 脚本作为一个整体执行，中间不会被其他命令打断，完全保证原子性
  • ✅ 支持复杂逻辑：可以在 Lua 脚本中编写条件判断、循环、变量运算等复杂业务逻辑
  • ✅ 性能更高：一次网络往返可以执行多个命令，减少网络 IO 开销
  • ✅ 支持跨槽位操作：在分片集群中，只要所有 key 都在同一个哈希槽（可通过{}指定哈希标签），Lua 脚本就能正常执行

跨节点支持 不支持 支持同哈希槽跨节点
错误处理 语法错误终止事务，运行时错误跳过 脚本执行出错立即终止，已执行命令不回滚
适用场景 简单的多命令串行执行 复杂业务逻辑、高并发原子操作

数据一致性

  • 强一致性：任何时刻，任何客户端看到的 Redis 和 MySQL 数据完全相同（性能极差，极少使用）
  • 最终一致性：允许短暂不一致（通常毫秒到分钟级），但经过一段时间后，数据最终会达到一致（99% 互联网业务首选）
  • 因果一致性：有因果关系的操作必须保持一致（如用户下单后必须能看到自己的订单）

核心原则：缓存只是数据库的 “副本”，用于查询加速，永远以 MySQL 数据库的数据为准。

所有不一致问题本质上都是两个独立系统（Redis 和 MySQL）的操作无法原子执行导致的。

为什么会产生数据一致性？

1. 读流程的不一致

问题：在 “查 MySQL” 和 “写回 Redis” 之间，有其他写操作更新了 MySQL，导致 Redis 写入旧值。

2. 写流程的不一致（最常见）

更新 MySQL 和操作缓存的顺序、网络延迟、并发冲突都会导致不一致：

  • ❌ 先删缓存，再更新数据库：并发必产生脏数据
  • ⚠️ 先更新数据库，再删缓存：极端并发下仍有脏数据
  • ❌ 更新数据库后更新缓存：无效 IO 多，并发脏数据概率极高
  • ⚠️ 更新数据库成功，删缓存失败：永久脏数据

3. 异常场景的不一致

  • 数据库事务提交成功，但应用服务在删缓存前宕机
  • Redis 主从同步延迟，读从库读到旧数据
  • 网络分区导致缓存删除请求丢失
  • Redis 宕机重启后，缓存数据丢失

解决

为什么是 “删缓存” 不是 “更新缓存”？

  1. 避免无效更新：连续更新 10 次数据，缓存只需要最后一次有效
  2. 降低并发冲突：更新缓存会导致多个写请求同时修改缓存，产生脏数据
  3. 性能更高：删除是 O (1) 操作，比更新更轻量

经典脏数据场景（概率极低但真实存在）

  1. 线程 A 读：缓存未命中 → 查 MySQL 得到旧值 v1

  2. 线程 B 写：更新 MySQL 到 v2 → 删除缓存

  3. 线程 A 写：把旧值 v1 写入 Redis

    → Redis 永久脏数据，直到缓存过期

解决方案

方案 1：缓存过期时间（兜底方案）

给所有缓存设置合理的过期时间（通常 5~30 分钟）。

  • 优点：最简单，无需修改任何业务代码
  • 缺点：存在不一致窗口期
  • 适用：非核心数据（商品介绍、用户头像等）
方案 2：延迟双删（进阶版）
  • 原理：删除线程 A 可能写入的旧缓存
  • Δt 怎么确定？略大于一次读请求的平均耗时（可通过监控获取）
  • 优化：将第二次删除改为异步执行，不影响接口响应时间
方案 3：分布式锁（缓存重建加锁）
  • 优点：彻底解决并发脏数据
  • 缺点：性能有轻微损失
  • 适用：核心数据（商品价格、库存、用户余额等）

解决删缓存失败问题（最关键）

这是生产环境中最常见的不一致原因，必须有可靠的解决方案。

方案 1：同步重试（简单版）

  • 优点：实现简单，无需额外组件
  • 缺点：同步重试会增加接口响应时间
  • 适用：小型项目，非核心业务

方案 2：MQ 异步重试（企业版）

  • 优点：异步不影响主流程，可靠性高
  • 缺点：引入 MQ 依赖
  • 适用：中大型项目，核心业务

方案 3：监听 MySQL Binlog 异步删除（终极版，大厂标准）

这是目前最可靠、最优雅的解决方案，阿里、京东、美团等大厂普遍采用。

架构：

核心优势：

  1. 完全解耦：业务代码无需关心缓存删除，专注于业务逻辑
  2. 绝对可靠：只要 MySQL 事务提交成功，Binlog 一定会被消费
  3. 统一处理：所有数据库更新操作都会自动触发缓存删除
  4. 高性能：异步处理，不影响主流程性能

实现细节：

  • 只监听UPDATE和DELETE操作
  • 消费失败自动重试，保证至少一次语义
  • 幂等性处理，防止重复删除
  • 批量处理，提高消费性能

强一致性方案（金融、支付、库存等特殊场景）

强一致性会带来巨大的性能损失，只有在绝对必要时才使用。

4.1 TCC 分布式事务

最常用的强一致性方案，性能较好，适合高并发场景。

三个阶段

  1. Try

    ：预留资源

    • 扣减 Redis 库存
    • 冻结 MySQL 库存（而不是直接扣减）

  2. Confirm

    ：确认提交

    • 扣减 MySQL 冻结的库存

  3. Cancel

    ：回滚

    • 恢复 Redis 库存
    • 解冻 MySQL 库存

优点

  • 强一致性
  • 性能较好，无锁
  • 适合高并发场景

缺点

  • 实现复杂，需要处理空回滚、悬挂、幂等性等问题
  • 开发成本高

4.2 读写锁方案

  • 写操作：加排他锁 → 更新 MySQL → 删除缓存 → 释放锁
  • 读操作：加共享锁 → 读 Redis → 未命中读 MySQL → 写回 Redis → 释放锁

优点

  • 强一致性
  • 实现简单

缺点

  • 并发性能极差，所有读写请求串行化
  • 只适合低并发、强一致要求的场景

4.3 双写一致性方案

  • 写操作：同时更新 MySQL 和 Redis
  • 问题：网络延迟导致更新顺序错乱
  • 解决方案：给每个更新加版本号，只更新版本号更高的数据

优点

  • 强一致性

缺点

  • 性能差，两次网络 IO
  • 容易出现顺序问题
  • 不推荐使用

特殊场景的一致性保证

5.1 秒杀场景的一致性（结合 Redis+Lua）

核心原则：Redis 做流量拦截和预扣减，MySQL 做最终数据存储。

完整流程

  1. 秒杀前：将 MySQL 中的商品库存预热到 Redis
  2. 秒杀中：Lua 脚本原子性地检查库存、扣减 Redis 库存、记录用户购买
  3. 秒杀后：发送消息到 MQ，异步创建订单并扣减 MySQL 库存

一致性保证机制

  1. 订单失败回滚：如果 MQ 消费者创建订单失败，将库存回滚到 Redis
  2. 定时库存校准：每 5 分钟对比 Redis 和 MySQL 的库存，修正不一致
  3. 支付最终校验：用户支付完成后，再次校验 MySQL 库存，防止超卖
  4. 活动结束同步：秒杀结束后，将 Redis 剩余库存同步回 MySQL

5.2 大库存扣减的一致性

  • 问题：高并发下，MySQL 行锁冲突严重，性能极差

  • 解决方案

    ：

    1. Redis 预扣减（Lua 脚本原子操作）
    2. 批量扣减 MySQL 库存（每 100 个订单合并一次扣减）
    3. 定时校准，保证最终一致

5.3 缓存击穿 / 穿透 / 雪崩下的一致性

  • 缓存击穿：热点 key 过期 → 分布式锁重建缓存
  • 缓存穿透：不存在的 key → 布隆过滤器过滤
  • 缓存雪崩：大量 key 同时过期 → 过期时间加随机值（±10%）
