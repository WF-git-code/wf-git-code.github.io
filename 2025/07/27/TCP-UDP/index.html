TCP

TCP头部

序列号：在建立连接时由计算机生成的随机数作为其初始值，通过 SYN 包传给接收端主机，每发送一 次数据，就「累加」一次该「数据字节数」的大小。用来解决网络包乱序问题。

**确认应答号：**指下一次「期望」收到的数据的序列号，发送端收到这个确认应答以后可以认为在这个序 号以前的数据都已经被正常接收。用来解决不丢包的问题

控制位：

ACK：该位为 1 时，「确认应答」的字段变为有效，TCP 规定除了最初建立连接时的 包之外该位必须设置为 1 。

RST：该位为 1 时，表示 TCP 连接中出现异常必须强制断开连接。

SYN：该位为 1 时，表示希望建立连接，并在其「序列号」的字段进行序列号初始值的设定。

FIN：该位为 1 时，表示今后不会再有数据发送，希望断开连接

为什么需要TCP？

IP 层是「不可靠」的，它不保证网络包的交付、不保证网络包的按序交付、也不保证网络包中的数据 的完整性

因为 TCP 是一个工作在传输层的可靠数据传输的服务，它能确保接收端接收的网络包是无损坏、无间 隔、非冗余和按序的

什么是TCP

面向连接的，可靠的，流式服务

“TCP 是面向连接的、可靠的字节流协议，只能一对一通信，通过确认应答、超时重传等机制保证数据无丢失、无重复、按顺序到达，不保留应用层消息边界。”

但也不是一定不会出错，极端情况（如网络完全断开超过重传次数）会放弃传输并通知应用层，但它是所有传输层协议中可靠性最高的

TCP连接

就是两台主机在TCP规则下分别维护的一组状态信息

TCP四元组唯一确定一个连接

有一个 IP 的服务器监听了一个端口，它的 TCP 的最大连接数是多少？

最大连接数=客户端IP数*客户端端口数

三次握手

刚开始：客户端和服务器都处于close状态

第一步：客户端初始化序列号client_isn,将此序列号置于头部的序列号字段，同时设置SYN标志位为1，表示SYN报文发送给服务端，发送连接，不包含应用层数据，客户端处于SYN_SET状态

第二步：服务器收到SYN包，服务器也初始化sever_isn,并填入报头的序列号中，应答确认号改为client_isn+1，接着SYN和ACK设置为1，发送，之后客户端处于SYN_RCVD状态

第三步：客户端收到服务端报文后，还要向服务端回应最后一个应答报文，首先该应答报文 TCP 首部 ACK 标志位置为 1 ，其次「确认应答号」字段填入 server_isn + 1 ，最后把报文发送给服务 端，这次报文可以携带客户到服务器的数据，之后客户端处于 ESTABLISHED 状态。

服务器收到客户端的应答报文后，也进入 ESTABLISHED 状态。

✅ 服务器必须等双方都进入 ESTABLISHED 状态后才能发送应用数据

✅ 客户端只要自己进入 ESTABLISHED 状态就可以立即发送数据（此时服务器可能还未收到 ACK，仍处于 SYN_RCVD 状态）

从上面的过程可以发现第三次握手是可以携带数据的，前两次握手是不可以携带数据的，这也是面试常 问的题。

一旦完成三次握手，双方都处于 ESTABLISHED 状态，此时连接就已建立完成，客户端和服务端就可 以相互发送数据了。

标准 TCP 下，客户端在发送完第三次握手的 ACK 后立即进入 ESTABLISHED 并可以发送数据，服务器必须等收到 ACK 进入 ESTABLISHED 后才能发送；启用 TCP Fast Open 时，数据可以在 SYN 报文中提前发送。"

为什么是三次握手？不是两次/四次

TCP连接：用于保证可靠性和流量控制维护的某些状态信息（包括socket,序列号和窗口）

三次握手可以重复历史连接的初始化

旧的SYN报文比新的SYN更早到达了服务端，服务端回给客户端SYN+ACK，此时客户端就可以判断这是一个历史连接

如果是两次握手，服务器在回ACK+SYN后已经认为建立连接完成，很可能直接带数据过去，客户端没有足够时间来判断

三次握手可以同步双方的初始序列号

序列号：

接收方可以去除重复数据

接收方可以根据序列号按序号接收

可以标识发送的数据那些已经被接收了

客户端发送带初始序列号的SYN报文，服务器要回ACK+SYN（服务器），客户端也要回SYN(客户端)+ACK，这样一来一回才能保证双方序列号被可靠的同步

如果四次握手，则把中间一步拆开就好，但是二三可以优化一步，两次握手只保证了一方的序列号可以被对方接收，另一方回复之后什么都没收到不能确认对方的收数据能力

三次握手才可以避免资源浪费

如果客户端的 SYN 阻塞了，重复发送多次SYN ，服务器在收到会给每一个SYN都建立的无效链接，造成不必要的资源浪费。

监听队列

四次挥手

主动关闭的一方才存在TIME_WAIT状态

每个方向都需要一个 FIN 和一个 ACK，因此通常被称为四次挥手。

TIME_WAIT状态

保证连接正确关闭，防止旧链接的数据包

主动关闭的一方

如果客户端第四此挥手的ACK丢了，此时TIME_WAIT状态可以在服务端超时重传机制下重新发ACK，如果没有改状态则TCP连接不能被可靠的关闭。

如果TIME_WAIT时间过短，结束的TCP连接的数据包可能在建立新连接是到达，导致数据错乱

为什么时2MSL?

MSL是报文最大生存时间，报文在网络上存在超过这个时间就会被丢弃，2MSL是客户端发送FIN后收到ACK开始计时的，如果服务器超时重传则TIME_WAIT会重新计时 linux 2MSL==60s

也可以修改TIME_WAIT时间，单要修改内核重新编译

TIME_WAIT过多的危害

内存的占用，端口的资源

为什么四次挥手

关闭连接时，客户端发送FIN只表示客户端没数据发了，但还能收，服务器收到FIN回复ACK，但还可能有数据发，等数据发完才会回FIN，此时客户端在回ACK保证连接被关闭

如果服务器没有数据发则第二次和第三次合并为FIN+ACK发送

Linux看TCP状态

为什么序列号客户端和服务器不同？

如果一个已经失效的连接被重用了，但是该旧连接的历史报文还残留在网络中，如果序列号相同，那么 就无法分辨出该报文是不是历史报文，如果历史报文被新的连接接收了，则会产生数据错乱。

所以，每次建立连接前重新初始化一个序列号主要是为了通信双方能够根据序号将不属于本连接的报文 段丢弃。

另一方面是为了安全性，防止黑客伪造的相同序列号的 TCP 报文被对方接收

起始序列号怎么产生

“TCP 起始序列号 ISN 不是从 0 开始，而是基于 RFC 6528 标准，使用密码学哈希算法结合连接四元组、系统密钥和时间戳生成的伪随机数，核心目的是防止序列号预测攻击和避免旧数据包混淆。”

为什么 ISN 需要单调递增？

为了保证 TCP 的可靠性和有序性。如果 ISN 不单调递增，旧连接的延迟数据包可能会被新连接误认为是有效数据，导致数据错乱。

TCP 序列号为什么是 32 位的？

32 位序列号可以提供足够大的循环空间，在正常网络速度下，循环周期足够长（约 4.5 小时），可以保证旧数据包在循环周期内已经从网络中消失。

什么是序列号回绕问题？怎么解决？

当序列号达到 2^32-1 后，会从 0 重新开始，这就是序列号回绕。解决方法是使用时间戳选项（TCP Timestamps），通过时间戳来区分新旧序列号。

TCP Fast Open 中的 ISN 生成有什么不同？

没有不同。TFO 只是允许在 SYN 包中携带数据，ISN 的生成机制与标准 TCP 完全一致。

SYN攻击导致服务器syn内存耗尽

攻击者短时间伪造不同IP地址的SYN报文，服务端在收到一个报文后进入SYN_RCVD状态，服务端发出的SYN+ACK无法得到回答，则会占满服务端的SYN接收队列（未连接队列）

解决

①修改linux内核参数，控制队列大小和队列满时做什么处理，改SYN_RCVD最大数和满了对新的SYN直接回RST丢弃连接

②采用tcp_syncookies方式，

服务器不分配任何内存来保存这个半连接的状态，而是计算一个SYN Cookie 值，并把这个 Cookie 值作为SYN+ACK 包的起始序列号 ISN返回给客户端。

SYN Cookie 的本质：就是一个特殊的 ISN，它把所有需要保存的连接状态信息编码到了 32 位的序列号里，这样服务器就不需要在本地保存任何状态了。

服务器收到这个 ACK 包后，不会去 SYN 队列里找对应的条目（因为根本就没存），而是直接从 ACK 号里提取出 Cookie 值，然后重新计算一遍 Cookie：合法则放入连接队列不合法丢弃改ACK，最后通过accept()从accept队列取出开始正常数据传输

缺点：

丢失 TCP 选项：因为 32 位的序列号空间有限，无法编码 TCP 选项（如窗口缩放、SACK、时间戳等），所以启用 SYN Cookie 建立的连接会丢失这些高级特性，性能会略有下降。

计算开销：每次都要计算哈希值，会增加服务器的 CPU 开销。

无法防御带宽耗尽攻击：它只能防御 SYN 洪水导致的内存耗尽，如果攻击流量足够大，还是会占满服务器的带宽。

时间限制：Cookie 有有效期，客户端必须在有效期内回复 ACK，否则连接会失败

服务器应用程序太慢会导致Accept（连接队列）饱满

受到SYN攻击，会导致SYN队列占满

超时重传

TCP实现可靠传输的方式之一是通过序列号确认应答

网络中可能发出去的包因为某些问题导致丢失，则诞生了超时重传

超时重传

TCP会在数据包丢失和确认应答丢失发生超时重传

RTT 数据从网络一端发到另一端所需要的时间

RTO 超时重传时间

RTO太大 当超时时间 RTO 较大时，重发就慢，丢了老半天才重发，没有效率，性能差

RTO太小 ，会导致可能并没有丢就重发，于是重发的就快，会增加网络拥塞，导致 更多的超时，更多的超时导致更多的重发

RTO的计算

超时重发的数据再次超时则TCP的策略是RTO*2,出现2次以上的超时说明网络环境差不能频繁发送

快速重传

超时触发重传存在的问题是，超时周期可能相对较长。

快速重传以数据为驱动

同时发送四个数据包，第一个没收到但收到了后三个的ACK，则发送端就会在定时器过期之前重传丢失的1，送到就会收到接收方的4，代表4之前的全部收到了

SACK

这种方式需要在 TCP 头部「选项」字段里加一个 SACK 的东西，它可以将缓存的地图发送给发送 方，这样发送方就可以知道哪些数据收到了，哪些数据没收到，知道了这些信息，就可以只重传丢失的 数据

D-SACK

由于网络原因收到数据但没有收到确认报文，则会启动超时重传，接收方发现数据重复则会发D-SACK说明自己数据收到了

使用了 SACK 来告诉「发送方」有哪些数据被重复接收了。

滑动窗口

数据包的往返时间越长，通信效率就越低

TCP引入窗口，即使在往返时间较长的情况下依然不会影响网络通信，窗口大小就是无需确认应答而可以继续发送数据的最大值

窗口的实际实现是操作系统开辟的缓存空间，发送方在等到确认应答之前必须在缓冲区保留已经发送的窗口，收到确认应答则清除

累计确认

在有三个TCP段的窗口中，发送三个TCP数据段，如果中间一个ACK没收到但后续的ACK收到了，表明这段数据已经全部被接收不会触发超时重传

窗口大小由谁决定？

TCP头里有个window，就是窗口大小，该字段用于告诉发送端自己还有多少缓冲区接收数据，发送端根据这个接收端window大小来发送数据

接收窗口和发送窗口不一定相等

滑动窗口是实时改变的，接收窗口大小要通过window字段告知发送方，存在延迟

流量控制决定接收方 接收窗口

拥塞控制决定发送方 发送窗口

发送窗口大小 swnd = min(接收窗口 rwnd, 拥塞窗口 cwnd)

“发送方发太快，接收方来不及接收导致数据丢失” 的问题。它的核心实现是滑动窗口协议，通过接收方动态通告自己的接收能力，让发送方调整发送速度，保证数据可靠有序传输。

流量控制：解决端到端的速度不匹配问题（接收方处理不过来）

流程

连接建立阶段：双方在 SYN 报文中通告自己的初始接收窗口大小

数据传输阶段：

  • 接收方在每个 ACK 报文中都会携带自己当前的接收窗口大小 rwnd
  • 发送方根据 rwnd 调整自己的发送窗口大小 swnd
  • 发送方只能发送落在 swnd 范围内的数据

接收缓冲区满：接收方通告 rwnd=0，发送方停止发送数据

接收缓冲区有空间：接收方通告新的 rwnd>0，发送方恢复发送

1.零窗口问题与窗口探测

问题：当接收方的接收缓冲区满了，会通告 rwnd=0，发送方就会停止发送数据。如果之后接收方的应用程序读取了一些数据，有了空间，但这个通告窗口的 ACK 报文丢失了，就会导致：

  • 发送方一直等待 rwnd>0 的通知
  • 接收方以为已经通知了发送方，也在等待数据
  • 双方陷入死锁

解决方案：窗口探测（Zero Window Probe）

  • 发送方会定期发送一个窗口探测报文（只携带 1 字节的数据）
  • 接收方收到探测报文后，会回复自己当前的接收窗口大小
  • 如果 rwnd 仍然是 0，发送方会继续等待，并重发探测报文
  • 探测报文的超时时间会指数退避，从 30 秒到 1 分钟不等
  1. 糊涂窗口综合征（Silly Window Syndrome）

问题：当接收方每次只能接收少量数据（比如几个字节），就会通告一个小窗口；发送方收到后，就会发送一个小报文。这样一来，网络中就会充满大量只有几个字节的小报文，传输效率极低。

举个例子：

  • 接收缓冲区大小 4096 字节
  • 应用程序每次只读取 1 字节
  • 接收方就会通告 rwnd=1
  • 发送方就会发送 1 字节的数据
  • 这样传输 4096 字节需要 4096 个报文，每个报文 40 字节头部，总开销 160KB，效率只有 2.5%

解决方案：

  • 接收方

    ：延迟通告窗口，直到满足以下两个条件之一：

    1. 有足够的空间接收一个最大段大小（MSS）的数据
    2. 接收缓冲区有一半以上的空间是空的

  • 发送方：使用 Nagle 算法，积累足够多的数据再发送

  1. Nagle 算法

Nagle 算法是发送方的优化机制，用于减少网络中的小报文数量：

  • 当发送方有数据要发送时，如果当前没有已发送未确认的数据，就立即发送
  • 如果有已发送未确认的数据，就把数据缓存起来，直到收到 ACK 或者积累了一个 MSS 的数据再发送

注意：Nagle 算法和延迟 ACK 一起使用时，可能会导致额外的延迟。在对延迟敏感的场景（如游戏、实时通信），可以通过 TCP_NODELAY 选项关闭 Nagle 算法。

拥塞控制

拥塞控制：解决全局网络的拥塞问题（路由器转发不过来）

慢启动
  • 逻辑：连接建立初期，发送方不知道网络承载能力，所以小窗口起步，指数级增长（避免一下把网络冲垮）。
  • 流程
    • 初始 cwnd=1~2 个 MSS（最大报文段）；
    • 每收到一个 ACK，cwnd 翻倍（比如 cwnd=1→2→4→8…）；
    • 直到 cwnd 达到 “慢启动阈值（ssthresh，初始通常是 64KB 左右）”，进入拥塞避免阶段。
拥塞避免

  • 逻辑：慢启动到阈值后，改为线性增长（避免指数增长触发拥塞）。

  • 流程

    • 每经过一个 “RTT（往返时延）”，cwnd+1（比如 cwnd=8→9→10…）；

    • 若出现

      超时丢包

      （认为网络拥塞）：

      • 更新 ssthresh = 当前 cwnd/2；
      • 重置 cwnd=1，重新进入慢启动。
拥塞发生

超时重传 快重传

快恢复
  • 逻辑：配合快重传使用，丢包后不 “重置 cwnd 到 1”，而是快速恢复窗口（避免吞吐量骤降）。
  • 流程
    • 收到 3 个重复 ACK 后，更新 ssthresh = 当前 cwnd/2；
    • 重置 cwnd = ssthresh；
    • 直接进入拥塞避免阶段（不再走慢启动）。

TCP粘包

解决 TCP 粘包有 3 类核心方案，工业界主流首选长度前缀法：

  1. 固定长度法：约定固定消息长度，接收方按长读取，实现简单但灵活性差，仅适用于定长报文场景；

  2. 特殊分隔符法：用唯一分隔符标记消息结尾，适合纯文本协议，二进制场景易出现边界歧义；

  3. 长度前缀法（标准方案）：消息前加固定头部标注消息体长度，接收方按长度精准读取，完美解决粘包 / 拆包，是 RPC、中间件、网络库的通用方案。

    辅助可关闭 Nagle 算法减少攒包，但无法根治粘包。

    粘包核心诱因（面试必提，体现底层理解）

    • 发送方：TCP 的 Nagle 算法会攒合小数据包批量发送，导致多个逻辑消息合并为一个 TCP 段；
    • 接收方：内核接收缓冲区会堆积多个未读取的消息，应用层一次读取会拿到多个逻辑包；
    • 固有特性：超过 MSS 的大消息会被 TCP 强制拆分为多个段传输（拆包），和粘包伴生出现。

UDP

无连接，不可靠，数据报服务

校验和：校验和是为了提供可靠的 UDP 首部和数据而设计

TCP/UDP区别

区别

1.连接：TCP 是面向连接的传输层协议，传输数据前先要建立连接。 UDP 是不需要连接，即刻传输数据

2.服务对象

TCP 是一对一的两点服务，即一条连接只有两个端点。 UDP 支持一对一、一对多、多对多的交互通信

3.TCP 是可靠交付数据的，数据可以无差错、不丢失、不重复、按需到达。 UDP 是尽最大努力交付，不保证可靠交付数据。

4.拥塞控制、流量控制

TCP 有拥塞控制和流量控制机制，保证数据传输的安全性。 UDP 则没有，即使网络非常拥堵了，也不会影响 UDP 的发送速率。

5.首部开销

TCP 首部长度较长，会有一定的开销，首部在没有使用「选项」字段时是 用了「选项」字段则会变长的。 UDP 首部只有 8 个字节，并且是固定不变的，开销较小。

6.传输方式

TCP 是流式传输，没有边界，但保证顺序和可靠。 UDP 是一个包一个包的发送，是有边界的，但可能会丢包和乱序。

7.分片不同

TCP 的数据大小如果大于 MSS 大小，则会在传输层进行分片，目标主机收到后，也同样在传输 层组装 TCP 数据包，如果中途丢失了一个分片，只需要传输丢失的这个分片。

UDP 的数据大小如果大于 MTU 大小，则会在 IP 层进行分片，目标主机收到后，在 IP 层组装完 数据，接着再传给传输层，但是如果中途丢了一个分片，则就需要重传所有的数据包，这样传输 效率非常差，所以通常 UDP 的报文应该小于 MTU

为什么 UDP 头部没有「首部长度」字段，而 TCP 头部有「首部长度」字段呢？

原因是 TCP 有可变长的「选项」字段，而 UDP 头部长度则是不会变化的，无需多一个字段去记录 UDP 的首部长度。

为什么 UDP 头部有「包长度」字段，而 TCP 头部则没有「包长度」字段呢？

TCP数据长度=IP总长度-IP首部长度-TCP首部长度

因为为了网络设备硬件设计和处理方便，首部长度需要是 4字节的整数倍。 如果去掉 UDP 「包长度」字段，那 UDP 首部长度就不是 4 字节的整数倍了，所以小林觉得这可能 是为了补全 UDP 首部长度是 4 字节的整数倍，才补充了「包长度」字段。

TCP/UDP应用场景

TCP

由于 TCP 是面向连接，能保证数据的可靠性交付，因此经常用于： FTP 文件传输 HTTP / HTTPS

UDP

由于 UDP 面向无连接，它可以随时发送数据，再加上UDP本身的处理既简单又高效，因此经常用于： 包总量较少的通信，如 DNS 、 SNMP 等 视频、音频等多媒体通信 广播通信
