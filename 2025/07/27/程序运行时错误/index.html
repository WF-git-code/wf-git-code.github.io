Core Dump

Core 文件 = 程序崩溃瞬间的完整内存镜像 + 所有 CPU 寄存器状态

当程序因为严重错误（如段错误、总线错误）崩溃时，操作系统会将程序崩溃瞬间的完整内存状态保存到一个文件中，这个文件就叫 core 文件

gdb 只是调试工具，而 core 文件是程序崩溃瞬间的完整 “尸体”**。没有 core 文件，你只能调试 “活的” 正在运行的程序；有了 core 文件，你可以调试 “死的” 已经崩溃的程序，完美还原它死亡前的最后一刻。

它记录了：

  • 程序崩溃时的调用栈（哪行代码出的错）
  • 所有寄存器的值
  • 所有变量的当前值
  • 内存的完整内容
  • 线程状态

总线错误

SIGBUS (7) 总线错误

本质：CPU 访问内存时违反了硬件级别的内存访问规则，比段错误更底层。

典型现象：程序直接崩溃，输出 “总线错误 (核心已转储)”

1. 未对齐的内存访问（最常见）

说明：大多数 CPU 要求基本数据类型的地址必须是其大小的倍数（如 int 需要 4 字节对齐，long long 需要 8 字节对齐）。

注意：x86_64 架构对未对齐访问有硬件支持（但性能差），ARM 架构会直接触发总线错误。

2. mmap 映射文件被截断

说明：使用 mmap 映射一个文件后，文件被其他进程截断，访问超出文件大小的部分会触发总线错误。

3. 访问不存在的物理地址

说明：访问的虚拟地址没有映射到任何物理内存页。

4. 访问已被 munmap 的内存区域

快速排查要点：

  • 检查所有强制类型转换，特别是 char * 转其他类型指针
  • 检查 mmap 映射文件的大小是否在访问过程中发生变化
  • 使用 gdb 查看崩溃时的指针地址，判断是否对齐

浮点异常

SIGFPE (8) 浮点异常

本质：CPU 执行算术运算时发生了非法的数学操作。

典型现象：程序直接崩溃，输出 “浮点异常 (核心已转储)”

1. 整数除以零（最常见）

2. 整数对零取模

3. 浮点除以零

4. 浮点溢出

5. 无效浮点运算

快速排查要点：

  • 检查所有除法和取模运算，确保除数不为零
  • 浮点运算默认不会触发 SIGFPE，需要显式启用异常捕获
  • gdb 中查看崩溃时的运算数，确认是否有零或非法值

非法指令

SIGILL (4) 非法指令

本质：CPU 尝试执行不存在或无效的机器指令。

典型现象：程序直接崩溃，输出 “非法指令 (核心已转储)”

1. 执行了无效的机器指令

2. 栈破坏导致跳转到错误地址

说明：缓冲区溢出覆盖了栈上的返回地址，导致函数返回时跳转到错误的地址执行。

3. 编译器生成了不支持的指令集

说明：在旧 CPU 上运行使用新指令集（如 AVX-512）编译的程序。

4. 动态链接库版本不兼容

说明：程序链接的动态库版本与运行时的版本不一致，导致跳转到错误的函数地址。

5. 代码段被破坏

说明：程序错误地修改了代码段的内容（代码段默认是只读的，通常会先触发 SIGSEGV）。

快速排查要点：

  • 检查是否有缓冲区溢出或栈破坏
  • 确认程序编译时使用的指令集是否与运行 CPU 兼容
  • 检查动态链接库版本是否正确
  • gdb 中查看崩溃时的指令指针 rip 的值，判断是否指向有效代码

异常中止

SIGABRT (6)异常中止

本质：程序主动请求终止，通常是因为检测到了不可恢复的错误。

典型现象：程序崩溃前会输出错误信息，如 “Assertion failed”、“stack smashing detected”

1. assert 断言失败（最常见）

2. 显式调用 abort () 函数

3. C++ 未捕获的异常

4. glibc 检测到堆损坏

说明：glibc 的内存分配器检测到堆内存结构被破坏，如 double-free、堆溢出。

5. 栈溢出检测（stack smashing detected）

说明：gcc 默认启用了栈保护机制，检测到栈溢出时会终止程序。

快速排查要点：

  • 查看程序的标准错误输出，通常会有明确的错误信息
  • gdb 中查看调用栈，会显示在 abort () 或__assert_fail () 处崩溃
  • 对于堆损坏，使用 valgrind --tool=memcheck 检测

被强制杀死

SIGKILL (9) 被强制杀死

本质：内核强制终止进程，进程无法捕获、忽略或阻塞这个信号。

典型现象：程序直接退出，输出 “已杀死”，不会生成 core 文件

1. OOM Killer 杀死进程（最常见）

说明：系统物理内存和 Swap 都耗尽时，内核的 OOM Killer 机制会自动杀死内存占用最高的进程。

2. 管理员手动执行 kill -9

3. cgroups 资源限制触发

说明：进程超过了 cgroups 设置的内存或 CPU 使用限制，被 cgroups 杀死。

4. 系统 panic 前清理进程

说明：系统发生严重错误即将 panic 时，会杀死所有用户进程。

5. 父进程退出时杀死子进程

说明：父进程退出时，会向所有子进程发送 SIGHUP 信号，如果子进程没有处理 SIGHUP，会被杀死。

快速排查要点：

  • 首先查看内核日志：dmesg | grep -i "killed process"
  • 检查是否有 OOM 日志，这是最常见的原因
  • 检查 cgroups 资源限制
  • 检查系统日志中是否有管理员操作记录

死锁进程的状态是S（睡眠），不是 R（运行）或 D（不可中断睡眠）

阻塞 IO：CPU 使用率接近 0%，但通常会在 IO 完成后自动恢复

权限错误

检查流程

确认你是谁：whoami 查看当前用户，id 查看用户 ID 和所属组

确认资源归谁：ls -l 目标文件/目录 查看所有者和权限位

确认路径权限：namei -l 完整路径 检查所有父目录的执行权限（最容易被忽略）

确认是否被安全模块拦截：检查 SELinux/AppArmor 日志

常见错误类型

排查解决

查看文件权限：

权限位解读：- rw- r-- r--

  • 第一位：文件类型（- 普通文件，d 目录，l 符号链接）
  • 第 2-4 位：所有者权限（rw-：可读可写不可执行）
  • 第 5-7 位：组权限（r–：只读）
  • 第 8-10 位：其他用户权限（r–：只读）

检查父目录权限：

注意：要访问一个文件，必须拥有该文件所有父目录的执行权限（x）。

解决

方法 1：修改文件权限

方法 2：修改文件所有者

常见坑

  • 不要随便用 chmod 777：这会给所有用户读写执行权限，存在严重安全隐患
  • 目录必须有 x 权限才能进入：即使有 r 权限，没有 x 权限也无法访问目录内容
  • 符号链接的权限永远是 777：实际权限由指向的文件决定

先看 CPU：top 查看id和wa

  • 如果wa高 → 是磁盘 IO 问题
  • 如果us或sy高 → 是 CPU 问题

再看内存：free -h 查看available

  • 如果available低 → 是内存问题

再看磁盘：df -h 和 iostat -x 1

  • 如果%util高 → 是磁盘 IO 问题

再看网络：iftop 和 ss

  • 如果流量异常 → 是网络问题

最后看文件句柄：lsof 和 ulimit

  • 如果文件句柄数接近限制 → 是文件句柄问题

资源问题

遇到资源问题时，按照这个顺序排查和解决：

  1. 先看系统全局限制：用sysctl查看系统级的总闸门是否足够
  2. 再看进程级限制：用prlimit --pid <pid>查看进程的实际限制
  3. 临时解决：用prlimit动态修改运行中进程的限制
  4. 永久解决：
    • 系统级：修改/etc/sysctl.d/下的配置文件
    • 服务级：在 systemd 的 service 文件中设置Limit*参数

ulimit

ulimit是 Linux 内核提供的进程级资源限制机制，用于防止单个进程耗尽系统所有资源导致整体崩溃

软限制与硬限制

类型 含义 修改权限 特性
软限制 (soft) 当前生效的警告阈值 任何用户都可以修改，但不能超过硬限制 超过时系统会发出警告，但通常不会立即终止进程
硬限制 (hard) 资源使用的绝对上限 只有 root 用户可以提升 超过时系统会强制阻止并可能终止进程
参数 含义 默认值 常见问题
-c core 文件最大大小 (KB) 0（不生成 core 文件） 程序崩溃只输出 “核心已转储” 但没有 core 文件
-n 单个进程可打开的最大文件描述符数 1024 “Too many open files” 错误
-u 单个用户可创建的最大进程数 4096 “fork: Resource temporarily unavailable” 错误
-s 栈大小 (KB) 8192 递归过深导致栈溢出
-m 最大常驻内存大小 (KB) unlimited
-v 最大虚拟内存大小 (KB) unlimited

systemd+cgroups 进程组级 一组进程 CPU、内存、IO、网络等 重启服务生效 服务级资源隔离、容器
quota 用户级 单个用户 / 组 磁盘空间 立即生效 多用户系统磁盘管理
tc 网络级 端口 / IP / 进程 网络带宽 立即生效 网络流量控制
ionice 进程级 单个进程 磁盘 IO 优先级 立即生效 防止 IO 密集型进程拖慢系统
nice 进程级 单个进程 CPU 优先级 立即生效 后台任务优先级调整

prlimit

prlimit是 Linux 内核 2.6.36 + 引入的标准工具，100% 兼容并替代 ulimit，专门解决 ulimit 的三大致命缺陷：

  • 是独立二进制文件，支持sudo，不会出现sudo ulimit -n无效的问题
  • 可以动态修改任意正在运行的进程的限制，不需要重启服务
  • 可以在启动程序时一次性设置所有限制，不需要修改配置文件

1. 核心语法

prlimit 参数 ulimit 参数 含义 单位
--core -c core 文件最大大小 bytes
--nofile -n 单个进程可打开的最大文件描述符数 个数
--nproc -u 单个用户可创建的最大进程 / 线程数 个数
--stack -s 进程栈大小 bytes
--rss -m 最大常驻内存大小 bytes
--as -v 最大虚拟内存大小 bytes
--cpu -t 最大 CPU 时间 seconds
--fsize -f 单个进程可创建的最大文件大小 bytes

sysctl

sysctl是 Linux 内核提供的系统级参数配置工具，用于查看和修改内核运行时参数。它控制着整个系统的所有核心行为，从内存管理、网络协议到进程调度。

和 prlimit 的本质区别：

  • prlimit：给单个进程设置 “天花板”
  • sysctl：给整个系统设置 “总闸门”
