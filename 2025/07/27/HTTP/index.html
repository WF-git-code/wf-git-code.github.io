常见字段

host

客户端发送请求时，用来指定服务器域名

Content-Length

服务器返回数据时，该字段表明本次回应数据的长度，超出该长度就属于下一个回应

Connection

常用于客户端要求服务器使用TCP持久连接，以便其他请求复用

HTTP/1.1 版本的默认连接都是持久连接，但为了兼容老版本的 HTTP，需要指定 段的值为 Keep-Alive 。

一个可以复用的 TCP 连接就建立了，直到客户端或服务器主动关闭连接。但是，这不是标准字段。

Content_Type

GET和POST方法的区别？

Get 方法的含义是请求从服务器获取资源，这个资源可以是静态的文本、页面、图片视频等。 比如，你打开我的文章，浏览器就会发送 GET 请求给服务器，服务器就会返回文章的所有文字及资 源

而POST 方法则是相反操作，它向 URI 指定的资源提交数据，数据就放在报文的 body 里。 比如，你在我文章底部，敲入了留言后点击「提交」（暗示你们留言），浏览器就会执行一次 POST 请 求，把你的留言文字放进了报文 body 里，然后拼接好 POST 请求头，通过 TCP 协议发送给服务器

GET 和 POST 方法都是安全和幂等的吗？

**安全：**在 HTTP 协议里，所谓的「安全」是指请求方法不会「破坏」服务器上的资源。

幂等： 所谓的「幂等」，意思是多次执行相同的操作，结果都是「相同」的。

那么很明显 GET 方法就是安全且幂等的，因为它是「只读」操作，无论操作多少次，服务器上的数据 都是安全的，且每次的结果都是相同的。

POST 因为是「新增或提交数据」的操作，会修改服务器上的资源，所以是不安全的，且多次提交数据 就会创建多个资源，所以不是幂等的。

HTTP（1.1）优缺点

优点

简单易懂：报文采用 header + body 结构，头部为 key-value 文本形式，学习门槛低。

灵活可扩展：协议的请求方法、状态码等支持自定义；作为应用层协议，下层可替换（如 HTTPS 加 TLS、HTTP/3 换 QUIC）。

应用广泛跨平台：覆盖从浏览器到各类 APP 的绝大多数互联网场景，天然适配跨平台需求。

缺点

无状态双刃剑：优点是减轻服务器存储负担，缺点是无法识别用户会话，导致关联操作（如购物流程）需反复验证身份，需通过 Cookie 技术弥补。

明文传输双刃剑：优点是调试方便，通过浏览器的 F12 控制台或 Wireshark 抓包都可 以直接肉眼查看，缺点是信息无加密易被窃取，存在隐私泄露风险。

安全缺陷明显：存在三大风险 —— 明文传输易被窃听、不验证通信方身份易遭伪装、无法保证报文完整性易被篡改，这些问题需通过 HTTPS（引入 SSL/TLS 层）解决。

HTTP（1.1）性能如何

HTTP 协议是基于 TCP/IP，并且使用了「请求 - 应答」的通信模式，所以性能的关键就在这两点里。

1**.持久连接：**避免每一次请求都要经历三次握手,减少资源消耗

2**.管道网络传输：**同一个 TCP 连接里面，客户端可以发起多个请求，只要第一个请求发出去了，不必等其回来， 就可以发第二个请求出去，可以减少整体的响应时间。但是服务器还是按照顺序，先回应 A 请求，完成后再回应 B 请求。要是前面的回应特别慢，后面就会 有许多请求排队等着。这称为「队头堵塞」

**3.队头堵塞：**因为当顺序发送的请求序列中的一个请求因为某种原因被阻塞时，在后面排队的所有请求也一同被阻塞 了，会招致客户端一直请求不到数据

HTTPS

  1. HTTP 是超文本传输协议，信息是明文传输，存在安全风险的问题。HTTPS 则解决 HTTP 不安全 的缺陷，在 TCP 和 HTTP 网络层之间加入了 SSL/TLS 安全协议，使得报文能够加密传输

  2. HTTP 连接建立相对简单， TCP 三次握手之后便可进行 HTTP 的报文传输。而 HTTPS 在 TCP 三次握手之后，还需进行 SSL/TLS 的握手过程，才可进入加密报文传输。

  3. HTTP 的端口号是 80，HTTPS 的端口号是 443。

  4. HTTPS 协议需要向 CA（证书权威机构）申请数字证书，来保证服务器的身份是可信的。

HTTPS解决HTTP问题？

对 “无状态” 问题的间接优化

HTTP 本身的 “无状态” 特性，需要 Cookie/Session 来解决会话关联问题。

  • 但在 HTTP 中，Cookie 是明文传输的，很容易被窃取、伪造，导致用户会话被劫持。
  • HTTPS 加密了 Cookie 的传输过程，让 Cookie 不再容易被窃听或篡改，让 Cookie/Session 这种 “无状态的补丁” 变得安全可用。

HTTPS 在 HTTP 与 TCP 层之间加入了 SSL/TLS 协议，可以很好的解决了上述的风险：

防窃听：数据全程加密，第三方抓包只能看到乱码，无法获取明文信息。

防伪装：通过数字证书验证服务器身份，杜绝中间人伪造网站。

防篡改：通过摘要校验，确保数据在传输中未被修改。

HTTP 的问题 HTTPS 的解决方式
明文传输，信息裸奔易被窃听 传输前用对称密钥加密数据，密钥本身用非对称加密安全交换，第三方无法直接读取内容。
无法验证通信方身份，易被钓鱼 / 伪装 通过 CA 颁发的数字证书验证服务器身份，确保你访问的是真正的目标网站，而非伪造的钓鱼站点。
无法验证报文完整性，内容易被篡改 通过消息摘要（哈希 + 数字签名）校验数据，一旦报文被中途篡改，校验会直接失败，防止网页被植入广告或恶意代码。****

HTTP/1.1 /2/3的演变

HTTP/1.1 相比 HTTP/1.0 的性能提升

  1. TCP 长连接：解决了 HTTP/1.0 短连接反复建立 / 断开的开销。
  2. 管道（pipeline）传输：可在未收到响应时就发送下一个请求，减少整体响应时间。

但 HTTP/1.1 仍有明显瓶颈：

  • 头部未压缩，冗余信息多、延迟大
  • 队头阻塞（请求串行处理，前一个请求慢会阻塞后续请求）
  • 无请求优先级控制
  • 只能客户端主动发起请求

HTTP/2 针对 HTTP/1.1 瓶颈的优化

  1. 头部压缩（HPACK）：通过维护头部信息表，消除重复字段，减少传输体积。
  2. 二进制帧格式：报文拆分为头部帧 / 数据帧，计算机解析效率更高。
  3. 数据流与优先级控制：每个请求 / 响应对应独立数据流，支持设置优先级，优化关键资源加载顺序。
  4. 多路复用：单 TCP 连接上并发处理多个请求 / 响应，彻底解决队头阻塞问题，提升连接利用率。
  5. 服务器推送（Server Push）：服务器可主动推送资源，减少客户端等待时间。

HTTP/2 仍有局限：

  • 底层依赖 TCP，一旦发生丢包，TCP 的重传机制会阻塞同连接内所有请求（TCP 层面的队头阻塞）。

HTTP/3 针对 HTTP/2 瓶颈的优化

  1. 底层协议改为 UDP + QUIC：
    • QUIC 自带丢包重传机制，单个数据流丢包仅阻塞自身，不影响其他流，彻底解决 TCP 层面的队头阻塞。
    • 合并 TCP 握手与 TLS 握手，将传统 6 次交互压缩为 3 次，大幅降低连接建立延迟。
  2. 升级头部压缩为 QPACK：兼容 HTTP/2 的 HPACK 优势，适配 UDP 传输场景。
  3. 基于 TLS 1.3：进一步提升加密性能与安全性。
