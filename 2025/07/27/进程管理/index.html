进程与线程

windiows

Windows 采用了严格的双轨制设计，进程和线程是两个完全不同的内核对象，有各自独立的内核数据结构：

Windows 进程（EPROCESS 结构）

  • 唯一身份：进程是资源分配的基本单位，拥有独立的虚拟地址空间
  • 核心资源：
    • 独立的页表和虚拟地址空间
    • 独立的句柄表（管理文件、套接字、互斥量等内核对象）
    • 独立的访问令牌（安全上下文）
    • 独立的 PEB（进程环境块，用户态数据结构）
    • 一个或多个线程
  • 调度属性：进程本身不能被调度，它只是线程的容器

Windows 线程（ETHREAD 结构）

  • 唯一身份：线程是CPU 调度的基本单位，必须依附于某个进程存在
  • 私有资源：
    • 独立的内核栈和用户栈
    • 独立的寄存器上下文（程序计数器、栈指针等）
    • 独立的 TLS（线程本地存储）
    • 独立的 TEB（线程环境块，用户态数据结构）
    • 线程优先级、调度状态等调度属性
  • 共享资源：同一进程内的所有线程共享进程的所有资源

Windows 模型的特点

  • 概念清晰：进程和线程的职责划分明确，符合大多数人的直觉
  • 实现复杂：内核需要维护两套完全不同的对象体系和管理机制
  • 调度开销大：线程切换需要同时操作 EPROCESS 和 ETHREAD 两个对象

Linux

Linux 采用了极简的单轨制设计，内核中没有专门的线程概念，所有的执行实体都称为 “进程”，都由同一个task_struct结构（也就是 PCB，进程控制块）描述

Linux 内核的调度器只认task_struct，不管它是 “进程” 还是 “线程”。所谓的 “线程”，本质上是与其他进程共享部分资源的特殊进程，被称为轻量级进程 (LWP, Lightweight Process)。

PCB

PCB 的具体实现是task_struct结构体，包含了进程运行所需的所有信息。

核心作用

进程标识：唯一标识系统中的每个进程

进程状态管理：记录进程的运行状态（运行、睡眠、停止等）

CPU 上下文保存：保存进程被切换时的 CPU 寄存器状态

资源管理：记录进程拥有的所有资源（内存、文件、信号等）

调度信息：记录进程的优先级、时间片、调度策略等

进程关系：记录进程的父进程、子进程、兄弟进程等关系

生命周期

PCB 的创建

当调用fork()、vfork()或clone()系统调用创建新进程时，内核会：

  1. 调用alloc_task_struct()函数分配一个task_struct结构体
  2. 初始化task_struct的各个字段
  3. 为新进程分配 PID
  4. 将新进程加入到进程链表和运行队列中
  5. 返回新进程的 PID
PCB 的销毁

当进程调用exit()系统调用终止时，内核会：

  1. 将进程的状态设置为EXIT_ZOMBIE
  2. 释放进程拥有的大部分资源（内存、文件、信号等）
  3. 保留 PCB，等待父进程调用wait()回收
  4. 当父进程调用wait()回收子进程时，内核会将进程的状态设置为EXIT_DEAD，并释放 PCB
线程私有资源

每个线程独立的task_struct字段

  • 栈指针（内核栈和用户栈）
  • 程序计数器 (PC) 和其他寄存器
  • 线程 ID (tid)
  • 调度状态、优先级、时间片等调度属性
  • 线程本地存储 (TLS)
  • 信号掩码（每个线程可以独立屏蔽某些信号）
线程共享资源

多个task_struct指向同一对象

虚拟地址空间（mm_struct）

文件描述符表（files_struct）

文件系统信息（fs_struct）

信号处理函数表（sighand_struct）

进程 ID (pid) 和进程组 ID

访问令牌（安全上下文）

clone

Linux 中所有的执行实体（普通进程、线程、内核线程）都是通过clone()系统调用创建的。fork()和vfork()本质上都是clone()的封装

关键参数flags：决定了子进程与父进程共享哪些资源，这是区分普通进程和线程的核心

普通进程：clone()时不设置任何共享标志，子进程拥有完全独立的资源

线程：clone()时设置上述所有共享标志，子进程与父进程共享几乎所有资源，仅拥有独立的栈和寄存器

线程组

为了兼容POSIX标准（同一个进程内所有线程共用一个PID），linux引入线程组概念

  • 每个线程组有一个线程组组长，它的 PID 就是整个线程组的 PID
  • 线程组内的所有线程拥有相同的tgid（Thread Group ID），但拥有不同的tid（Thread ID）
  • 用户态看到的 PID 实际上是tgid，而内核调度器使用的是tid

这就是为什么你在 Linux 中用ps命令只能看到进程，而用ps -L命令才能看到线程的原因。

协程

协程的本质是在用户态实现的上下文切换机制。当一个协程主动让出 CPU 时，它会保存自己的 CPU 寄存器状态和栈信息，然后切换到另一个协程执行。这个过程完全不需要内核参与，因此开销极小。

协程是一种用户态的轻量级线程，由程序员手动调度，而不是由内核调度

协程切换完全在用户态进行，开销极小，比线程切换快几个数量级

同一线程内的多个协程是串行执行的，一个协程阻塞会导致整个线程阻塞

协程适用于 IO 密集型任务，可以大幅提高系统的并发能力

实现

1. 有栈协程 vs 无栈协程（最核心的区别）

这是协程最本质的区别，也是面试中必问的问题。

特性 有栈协程 无栈协程
内存模型 每个协程有独立的栈空间 所有协程共享同一个栈
上下文切换 保存和恢复整个栈和寄存器 只保存状态机变量
切换开销 中等（几百个时钟周期） 极小（几十个时钟周期）
栈大小 固定（通常几 KB 到几 MB） 无限制（共享主栈）
递归调用 支持 不支持（会导致栈溢出）
代码侵入性 低（可以调用任意函数） 高（只能调用协程友好的函数）
调试难度 低（栈回溯清晰） 高（状态机代码难以调试）
代表实现 libco、ucontext_t C++20 协程、Rust async

适用场景：

  • 有栈协程：适合 IO 密集型任务，需要调用大量现有同步代码的场景
  • 无栈协程：适合计算密集型任务，需要极高并发的场景

2. 对称协程 vs 非对称协程

  • 非对称协程（Asymmetric Coroutines）：协程之间有明确的调用关系，一个协程只能让出 CPU 给它的调用者。这是最常见的协程类型，如 libco、C++20 协程。
  • 对称协程（Symmetric Coroutines）：协程之间是平等的，一个协程可以让出 CPU 给任意其他协程。如 ucontext_t 实现的协程。

boost.coroutine

c++协程库，支持有栈协程和无栈协程的实现

进程的特征

运行态：进程正在处理机上运行

就绪态：进程获得了，运行的所有条件，等待获得处理机

阻塞态：进程正在等待某一时间而暂停运行。如I/O阻塞

创建态：进程正在被创建，尚未转到就绪态

结束态：进程从系统中消失，可能是进程正常结束或者中断

进程的控制

就是对系统中所有进程进行有效的管理

每个物理内存页都会维护一个引用计数，当引用计数为0才会被释放

进程的切换

进程切换是指处理机从一个进程的运行转换到另一个进程上运行，在这个过程中，进程的运行环境发生了实质性的变化。

就绪到运行，调度器选择该进程，分配 CPU

运行到就绪，进程时间片用完，被抢占，回到就绪队列

运行到阻塞，进程发起 IO 请求或等待某个事件，主动放弃 CPU

阻塞到就绪，等待的事件发生，进程被唤醒，进入就绪队列

进程间通信

信号

本质：操作系统向进程发送的异步软中断通知

核心用途：

通知进程发生了某个事件，如按下 Ctrl 和 C 键发送 SIGINT 信号

进程间简单通信，只能传递信号编号，不能传递复杂数据

进程终止，如使用 kill 命令加数字 9 强制杀死进程

关键特点：

优点：最简单最基础的 IPC

优点：异步执行，不阻塞进程

缺点：只能传递整数信号编号，不能传递复杂数据

缺点：不可靠，信号可能丢失

面试高频考点：

常见信号包括 SIGINT 对应数字 2、SIGKILL 对应数字 9、SIGTERM 对应数字 15、SIGCHLD 对应数字 17

SIGKILL 和 SIGSTOP 不能被捕获、阻塞或忽略

子进程退出时会向父进程发送 SIGCHLD 信号

常用命令：

列出所有信号的命令是 kill 减 l

强制杀死进程的命令是 kill 减 9 加进程 ID

优雅终止进程的命令是 kill 减 15 加进程 ID

信号量

本质：一个内核维护的计数器，用于实现进程间的同步与互斥

核心用途：

保护临界资源，如共享内存，防止多个进程同时访问

实现进程间的同步，如生产者消费者模型

关键特点：

优点：专门用于同步互斥，不传递数据

优点：支持二元信号量即互斥锁和计数信号量

缺点：单独使用无意义，必须配合其他 IPC 机制

面试高频考点：

P 操作即减 1 操作，申请资源，若计数为 0 则阻塞

V 操作即加 1 操作，释放资源，唤醒等待的进程

信号量的生命周期由内核管理，进程退出不会自动释放

常用命令：

查看系统中信号量的命令是 ipcs 减 s

删除指定信号量的命令是 ipcrm 减 s 加信号量 ID

共享内存

本质：内核分配的一块物理内存区域，多个进程可以将其映射到自己的虚拟地址空间

核心用途：进程间高速大量数据传输，是最快的 IPC 机制

关键特点：

优点：速度最快，直接读写物理内存，无需内核拷贝

优点：支持多个进程同时访问

缺点：没有同步机制，必须配合信号量使用

缺点：生命周期独立于进程，进程退出不会自动释放

面试高频考点：

共享内存有自己的引用计数 nattch 即连接数

只有当 nattch 等于 0 且标记了 IPC_RMID 时，才会被内核释放

fork 之后子进程会自动继承父进程的共享内存连接

常用命令：

查看系统中共享内存的命令是 ipcs 减 m

删除指定共享内存的命令是 ipcrm 减 m 加共享内存 ID

核心 API：

shmget 函数用于创建或获取共享内存

shmat 函数用于将共享内存映射到进程地址空间

shmdt 函数用于解除映射

shmctl 函数用于控制共享内存，如删除

消息队列

本质：内核维护的一个消息链表，进程可以向队列中添加消息或从中读取消息

核心用途：进程间发送结构化数据

关键特点：

优点：全双工通信

优点：支持按消息类型选择性读取

优点：消息会被内核持久化，直到被读取

缺点：有大小限制，不适合传输大量数据

缺点：比共享内存慢，需要内核拷贝

面试高频考点：

消息队列的生命周期由内核管理

可以实现非阻塞读写

支持优先级，通过消息类型实现

常用命令：

查看系统中消息队列的命令是 ipcs 减 q

删除指定消息队列的命令是 ipcrm 减 q 加消息队列 ID

管道

本质：内核中的一个环形缓冲区，实现进程间的字节流通信

分类：

匿名管道：只能用于有亲缘关系的进程，如父子、兄弟进程

命名管道即 FIFO：可以用于任意进程之间的通信

核心用途：简单的单向数据传输

关键特点：

优点：简单易用，是 shell 命令管道的基础

缺点：半双工，同一时间只能单向传输

缺点：面向字节流，无消息边界

缺点：匿名管道生命周期随进程结束而结束

面试高频考点：

管道有两端，读端和写端

当写端关闭，读端读完所有数据后会返回 0

当读端关闭，写端继续写会收到 SIGPIPE 信号

常用命令：

创建命名管道的命令是 mkfifo 加管道名称

套接字

本质：一种通用的通信接口，支持本地进程间通信和跨网络通信

核心用途：

本地进程间通信，使用 UNIX 域套接字

网络通信，使用 TCP 或 UDP 套接字

关键特点：

优点：最通用的 IPC 机制

优点：全双工通信

优点：支持跨主机跨网络

缺点：实现相对复杂

面试高频考点：

UNIX 域套接字比 TCP 套接字快，因为不需要网络协议栈处理

TCP 是面向连接的可靠字节流，UDP 是无连接的不可靠数据报

信号和信号量的区别

本质不同：信号是异步通知，信号量是计数器

用途不同：信号用于发送事件通知，信号量用于同步与互斥

传递数据不同：信号只能传递信号编号，信号量不传递数据

执行方式不同：信号是异步的，信号量是同步的

管道和消息队列的区别

通信方式不同：管道是半双工，消息队列是全双工

数据格式不同：管道是字节流无边界，消息队列是结构化消息有边界

选择性读取不同：管道不能选择性读取只能按顺序读，消息队列可以按消息类型读取

生命周期不同：匿名管道随进程结束，消息队列由内核管理独立于进程

共享内存和消息队列的区别

速度不同：共享内存最快无内核拷贝，消息队列较慢需要内核拷贝

数据量不同：共享内存适合大量数据，消息队列适合小量结构化数据

同步机制不同：共享内存无同步机制需配合信号量，消息队列内核提供同步

实现复杂度不同：共享内存实现复杂度较高，消息队列实现复杂度较低

匿名管道和命名管道的区别

使用范围不同：匿名管道只能用于有亲缘关系的进程，命名管道可以用于任意进程

存在形式不同：匿名管道存在于内存中，命名管道存在于文件系统中是特殊文件

生命周期不同：匿名管道随进程结束，命名管道直到被删除才会消失

常易错点

  1. 共享内存的生命周期：进程退出不会自动释放共享内存，必须手动调用 shmctl 函数加 IPC_RMID 参数或使用 ipcrm 命令删除

  2. 信号的可靠性：SIGKILL 和 SIGSTOP 不能被捕获、阻塞或忽略

  3. 管道的半双工特性：同一时间只能单向传输，要实现双向通信需要两个管道

  4. 信号量的释放：进程异常退出时，信号量不会自动释放，可能导致死锁

  5. fork 与 IPC 的关系：

    子进程会继承父进程的管道、共享内存连接、信号量

    子进程不会继承父进程的消息队列

进程调度

进程调度是操作系统内核选择下一个要运行的进程的过程

进程调度的目标

公平性，每个进程都能获得合理的 CPU 时间

吞吐量，单位时间内完成的进程数量

响应时间，用户操作到系统响应的时间

周转时间，进程从提交到完成的总时间

实时性，满足实时进程的时间要求

Linux调度器

主调度器

当进程主动放弃cpu时调用，如进程阻塞

周期调度器

每隔固定时间运行一次，检查是否需要抢占当前进程

上下文切换

当调度器选择一个新进程运行时，保存当前进程的上下文，加载新进程的上下文的过程

上下文

下文包括寄存器值、程序计数器、栈指针、页表等

发生时机

进程主动放弃 CPU，如阻塞、调用 sleep

进程时间片用完被抢占

中断处理完成后返回用户态

开销

保存和恢复上下文的 CPU 开销

缓存失效的开销，因为新进程的数据可能不在 CPU 缓存中

调度策略

Linux分为两大类调度策略：实时调度策略和普通调度策略

实时调度策略

用于对于时间要求严格的进程，优先级高于所有普通进程

SCHED_FIFO（先进先出）

同一优先级的进程按先来先服务的顺序运行

进程一旦获得cpu，就会一致运行直到主动放弃或者被更高优先级的进程抢占，没有时间片概念

SCHED_RR（时间片轮转）

同一优先级的进程按照时间片轮转运行

每个进程用完自己的时间片后，会被放到同优先级队列的末尾

解决了SCHED_FIFO可能导致低优先级进程饥饿问题

实时进程优先级范围时1-99，数字越大优先级越高

普通调度策略

用于大多数普通进程，Linux 默认使用 CFS 完全公平调度器

SCHED_OTHER，默认的普通进程调度策略，使用 CFS 调度

SCHED_BATCH，批处理调度，适合后台运行的批处理任务，会降低优先级，减少抢占

SCHED_IDLE，空闲调度，只有当系统没有其他进程运行时才会被调度

CFS完全公平调度器

CFS 是 Linux 2.6.23 版本之后引入的默认调度器，核心思想是完全公平

虚拟运行时间 vruntime，每个进程的实际运行时间乘以权重系数得到的时间

权重，根据进程的 nice 值计算得到，nice 值越低，权重越高，相同实际运行时间对应的 vruntime 增长越慢

nice 值范围是负 20 到正 19，默认是 0

调度周期，CFS 会把一个调度周期内的 CPU 时间按照权重分配给所有就绪进程

调度周期的长度根据就绪进程的数量动态调整

工作原理

CFS 维护一个红黑树，所有就绪进程按照 vruntime 从小到大排序

每次调度时，CFS 选择红黑树中最左边的节点，也就是 vruntime 最小的进程运行

进程运行时，它的 vruntime 不断增加

当进程的 vruntime 超过其他进程时，就会被调度器抢占，回到红黑树中

CFS 的优点：

完全公平，每个进程获得的 CPU 时间与它的权重成正比

响应速度快，适合交互式进程
