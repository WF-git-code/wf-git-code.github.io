DIE (Detect It Easy)

DIE 是目前逆向领域识别率最高的开源查壳工具，支持 PE/ELF/Mach-O 全平台文件，特征库持续更新，是查壳的首选主力工具。

1. 基础操作

  1. 启动 die.exe，直接将目标 exe/dll 文件拖入主窗口，工具自动完成扫描。
  2. 识别模糊时，点击右侧「深度扫描」按钮，遍历全文件特征提升识别准确率。
  3. 支持批量拖入多个文件，批量返回查壳结果。

2. 核心结果解读

主界面直接输出最关键的三项结论：

  • Compiler：程序的编译器与开发语言，如 Microsoft Visual C++、MinGW GCC，用于判断程序开发栈。
  • Packer：检测到的加壳 / 保护工具，该字段为空则大概率未加壳。
  • Entry point：程序入口点相对偏移（RVA），脱壳时用于对照真实入口 OEP。

3. 进阶分析标签页

  • Sections（节区）：查看所有内存段的名称、大小、权限。正常程序包含 .text（代码段）、.data（数据段）、.rdata、.rsrc 等标准节区；加壳后常出现 UPX0、.vmp 等非标准节区。
  • Entropy（熵值）：量化数据加密程度，熵值越接近 8 说明加密程度越高。正常代码段熵值约 5~6，全加密段接近 7.5~8。
  • Imports（导入表）：查看程序调用的系统 API。加壳程序导入表通常极度精简，仅保留 LoadLibrary、GetProcAddress 等核心加载函数。

4. 实用设置

官方原生支持简体中文，通过 Options → Language → 简体中文 即可切换，无需第三方汉化包。

Exeinfo PE

Exeinfo PE 是轻量级 PE 文件分析工具，体积小巧，除查壳外可查看完整 PE 结构细节，常作为辅助工具交叉验证结果。

1. 基础操作

直接将目标文件拖入窗口，工具自动扫描并在底部输出结论；也可点击左上角 File 按钮手动选择文件。

2. 核心信息字段

  • Entry Point：入口点地址与所属节区
  • Linker Info：链接器版本，辅助判断编译器版本
  • SubSystem：程序子系统，GUI 为图形界面，Console 为控制台程序
  • Overlay：是否存在附加数据，脱壳时需完整保留，否则程序损坏
  • 底部诊断栏：直接输出壳类型与编译器信息，附带基础脱壳思路提示

3. 辅助功能

内置简易 PE 编辑器，可修改节区属性、导入表等 PE 结构；附带部分入门壳的脱壳插件，可快速处理常见压缩壳。

查壳方法步骤

  1. 所有工具都失效时，通过 PE 结构四个维度判断，准确率远高于工具自动扫描，也是逆向工程师的基本功。

    1. 看节区结构

    用 DIE 的节区标签、CFF Explorer 等 PE 编辑器查看：

    • 无壳特征：标准节区命名 .text .rdata .data .rsrc .reloc，节区数量 5 个左右。
    • 加壳特征：节区名为非标准名称（UPX0、.vmp、.themida等）；节区数量极少（仅 2-3 个）；存在同时具备「可读 + 可写 + 可执行」权限的大段内存；虚拟大小远大于磁盘大小。

    2. 看导入表

    • 无壳特征：导入几十到上百个 API，覆盖 kernel32、user32、gdi32 等多个系统库。
    • 加壳特征：导入函数极度精简，通常仅保留 LoadLibraryA/W、GetProcAddress、VirtualProtect 等核心加载与内存修改函数，真实 API 由壳在运行时动态填充。

    3. 看入口点代码

    用 x64dbg 打开程序，观察入口处的汇编指令：

    • 无壳特征：标准栈帧初始化（32 位 push ebp; mov ebp, esp，64 位 sub rsp, xxx），逻辑清晰。
    • 加壳特征：入口直接保存全部寄存器（pushad/pushfd）；大量异或、循环解密逻辑；代码短小，执行几步后就发生跨段大跳转。

    4. 看熵值

    熵值量化数据的无序程度，加密 / 压缩后的数据熵值会显著升高：

    • 无壳代码段：熵值约 5~6.5

    • 加壳加密段

      ：熵值大于 6.8，强加密壳接近 7.5~8

      DIE 的 Entropy 标签可直接查看各节区熵值，高熵值节区基本可以确定存在加密 / 压缩。

查壳软件失效

最常见：识别不出壳（显示 Unknown / 无壳）

这是 90% 的 “失效” 场景，核心原因是壳的特征不在工具的签名库中，并非工具损坏。

典型原因
  1. 新型 / 自定义 / 变种壳：自研壳、修改过特征的改版壳、新发布的商业保护壳，工具特征库未收录，静态特征匹配失效。
  2. 反查壳对抗：壳主动修改节区名称、伪装 PE 头、插入花指令、添加虚假特征，故意干扰特征码匹配。
  3. 多重加壳：程序嵌套两层及以上的壳，工具只能匹配最外层，甚至完全无法识别。
  4. 代码虚拟化保护：VMProtect、Themida 这类强壳会将代码转为虚拟机指令，静态特征完全改变，常规查壳工具很难精准识别。
解决步骤

  1. 开启深度 / 启发式扫描

    DIE 点击界面右下角「扫描」按钮启用深度扫描，会遍历全文件字节做特征匹配，比默认快速扫描识别率高很多；命令行模式可添加

    参数开启深度检测。

  2. 多工具交叉验证

    不要只依赖一款工具，按「DIE → Exeinfo PE → PEiD」顺序依次检测，至少两款工具结论一致才可确认。不同工具的特征库互补，能覆盖更多壳类型。

  3. 更新工具与特征库

    查壳工具的识别能力完全依赖签名库，老版本识别不出新壳是正常现象。DIE 前往官方 GitHub 更新最新版，新版会同步新增壳的特征规则。

  4. 转向手动结构分析

    工具只是自动匹配特征，手动分析 PE 结构是 100% 准确的兜底方案，详见本文第四节。

脱壳篇

壳的核心作用是保护原始代码与数据，按保护强度可分为压缩壳与加密保护壳，脱壳思路差异显著。

壳的基础分类

壳类型 代表工具 核心特点 标准处理方案
压缩壳 UPX、ASPack、FSG 仅压缩代码与数据，无反调试机制 可完整脱壳，还原后与原程序一致
加密保护壳 VMProtect、Themida、Enigma 代码虚拟化、反调试、完整性校验 不建议完整脱壳，采用动态调试对抗分析
安装包壳 NSIS、Inno Setup 包裹真实程序的安装容器 提取内部真实程序后再分析

UPX 压缩壳脱壳实战

UPX 是最经典的开源压缩壳，其脱壳思路是所有压缩壳的通用范式，也是逆向入门与面试的必考点。

1. 核心原理

UPX 将原始程序的代码段、数据段进行压缩，程序运行时壳代码先执行，在内存中解密还原原始代码与导入表，最后跳转至真实程序入口点（OEP）执行。

x64dbg 手动脱壳（找 OEP）

手动脱壳的核心本质可以一句话概括：在壳程序完成内存自解密、即将跳转到原始程序入口的瞬间，把内存中已经还原的干净程序镜像提取出来，再修复 PE 文件结构，得到可正常运行、可静态分析的原始程序。所有脱壳技巧本质上都是在 “精准抓住这个时间窗口”。

加壳程序运行时到底发生了什么

理解壳的执行流程，是理解所有脱壳方法的前提。一个加壳后的 exe，运行时会经历完整的 5 个阶段：

  1. 系统加载

    操作系统加载器把加壳文件映射到内存，此时代码段、数据段都是加密压缩状态，无法直接执行。程序的入口点（Entry Point）指向壳自身的代码，而非原始程序。

  2. 环境保存

    壳代码第一行通常是 pushad / pushfd（32 位），把所有寄存器、标志位压栈保存，保证解密过程不破坏原始程序的运行环境。

  3. 自解密还原

    壳执行解密算法，把加密的代码段、数据段在内存中逐段解密，还原成原始的明文指令和数据。此时内存里已经有了完整的原始程序，但控制权还在壳手里。

  4. 重建导入表（IAT）

    这是所有壳都绕不开的一步：原始程序的导入表被壳加密隐藏了，壳会调用 LoadLibrary 加载依赖的 DLL，再循环调用 GetProcAddress 获取所有 API 的真实地址，把原始导入表（IAT）在内存中重新填充完整。

  5. 跳转至真实入口（OEP）

    壳执行 popad / popfd 恢复寄存器环境，最后执行一个跨段大跳转（jmp），跳转到原始程序的入口点（Original Entry Point，简称 OEP）。跳转完成后，原始程序开始正常执行，和未加壳时完全一致。

脱壳的黄金时机：第 5 步跳转的前一刻。此时内存中的程序已经 100% 解密完成、导入表也已填充，是最完整的原始状态。

手动脱壳的三大核心环节

所有手动脱壳操作，最终都对应这三步，缺一不可。

1. 定位真实入口点 OEP

这是手动脱壳最核心、难度最高的一步，所有脱壳技巧都是围绕 “怎么精准找到跳转到原程序的那个位置” 展开。

  • 为什么不能直接搜？：原始程序的代码在磁盘上是加密的，静态看不到；内存里只有等壳解密后才会出现。
  • 成功标志：跳转后看到标准的程序入口代码（32 位典型的 push ebp / mov ebp, esp 栈帧，64 位的 sub rsp, xxx 初始化），且代码所在的内存段属于原始程序的代码段，不再是壳的代码段。
2. 内存转储（Dump）

把内存中已经完全解密的程序镜像，完整保存成本地的 exe 文件。

  • 为什么不能直接复制内存？：磁盘上的 PE 文件是文件对齐，内存中的 PE 是内存对齐，两者的节区偏移、大小都不一样，直接保存会导致 PE 结构错乱。
  • 转储工具做了什么：修正节区的文件偏移和大小，把内存对齐的结构重新映射成符合 PE 规范的磁盘文件结构。
3. PE 结构修复

直接 Dump 出来的文件通常无法直接运行，必须修复三个核心部分：

  • 导入表（IAT）修复

    内存里的 IAT 填充的是当前运行时的 API 绝对地址，下次程序加载基址变了就会失效。修复工具会根据内存中的 API 地址，反向重建标准的导入表结构（包含 DLL 名、函数名 / 序号），让程序下次加载时能由系统自动填充地址。

  • 重定位表修复

    开启 ASLR 的程序每次加载基址都不同，需要重定位表修正硬编码地址。如果脱壳时丢失重定位信息，程序只能在特定基址运行，换环境就会崩溃。

  • 资源段与附加数据修复

    图标、菜单、字符串等资源，以及文件尾部的附加数据（Overlay），必须完整保留，否则程序界面、功能会异常。

手动脱壳是面试核心考点，以下介绍两种通用方法：

方法一：API 断点法（通用性最强）

核心原理：

所有用户态的壳，只要是 PE 程序，就必须调用 GetProcAddress 来重建导入表，这是壳的命门。壳调用该 API 的阶段，一定处于 “代码已解密、但还没跳 OEP” 的区间。

我们给 GetProcAddress 下断点，等它最后一次返回后，壳的导入表重建工作就全部完成了，接下来很快就会执行跳转到 OEP 的指令。此时单步跟踪很短的距离，就能找到最终的大跳转。

适用场景：几乎所有压缩壳、简单加密壳，是最稳妥的兜底方法。

核心依据：所有压缩壳都必须重建导入表，因此必然调用 GetProcAddress。

  1. 将加壳后的程序拖入 x64dbg，程序停在壳的入口点。
  2. 底部命令栏输入 bp GetProcAddress 下 API 断点，按 F9 运行。
  3. 断点触发后，按 Ctrl+F9 执行到返回，回到壳代码。
  4. 按 F8 单步步过，很快会遇到一个跨段大跳转 jmp。
  5. 跳转后代码呈现标准 C 程序入口特征，此处即为真实 OEP。
  6. 使用 Scylla 插件转储内存并修复导入表，得到可运行的脱壳文件。
方法二：ESP 定律法（经典技巧）

  1. 核心原理：

    壳在入口处会用 pushad 把所有寄存器压栈保存；解密完成、准备跳 OEP 前，一定会用 popad 恢复所有寄存器。此时栈指针 ESP 会回到程序刚入口时的位置，栈顶的数据变回最初保存的寄存器值。

    我们对入口处 ESP 指向的栈内存，设置硬件访问断点，当壳执行 popad 读取栈数据时，断点就会触发。此时距离最终的 OEP 跳转通常只剩几步指令，稍加单步就能找到。

    适用场景：绝大多数压缩壳，尤其是 UPX 这类结构简单的壳，通用性极强。

方法三：单步跟踪法

核心原理：

壳的代码量通常极小，核心就是解密循环 + 填充导入表，全程都在壳自己的代码段内执行；只有最后一步，会执行一个跨段的远跳转，跳到原始程序的代码段。

我们从入口开始一直按 F8 步过，只要遇到一个跳转到完全陌生内存区域的 jmp，跳过去之后代码风格从 “解密循环” 变成 “正常业务逻辑”，这里就是 OEP。

适用场景：简单壳、其他方法失效时，缺点是耗时长。

方法四：内存断点执行法

核心原理：

原始程序的代码段初始是加密状态，权限不可执行；壳解密完成后，会调用 VirtualProtect 修改代码段为可执行权限，然后才会跳过去执行。

我们对原始代码段的内存区域设置内存执行断点，第一次触发断点时，就是壳第一次跳转到原始代码的时刻，也就是 OEP 附近。

4. 脱壳后验证

  1. 用查壳工具重新检测，确认无壳特征。
  2. 运行脱壳后程序，验证功能与原程序完全一致。
  3. 拖入 IDA 分析，确认函数、字符串可正常识别。

静态分析篇

静态分析是在不运行程序的前提下，通过反汇编、反编译还原程序逻辑，IDA Pro 是静态逆向分析的行业标准工具。

IDA Pro

1. 程序加载与初始分析

  1. 启动 IDA，拖入目标 exe 文件，保持默认选项点击 OK。
  2. 等待自动分析完成，IDA 会自动识别函数、字符串、导入表等结构。
  3. 64 位 EXE 默认加载基址通常为 0x140000000，32 位通常为 0x400000。

2. 核心功能窗口

  • 函数窗口（Functions）：列出所有识别出的函数，可按名称、地址筛选。
  • 字符串窗口（Strings）：程序内所有字符串常量，是定位业务功能的最快入口，快捷键 Shift+F12。
  • 导入表（Imports）：程序调用的所有系统 API 与第三方函数，可快速判断程序功能范围。
  • 反汇编窗口：核心分析区，显示汇编指令与注释。
  • Hex-Rays 反编译窗口：将汇编还原为伪 C 代码，大幅提升分析效率，快捷键 F5。

3. 常用操作快捷键

快捷键 功能
G 输入地址快速跳转
N 重命名函数 / 变量
; 添加行注释
X 查看交叉引用（谁调用了当前函数 / 引用了当前字符串）
F5 打开 Hex-Rays 反编译视图
Shift+F12 快速打开字符串窗口

4. 标准逆向分析流程

  1. 打开字符串窗口，搜索业务关键词定位功能代码。
  2. 双击字符串跳转到引用位置，通过交叉引用回溯找到所属函数。
  3. 按 F5 查看伪 C 代码，梳理函数逻辑与参数传递关系。
  4. 重命名关键函数与变量，补充注释，逐步完善分析图谱。
  5. 定位核心校验、加密、业务逻辑函数，标记关键地址用于动态调试。

IDA 与x86dbg协同

联动的核心逻辑是：IDA 里的是静态默认基址，x64dbg 里的是运行时实际基址。受 Windows ASLR 地址随机化影响，两者通常不一样，但相对偏移（RVA）永远不变，联动的本质就是用 RVA 做中转换算。

方法 1：段窗口查看（最标准，推荐）

这是逆向工程师最常用的方式，直观准确。

  1. 按快捷键 Shift + F7 打开「段（Segments）窗口」

  1. 列表中

    地址最小的段的起始地址

    ，就是整个程序的加载基址（ImageBase）

    • 32 位 EXE 默认通常为 0x00400000
    • 64 位 EXE 默认通常为 0x0000000140000000（即简写的 0x140000000）

  2. 你之前用 DIE 查到的「基址（BA）」数值，和这里完全一致，两者通用。

方法 2：重基址窗口直接看

适合快速确认，不用翻段表。

  1. 顶部菜单依次点击 Edit → Segments → Rebase program
  2. 弹出窗口的 New base 输入框里，默认显示的数值就是当前 IDA 的加载基址
  3. 看完直接点「Cancel」取消即可，不要随意修改，否则所有地址都会偏移错乱Hex-Rays。

  2. 用 DIE 查到该函数的 RVA（相对偏移，比如 0x1800）

  3. 基址 = 虚拟地址 - RVA

同步

1. RVA 手动偏移法（无插件）

核心原理：同一代码的相对虚拟地址（RVA）恒定，与加载基址无关。

在 x64dbg 中按 Ctrl+G 输入计算出的实际地址，即可精准定位对应代码。

2. MAP 文件批量同步

将 IDA 分析结果批量同步到调试器：

  1. IDA 中执行 File → Produce file → Create MAP file 导出符号表。
  2. x64dbg 中执行 文件 → 导入 → 选择 MAP 文件。
  3. 导入完成后，调试器可直接搜索 IDA 中命名的所有函数。

3. ret-sync 实时同步插件

进阶高效方案，实现 IDA 与 x64dbg 双向实时联动：

  • 调试器单步跳转时，IDA 自动定位对应代码。
  • IDA 中设置的断点可同步到调试器。
  • 支持 Hex-Rays 反编译窗口联动。

动态调试篇

动态调试通过运行程序、设置断点、跟踪执行流，验证静态分析结论，观察运行时状态。x64dbg（包含 32 位 x32dbg 版本，常统称为 x86dbg）是 Windows 平台主流的用户态调试器。

主界面

x64dbg 界面与核心功能

CPU 标签页是调试核心，分为四个协同工作的核心面板：

1. 反汇编面板（左上）

显示当前线程的汇编指令，黄色高亮行为下一条即将执行的指令，由 RIP 指令指针寄存器指向。

  • 核心操作：按 F2 设置 / 取消软件断点，按 F4 运行到选中行，按 Ctrl+G 跳转到指定地址，按 空格 修改汇编指令临时测试逻辑。

2. 寄存器面板（右上）

实时显示 CPU 寄存器与标志位状态，是查看参数、返回值、程序状态的核心。

  • RAX：默认存放函数返回值，函数执行完成后查看该寄存器即可获取返回结果。
  • RCX / RDX / R8 / R9：x64 调用约定下前 4 个函数参数。
  • RSP：栈指针，永远指向当前栈顶，与栈面板第一行对应。
  • 标志位：亮显为 1，灰显为 0，直接决定跳转指令是否生效。

3. 内存转储面板（左下）

查看任意地址的原始内存数据，支持切换 ASCII、Unicode、十六进制等显示格式。

  • 右键寄存器选择「在内存转储中跟随」，可快速查看指针指向的具体数据。
  • 支持搜索字符串、特征字节，用于定位加密数据、缓冲区内容。

4. 栈面板（右下）

显示当前栈顶数据，用于查看函数参数、局部变量、函数返回地址，回溯调用链。

断点体系与使用场景

断点类型 设置方式 适用场景
软件断点 F2 /bp 命令 绝大多数普通代码下断，通过改写指令为 int3 实现
硬件断点 右键 → 断点 → 硬件断点 反调试场景、代码段只读无法改写的情况
内存断点 内存映射中右键设置 监控数据读写、跟踪内存访问行为
条件断点 F6 编辑条件 过滤大量无效触发，仅特定条件下中断
API 断点 符号页搜索后下断 跟踪系统调用、拦截文件 / 网络操作

单步调试操作

快捷键 功能 使用场景
F7 步入（Step Into） 进入 call 函数内部，深入分析函数逻辑
F8 步过（Step Over） 跳过系统 API、已知功能函数，不进入内部
Ctrl+F9 执行到返回 快速跳出当前函数，回到调用位置
F9 继续运行 恢复程序执行，直到下一个断点或异常
Ctrl+F2 重启调试 重新加载程序，保留所有断点配置

标准调试实战流程

以密码验证程序为例，完整演示调试流程：

  1. 加载程序，切换到字符串标签，搜索 “密码错误” 等业务关键词。
  2. 双击跳转到对应反汇编代码，向上回溯找到校验函数入口。
  3. 在函数入口按 F2 设置断点，按 F9 运行程序。
  4. 输入测试密码触发断点，程序暂停后查看参数寄存器确认输入内容。
  5. 单步跟踪比较逻辑，观察标志位变化判断跳转走向。
  6. 验证函数返回值，确认校验逻辑，可临时修改跳转或返回值测试效果。

常见问题与注意事项

  • ASLR 地址随机化：开启 ASLR 的程序每次运行基址不同，禁止使用绝对地址下断，改用函数名或 RVA 偏移。
  • 停在系统模块：程序刚加载时常停在 ntdll.dll 等系统模块，按 Alt+M 打开模块列表，双击目标程序模块切回用户代码。
  • 断点不触发：确认断点位置在可执行代码段，检查程序是否加壳、目标代码是否尚未解密加载。
