一.I/O框架

I/O框架以库函数的形式，封装了较为底层的系统调用。这些库函数经过时间与压力的考验，更加合理与高效，通常是Reactor和Proactor模式，而Libevent就是一种基于Reaclor模式实现的轻量级I/O库。

先简单介绍Proactor与reactor模式

Proactor模式（主动器模型）

windows系统使用主动器模型，异步I/O

Proactor 模式是异步 I/O 的设计模式，可以把它理解为 “主动型的 I/O 管理者”：

  • 你（应用程序）只需要告诉 Proactor“要做什么 I/O 操作（比如读文件、收网络数据）”，并留下 “做完后通知我的方式（回调函数）”；
  • Proactor 会全权负责：等待 I/O 就绪 → 执行实际的 I/O 读写操作 → 操作完成后调用你的回调函数，把结果（数据 / 错误）返回给你；
  • 对比更常见的 Reactor 模式（反应器）：Reactor 只通知你 “I/O 就绪了”，实际的读写操作需要你自己做，而 Proactor 是 “一站式服务”

工作流程

用 “读取网络数据” 为例，Proactor 的核心步骤如下：

  1. 发起异步请求：应用程序向 Proactor 提交异步读请求，包含：要读取的套接字、数据缓冲区、操作完成后的回调函数。
  2. 内核等待并执行 I/O：Proactor（通常由操作系统内核 / 框架实现）接管请求，内核等待套接字有数据可读，主动将数据从内核缓冲区拷贝到应用程序缓冲区。
  3. 回调通知结果：当 I/O 操作完成（成功 / 失败），Proactor 触发应用程序注册的回调函数，传递操作结果（如读取到的数据、错误码）。

Proactor 模式的适用场景

  1. 高并发 I/O 场景：如异步 Web 服务器（aiohttp）、大数据传输、消息队列消费，减少应用层对 I/O 操作的管理成本；
  2. 操作系统支持异步 I/O 的环境：如 Windows 的 IOCP（I/O 完成端口）、Linux 的 io_uring（新一代异步 I/O），Proactor 模式能充分利用内核级异步能力；
  3. 希望简化编程模型的场景：无需区分 “I/O 就绪” 和 “I/O 操作”，只需关注 “操作结果”，代码更简洁。

Reactor模式（反应堆模型）

Linux系统使用的是Reactor模型，底层采用epoll

Reactor 模式可以理解为 “事件驱动的 I/O 反应器”，核心是 “只监控、只通知，不干活”：

  • 你（应用程序）需要先向 Reactor 注册 “关注的 I/O 事件”（比如 “某个套接字有数据可读”“某个文件可写”），并绑定对应的 “事件处理函数”；
  • Reactor 的核心是一个事件循环（Event Loop），它会持续监控所有注册的 I/O 事件，等待事件就绪（比如套接字有数据到达）；
  • 一旦检测到某个 I/O 事件就绪，Reactor 会立刻 “反应”—— 调用你预先注册的处理函数，但不会帮你执行实际的 I/O 读写操作；
  • 实际的读写操作（比如从内核缓冲区拷贝数据到应用缓冲区），需要你在处理函数中自己完成。

Reactor的工作流程

以 “监听网络端口并读取客户端数据” 为例，Reactor 的核心步骤如下：

  1. 注册事件：应用程序向 Reactor 注册 “监听套接字的连接事件”+“连接成功后的处理函数”，以及 “客户端套接字的读事件”+“读就绪后的处理函数”。
  2. 事件循环监控：Reactor 启动事件循环，通过 I/O 多路复用机制（如 select/poll/epoll）持续监控所有注册的套接字，等待事件就绪。
  3. 事件分发：当检测到 “监听套接字有新连接” 或 “客户端套接字有数据可读” 时，Reactor 将就绪事件分发给对应的处理函数。
  4. 应用执行 I/O 操作：处理函数被调用后，由应用程序自己执行实际的 I/O 操作（比如 accept () 建立连接、recv () 读取数据），并处理数据。

核心区别

Reactor ：就绪通知 监控I/O就绪事件——通知应用 应用执行I/O操作

Proactor: 完成通知 监控I/O事件完成——通知应用 由框架或内核自己执行

两者都要通过注册回调函数，只不过是Proector模式是通过回调函数将结果或者错误返回给程序调用者；而Reactor是通过回调函数执行通知的具体的I/O事件。

Proactor模式通过回调函数拿内核执行I/O后的结果 触发时机在I/O完成后

Reactor模式通过具体的回调函数执行相对应的I/O操作 触发时机在I/O就绪后

Reactor 是 “就绪通知 + 开发者做 I/O”，Proactor 是 “内核做 I/O + 完成通知 + 开发者拿结果

Proactor通过结构化的异步I/O请求（指令包包含了对谁做，做什么，数据放到哪里，做到哪，做完怎么说）内核会解析这个指令包并完成相应的IO操作后通知开发者，开发者通过回调函数拿取结果或者错误。

Libevent在windows和linux下的差异

windows

IOCP是windows内核提供的异步I/O管理机制

Proactor 模型的 “终极实现”：IOCP 完美契合 Proactor 的核心逻辑 —— 应用提交异步 I/O 请求，内核执行全流程，完成后将结果放入 IOCP，应用仅处理结果。

内核做的事：仅负责 I/O 操作本身（等待设备就绪 → 拷贝数据到应用缓冲区），最终给应用返回 “原料”（比如读取到的文件内容、客户端发来的网络数据、写入成功的状态）；

应用做的事：拿到这些 “原料” 后，执行所有业务逻辑（比如解析数据、计算、生成响应、调用数据库、写日志等），内核完全不参与。

windows系统下优先采用Proactor（主动器）模型

windows的核心异步I/O接口IOCP是天生的Proactor接口

  • 接口逻辑：应用只需调用ReadFile/WriteFile（带FILE_FLAG_OVERLAPPED标记）向内核提交 “异步 I/O 请求”，无需关注 “是否就绪”；
  • 内核行为：内核自动接管全流程 —— 等待 I/O 就绪（比如网卡收到数据、文件可读取）→ 执行实际 I/O（拷贝数据到应用缓冲区）→ 把 “完成结果” 放入 IOCP 队列；
  • 应用行为：应用只需从 IOCP 队列中取 “完成事件”（或通过回调接收），处理结果即可，无需执行任何 I/O 操作。

Windows 没有像 epoll 那样的 “就绪通知” 接口，核心异步 I/O 接口就是 Proactor 式的，用 Proactor 模型是 “顺势而为”，而非刻意选择。

linux

Linux：epoll 是 “天生的 Reactor 接口”

Linux 长期以来的核心高并发 I/O 接口是epoll（以及更早的 select/poll），它的设计逻辑完全贴合 Reactor 模型：

  • 接口逻辑：epoll 的核心是epoll_ctl（注册事件）+ epoll_wait（等待就绪），只负责 “监控 I/O 就绪事件并通知”；
  • 内核行为：内核仅告诉应用 “某个套接字 / 文件可以读 / 写了”，不做任何实际的 I/O 拷贝操作；
  • 应用行为：应用必须在收到 “就绪通知” 后，自己调用recv/send/read等函数执行 I/O 操作，否则拿不到数据。

更关键的是：Linux 早期的异步 I/O 接口（libaio）存在严重缺陷 —— 仅支持O_DIRECT（绕开文件缓存），不支持普通文件的异步读写，易用性和兼容性极差，开发者根本没法基于它构建稳定的 Proactor 模型，只能选择 epoll+Reactor

总结

Windows 选择 Proactor：牺牲一点控制粒度，换 “更少的内核 / 用户态切换”，适配 IOCP 的高并发优势；

Linux 选择 Reactor：牺牲一点切换次数，换 “全场景适配 + 精细控制”，利用 epoll 的极致性能。

底层接口决定模型：Windows IOCP 是 Proactor 式接口，天然适配 Proactor 模型；Linux epoll 是 Reactor 式接口，天然适配 Reactor 模型；

历史生态固化选择：Windows 围绕 IOCP 构建了 Proactor 生态，Linux 围绕 epoll 构建了 Reactor 生态；

性能适配权衡：Windows 优先 “少切换、高并发”，Linux 优先 “全兼容、细控制”；

Reactor模式

基于Reactor模式的I/O框架包含如下组件：

句柄

I/O框架要处理的对象，即I/O事件，信号事件和定时器事件，统一称为事件源，一个事件源和一个句柄绑定在一起。

句柄的作用：当内核检测到就绪事件，通过句柄来通知程序该事件发生，在linux系统下I/O事件对应的句柄是文件描述符，信号事件位信号值。

事件多路分发器

事件的到来是随机的，所以我们要循环的等待并处理事件，这就是事件循环，在事件循环中等待事件一般使用I/O复用技术来实现，I/O框架库将系统支持的I/O复用系统封装为统一的接口，称为事件多路分发器，事件分发器底层的demultiplex方法是等待事件的核心函数，内部调用是select poll epoll等。

事件处理器和具体事件处理器

事件处理器执行对应的业务逻辑，它通常包含一个或者多个handel_event回调函数，I/O框架库提供的事件处理器通常是一个接口，开发者要继承它来实现自己具体的事件处理器。

事件处理器还提供一个get_handle方法，返回与事件处理器关联的句柄。

当事件多路分发器检测到有事件发生时，通过句柄来通知应用程序，因此必须将事件处理器与句柄绑定，才能在事件发生时获取正确的事件处理器。

Reactor时I/O框架库的核心

handel_events 该方法执行事件循环 它重复如下过程：等待事件，然后依次处理所有就绪事件对应的事件处理器。

register_handler 该方法调用时间多路分发器的register_event方法在事件多路分发器中注册事件

remove_handler 该方法调用时间多路分发器的remove_event方法在事件多路分发器中删除事件
