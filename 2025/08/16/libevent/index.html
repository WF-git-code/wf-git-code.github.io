一.I/O框架

I/O框架以库函数的形式，封装了较为底层的系统调用。这些库函数经过时间与压力的考验，更加合理与高效，通常是Reactor和Proactor模式，而Libevent就是一种基于Reaclor模式实现的轻量级I/O库。

先简单介绍Proactor与Reactor模式

Proactor模式（主动器模型）

windows系统使用主动器模型，异步I/O

Proactor 模式是异步 I/O 的设计模式，可以把它理解为 “主动型的 I/O 管理者”：

  • 你（应用程序）只需要告诉 Proactor“要做什么 I/O 操作（比如读文件、收网络数据）”，并留下 “做完后通知我的方式（回调函数）”；
  • Proactor 会全权负责：等待 I/O 就绪 → 执行实际的 I/O 读写操作 → 操作完成后调用你的回调函数，把结果（数据 / 错误）返回给你；
  • 对比更常见的 Reactor 模式（反应器）：Reactor 只通知你 “I/O 就绪了”，实际的读写操作需要你自己做，而 Proactor 是 “一站式服务”

工作流程

用 “读取网络数据” 为例，Proactor 的核心步骤如下：

  1. 发起异步请求：应用程序向 Proactor 提交异步读请求，包含：要读取的套接字、数据缓冲区、操作完成后的回调函数。
  2. 内核等待并执行 I/O：Proactor（通常由操作系统内核 / 框架实现）接管请求，内核等待套接字有数据可读，主动将数据从内核缓冲区拷贝到应用程序缓冲区。
  3. 回调通知结果：当 I/O 操作完成（成功 / 失败），Proactor 触发应用程序注册的回调函数，传递操作结果（如读取到的数据、错误码）。

Proactor 模式的适用场景

  1. 高并发 I/O 场景：如异步 Web 服务器（aiohttp）、大数据传输、消息队列消费，减少应用层对 I/O 操作的管理成本；
  2. 操作系统支持异步 I/O 的环境：如 Windows 的 IOCP（I/O 完成端口）、Linux 的 io_uring（新一代异步 I/O），Proactor 模式能充分利用内核级异步能力；
  3. 希望简化编程模型的场景：无需区分 “I/O 就绪” 和 “I/O 操作”，只需关注 “操作结果”，代码更简洁。

Reactor模式（反应堆模型）

Linux系统使用的是Reactor模型，底层采用epoll

Reactor 模式可以理解为 “事件驱动的 I/O 反应器”，核心是 “只监控、只通知，不干活”：

  • 你（应用程序）需要先向 Reactor 注册 “关注的 I/O 事件”（比如 “某个套接字有数据可读”“某个文件可写”），并绑定对应的 “事件处理函数”；
  • Reactor 的核心是一个事件循环（Event Loop），它会持续监控所有注册的 I/O 事件，等待事件就绪（比如套接字有数据到达）；
  • 一旦检测到某个 I/O 事件就绪，Reactor 会立刻 “反应”—— 调用你预先注册的处理函数，但不会帮你执行实际的 I/O 读写操作；
  • 实际的读写操作（比如从内核缓冲区拷贝数据到应用缓冲区），需要你在处理函数中自己完成。

Reactor的工作流程

以 “监听网络端口并读取客户端数据” 为例，Reactor 的核心步骤如下：

  1. 注册事件：应用程序向 Reactor 注册 “监听套接字的连接事件”+“连接成功后的处理函数”，以及 “客户端套接字的读事件”+“读就绪后的处理函数”。
  2. 事件循环监控：Reactor 启动事件循环，通过 I/O 多路复用机制（如 select/poll/epoll）持续监控所有注册的套接字，等待事件就绪。
  3. 事件分发：当检测到 “监听套接字有新连接” 或 “客户端套接字有数据可读” 时，Reactor 将就绪事件分发给对应的处理函数。
  4. 应用执行 I/O 操作：处理函数被调用后，由应用程序自己执行实际的 I/O 操作（比如 accept () 建立连接、recv () 读取数据），并处理数据。

核心区别

Reactor ：就绪通知 监控I/O就绪事件——通知应用 应用执行I/O操作

Proactor: 完成通知 监控I/O事件完成——通知应用 由框架或内核自己执行

两者都要通过注册回调函数，只不过是Proector模式是通过回调函数将结果或者错误返回给程序调用者；而Reactor是通过回调函数执行通知的具体的I/O事件。

Proactor模式通过回调函数拿内核执行I/O后的结果 触发时机在I/O完成后

Reactor模式通过具体的回调函数执行相对应的I/O操作 触发时机在I/O就绪后

Reactor 是 “就绪通知 + 开发者做 I/O”，Proactor 是 “内核做 I/O + 完成通知 + 开发者拿结果

Proactor通过结构化的异步I/O请求（指令包包含了对谁做，做什么，数据放到哪里，做到哪，做完怎么说）内核会解析这个指令包并完成相应的IO操作后通知开发者，开发者通过回调函数拿取结果或者错误。

Libevent在windows和linux下的差异

windows

IOCP是windows内核提供的异步I/O管理机制

Proactor 模型的 “终极实现”：IOCP 完美契合 Proactor 的核心逻辑 —— 应用提交异步 I/O 请求，内核执行全流程，完成后将结果放入 IOCP，应用仅处理结果。

内核做的事：仅负责 I/O 操作本身（等待设备就绪 → 拷贝数据到应用缓冲区），最终给应用返回 “原料”（比如读取到的文件内容、客户端发来的网络数据、写入成功的状态）；

应用做的事：拿到这些 “原料” 后，执行所有业务逻辑（比如解析数据、计算、生成响应、调用数据库、写日志等），内核完全不参与。

windows系统下优先采用IOCP（Proactor（主动器））模型

windows的核心异步I/O接口IOCP是天生的Proactor接口

  • 接口逻辑：应用只需调用ReadFile/WriteFile（带FILE_FLAG_OVERLAPPED标记）向内核提交 “异步 I/O 请求”，无需关注 “是否就绪”；
  • 内核行为：内核自动接管全流程 —— 等待 I/O 就绪（比如网卡收到数据、文件可读取）→ 执行实际 I/O（拷贝数据到应用缓冲区）→ 把 “完成结果” 放入 IOCP 队列；
  • 应用行为：应用只需从 IOCP 队列中取 “完成事件”（或通过回调接收），处理结果即可，无需执行任何 I/O 操作。

Windows 没有像 epoll 那样的 “就绪通知” 接口，核心异步 I/O 接口就是 Proactor 式的，用 Proactor 模型是 “顺势而为”，而非刻意选择。

linux

Linux：epoll 是 “天生的 Reactor 接口”

Linux 长期以来的核心高并发 I/O 接口是epoll（以及更早的 select/poll），它的设计逻辑完全贴合 Reactor 模型：

  • 接口逻辑：epoll 的核心是epoll_ctl（注册事件）+ epoll_wait（等待就绪），只负责 “监控 I/O 就绪事件并通知”；
  • 内核行为：内核仅告诉应用 “某个套接字 / 文件可以读 / 写了”，不做任何实际的 I/O 拷贝操作；
  • 应用行为：应用必须在收到 “就绪通知” 后，自己调用recv/send/read等函数执行 I/O 操作，否则拿不到数据。

更关键的是：Linux 早期的异步 I/O 接口（libaio）存在严重缺陷 —— 仅支持O_DIRECT（绕开文件缓存），不支持普通文件的异步读写，易用性和兼容性极差，开发者根本没法基于它构建稳定的 Proactor 模型，只能选择 epoll+Reactor

macos

在 macOS 上，libevent 会优先使用 kqueue 作为 I/O 多路复用的后端机制（kqueue 是 macOS/BSD 系统特有的高效事件通知接口）

kqueue可以理解为更加强大的epoll，其支持的事件类型更多更丰富，具体请看I/O多路复用部分。

总结

无论后端I/O复用用的是什么模型，libevent库始终是基于Reactor模型实现的

Windows 选择 Proactor：牺牲一点控制粒度，换 “更少的内核 / 用户态切换”，适配 IOCP 的高并发优势；

Linux 选择 Reactor：牺牲一点切换次数，换 “全场景适配 + 精细控制”，利用 epoll 的极致性能。

底层接口决定模型：Windows IOCP 是 Proactor 式接口，天然适配 Proactor 模型；Linux epoll 是 Reactor 式接口，天然适配 Reactor 模型；

历史生态固化选择：Windows 围绕 IOCP 构建了 Proactor 生态，Linux 围绕 epoll 构建了 Reactor 生态；

性能适配权衡：Windows 优先 “少切换、高并发”，Linux 优先 “全兼容、细控制”；

Reactor模式

Reactor属于一个典型的事件驱动该模型，程序执行流程由**事件（IO就绪、定时器超时、信号触发）**驱动，而非顺序执行。

其事件驱动模型的核心流程，程序初始化进入事件循环，阻塞事件循环，事件就绪后调用回调函数进行处理，处理完成进入事件循环持续等待。

基于Reactor模式的I/O框架包含如下组件：

事件源（句柄）

Linux 上是文件描述符，Windows上就是Socket或者Handle 了，这里统一称为“句柄 集”。

I/O框架要处理的对象，即I/O事件，信号事件和定时器事件，统一称为事件源，一个事件源和一个句柄绑定在一起。

句柄的作用：当内核检测到就绪事件，通过句柄来通知程序该事件发生，在linux系统下I/O事件对应的句柄是文件描述符，信号事件位信号值。

事件多路分发器

事件的到来是随机的，所以我们要循环的等待并处理事件，这就是事件循环，在事件循环中等待事件一般使用I/O复用技术来实现，I/O框架库将系统支持的I/O复用系统封装为统一的接口，称为事件多路分发器，事件分发器底层的demultiplex方法是等待事件的核心函数，内部调用是select poll epoll等。

事件处理器和具体事件处理器

事件处理器执行对应的业务逻辑，它通常包含一个或者多个handel_event回调函数，I/O框架库提供的事件处理器通常是一个接口，开发者要继承它来实现自己具体的事件处理器。

事件处理器还提供一个get_handle方法，返回与事件处理器关联的句柄。

当事件多路分发器检测到有事件发生时，通过句柄来通知应用程序，因此必须将事件处理器与句柄绑定，才能在事件发生时获取正确的事件处理器。对应到libevent中就是event结构体。

两种实现方式

第一种设计（多回调函数）

子类需要实现四个 handle_* 函数，每个函数职责单一，逻辑清晰，易于维护。

缺点：如果子类只关心部分事件（如只处理读），仍需实现所有四个函数（可空实现），存在代码冗余。

扩展性差，会破坏现有子类接口

无分支判断性能略高，但需要维护事件类型与对应函数映射表，实现复杂度较高

多回调设计：清晰、职责单一，但扩展性差，适合事件类型稳定的场景。

第二种设计（统一回调函数）

子类只需实现一个 handle_events 函数，通过位运算（if (events & READ)）分支处理不同事件。

优点：代码更紧凑，无需为不关心的事件写空实现；缺点：逻辑集中，分支判断多，可读性稍差。

扩展性较好，只需要在events掩码中增加新的位，框架与应用解耦，秩序传递掩码并不关心具体事件处理逻辑

子类内部事件需要解析掩码进行分支判断，但框架实现更简单，事件映射灵活

统一回调设计：灵活、紧凑，适合需要动态扩展事件类型的高性能框架。

Rector模式的优点

响应快：不必为单个同步时间所阻塞，虽然Rector本身依然是同步的

编程相对简单：可以最大程度的避免复杂的多线程及同步问题，避免线程和进程之间的切换开销

可扩展性：可以方便的通过增加Rector的实例来充分利用cpu资源

可复用性：Rector框架本身与具体的处理逻辑无关，具有高复用性

I/O多路复用

IO多路复用是**用一个系统调用监听多个文件描述符（fd）**的技术，任意fd就绪时系统调用返回，程序再遍历就绪列表处理事件。

libevent支持的IO多路复用模型（Linux下默认epoll）：

模型 支持系统 核心特点
epoll Linux 2.6+ 性能最高，支持边缘触发，fd数量无上限，时间复杂度O(1)
kqueue FreeBSD/macOS 高性能，支持文件、信号等多种事件
select 全平台通用 跨平台兼容性好，但性能低，fd上限默认1024，时间复杂度O(n)
poll 全平台通用 无fd上限，但性能仍低于epoll
iocp Windows 异步IO（Proactor模式）

epoll比select性能更高的核心原因：

  1. 事件表机制：通过epoll_ctl预先将fd加入内核事件表，无需每次调用都传递全量fd；

  2. 就绪列表：内核维护就绪列表，epoll_wait直接返回就绪fd，无需遍历全量fd；

  3. 内存共享：内核与用户空间共享内存，避免就绪列表的拷贝开销。

kueue（了解）

kqueue 是 FreeBSD/macOS/iOS 等 BSD 系操作系统上的 IO 多路复用与事件通知机制，定位完全对应 Linux 上的 epoll，但设计理念更通用、更强大 —— 它不仅能监听 socket IO 事件，还能监听文件、信号、定时器、进程状态变化等多种事件类型。

通用事件模型：不局限于 socket IO，支持 EVFILT_READ（读）、EVFILT_WRITE（写）、EVFILT_SIGNAL（信号）、EVFILT_TIMER（定时器）、EVFILT_PROC（进程）等多种过滤器；

一次性注册，多次触发：类似 epoll 的 ET（边缘触发），但更灵活；

用户态与内核态交互高效：通过 kevent() 一个系统调用完成「注册事件 + 等待事件 + 获取就绪事件」，减少系统调用次数；

无 fd 数量上限：仅受系统内存限制，类似 epoll。

维度 epoll（Linux） kqueue（BSD/macOS）
事件类型 仅支持 socket IO 事件（可读 / 可写 / 出错） 通用事件模型，支持 IO、信号、定时器、进程、文件等多种事件
系统调用次数 需 3 个系统调用：epoll_create()（创建）、epoll_ctl()（注册）、epoll_wait()（等待） 仅需 2 个系统调用：kqueue()（创建）、kevent()（注册 + 等待二合一）
定时器实现 无原生定时器，需结合时间堆（如 libevent 那样） 原生支持 EVFILT_TIMER 过滤器，内核直接管理定时器，无需用户态时间堆
使用复杂度 相对简单，专注于 IO 多路复用 更灵活，但 API 稍复杂，支持多种事件类型
跨平台 仅 Linux 仅 BSD 系（FreeBSD/macOS/iOS）

事件处理流程

当应用程序向libevent注册一个事件后，libevent处理流程

①首先初始化event,设置事件类型和回调函数

②向libevent中添加事件event，对于定时事件，libevent使用一个小根堆进行管理，key为超时时间，对于Sianal和I/O事件，libevent将其放入到等待链表（waitList）,这是一个双向链表结构。

③程序调用event_base_dispatch()系列函数进入无限循环，等待事件，以select()函数为例；每次循环前libevent会检查定时事件的最小超时事件tv，根据tv设置select()的最大等待事件，以便后面及时处理超时事件，后检查I/O事件。

Libevent将所有就绪事件，放到激活链表中，然后对激活链表中的事件，调用事件的回调函数执行事件处理。

三.核心API

事件循环核心API

API 核心作用
struct event_base* event_base_new() 创建并初始化事件循环对象，自动选择最优IO后端
int event_base_dispatch(struct event_base *base) 进入事件循环，阻塞直到loopbreak/loopexit被调用
int event_base_loopbreak(struct event_base *base) 立刻中断事件循环，不处理剩余就绪事件
int event_base_loopexit(struct event_base *base, const struct timeval *tv) 优雅退出，处理完当前就绪事件后退出，支持超时等待
void event_base_free(struct event_base *base) 释放事件循环对象内存

事件操作核心API

参数详解：

  1. base：事件循环对象指针；

  2. fd：socket事件传文件描述符，定时器事件传-1，信号事件传信号编号；

  3. events：事件类型宏，常用组合：

    • EV_READ/EV_WRITE：监听fd可读/可写；

    • EV_PERSIST：持续触发，不设置则触发一次后自动移除；

四.为什么需要缓冲事件（bufferevent）？

bufferevent 主要为带缓冲的 socket fd 设计，但并非只服务于 socket fd，它是 libevent 中「带缓冲的通用 IO 事件封装」，只是 socket 是其最核心、最常用的场景。

  1. 管道（pipe/socketpair）
  • 核心 API：bufferevent_pair_new()（创建一对关联的 bufferevent，基于 socketpair 管道）；
  • 典型场景：进程 / 线程间通信（比如 libevent 内部用它处理信号事件，把异步信号转为同步 IO）。
  1. 文件 fd（有限支持）
  • 对普通文件 fd，bufferevent 也能封装，但文件 IO 是「阻塞 / 非阻塞无意义」的（文件没有内核缓冲区满的情况），因此 bufferevent 的缓冲区优势体现不出来；
  • 典型场景：极少用，一般直接用 read()/write()，但 API 层面完全兼容。
  1. 过滤型 bufferevent（Filter bufferevent）
  • 这是「bufferevent 套 bufferevent」的模式，不直接绑定 fd，而是绑定另一个 bufferevent；
  • 典型场景：SSL/TLS 加密（bufferevent_openssl_new()）、数据编解码（比如 protobuf 序列化 / 反序列化）。

bufferevent

buffervent是libevent对socket事件的带缓冲封装，它在普通 event 的基础上，自动管理输入 / 输出双缓冲区、自动处理非阻塞 IO 的读写、自动触发回调

之前写的 Echo 服务器，用 bufferevent_read() 直接读完整数据，不用自己循环 read() 处理 EAGAIN，也不用自己拼包处理粘包，这就是 bufferevent 的价值。

裸用 event 的痛点 bufferevent 的解决方案
1. 手动处理非阻塞 IO 的 EAGAIN 错误非阻塞 socket 读 / 写时，若缓冲区空 / 满，会返回 EAGAIN，需手动监听 EV_READ/EV_WRITE 重试 自动管理非阻塞 IO内部自动处理 EAGAIN，自动监听 / 取消监听读写事件，用户完全不用管
2. 手动处理粘包 / 拆包TCP 是流协议，一次 read() 可能读不完一个包，或读多个包，需手动维护缓冲区拼包 自动维护输入缓冲区数据自动写入输入缓冲区，用户回调时直接从缓冲区取完整数据（或按需取），简化粘包处理
3. 手动管理写缓冲区满的情况非阻塞 socket 写时，若内核发送缓冲区满，会返回 EAGAIN，需手动把剩余数据存起来，监听 EV_WRITE 再写 自动维护输出缓冲区用户调用 bufferevent_write() 时，数据自动写入输出缓冲区；若写满，自动监听 EV_WRITE，socket 可写时自动把剩余数据写入，用户完全不用管

bufferevent核心结构

bufferevent的核心工作流程

bufferevent 的 “底层” 就是普通 event + evbuffer + 系统调用，它把这些包起来，给你一个简单的接口。****

  1. 读流程（socket 有数据 → 用户读回调）——————

  2. socket 内核读缓冲区有数据 → 触发底层读事件；

  3. bufferevent 自动读数据 → 从 socket 内核读缓冲区读入用户态输入缓冲区；

  4. 触发用户读回调 → 调用你设置的 read_cb；

  5. 用户取数据 → 在 read_cb 中调用 bufferevent_read() 从输入缓冲区取数据（Echo 服务器中直接回写）。

  6. **写流程（用户写数据 → socket 发送） **——————

  7. 用户调用 bufferevent_write() → 数据写入用户态输出缓冲区；

  8. bufferevent 自动监听写事件 → 若输出缓冲区有数据，自动监听 socket 可写；

  9. socket 可写时自动发送 → 从输出缓冲区取数据写入 socket 内核写缓冲区；

  10. （可选）触发写回调 → 若输出缓冲区数据全部写完，触发你设置的 write_cb。

核心问题：

bufferevent和event的区别？

定位不同：普通 event 是通用事件对象，可监听 IO、信号、定时器；bufferevent 是专门针对 socket 的带缓冲封装，仅用于 socket 网络 IO。

缓冲区不同：普通 event 无缓冲区，需用户手动维护；bufferevent 有输入 / 输出双缓冲区，自动管理数据读写。

复杂度不同：普通 event 需手动处理非阻塞 IO 的 EAGAIN、粘包、写缓冲区满；bufferevent 自动处理这些细节，API 更简单。

适用场景不同：普通 event 适合通用事件、简单 IO；bufferevent 适合网络编程，尤其是 TCP 流协议的通信。

bufferevent 如何处理粘包？

bufferevent并不会自动解决粘包，但是提供了自动维护的输入缓冲区，简化了粘包处理。

裸用 event 时，需自己维护一个用户态缓冲区，每次 read() 后拼包，判断是否有完整包；

用 bufferevent 时，数据会自动写入输入缓冲区，你只需在 read_cb 中：

  • 调用 bufferevent_read() 读取输入缓冲区的数据；
  • 根据协议判断是否有完整包（比如先读 4 字节长度，再读对应长度的包体）；
  • 若有完整包，取出处理；若没有，留在输入缓冲区，等下次 read_cb 触发再处理。

bufferevent 的输出缓冲区满了怎么办？

  1. 当你调用 bufferevent_write() 时，若输出缓冲区未满，数据直接写入缓冲区，bufferevent 自动监听 socket 可写；
  2. 若输出缓冲区满了，bufferevent_write() 仍会成功返回（不会报错），数据会被追加到输出缓冲区；
  3. bufferevent 会持续监听 socket 可写事件，一旦 socket 可写，自动从输出缓冲区取数据写入 socket，直到输出缓冲区清空。

bufferevent_setwatermark() 可以设置水位线，控制 “输入缓冲区有多少数据时触发读回调”“输出缓冲区低于多少时触发写回调”，进一步优化性能。

bufferevent缓冲区是动态扩容的，理论上只要系统存在物理内存就可以无限扩容

如果系统内存真的耗尽了，bufferevent_write() 会返回 -1，并设置 errno 为 ENOMEM，但这是系统级的错误，不是 “缓冲区满” 的逻辑错误

libevent 允许你通过 bufferevent_setwatermark() 设置两个水位线：

参数详解：

events：EV_READ（设置输入缓冲区水位线）或 EV_WRITE（设置输出缓冲区水位线）；

lowmark：低水位线；

highmark：高水位线。

输出缓冲区高 / 低水位线的真正作用

（控制发送速度，避免 “本地发太快，对端收太慢”）

  1. 当输出缓冲区的数据量 ≥ 高水位线时，说明 “我发得太快，对端收得太慢”，此时 libevent 会自动暂停监听本地 socket 的 EV_READ（不要再读对端的新数据了，先把本地输出缓冲区里的数据发完）；
  2. 当输出缓冲区的数据量 ≤ 低水位线时，说明 “发得差不多了”，此时 libevent 会自动恢复监听本地 socket 的 EV_READ（继续读对端的新数据）。

输入缓冲区高 / 低水位线的真正作用

（控制接收速度，避免 “对端发太快，本地收太慢 / 读太慢，导致内存爆炸”）

  1. 当输入缓冲区的数据量 ≥ 高水位线时，说明 “对端发得太快，我收得太慢 / 读得太慢”，此时 libevent 会自动暂停监听本地 socket 的 EV_READ（不要再读对端的新数据了，先把本地输入缓冲区里已收到的数据读走处理完）；
  2. 当输入缓冲区的数据量 ≤ 低水位线时，说明 “已收数据读得差不多了”，此时 libevent 会自动恢复监听本地 socket 的 EV_READ（继续读对端的新数据）。

一句话总结：高水位线是 “流量控制开关”，不是 “写入拦截器”。

bufferevent输入/输出缓冲区底层结构：

bufferevent 的输出缓冲区是「动态链表结构（evbuffer）」，无固定物理上限

bufferevent 的底层缓冲区是 libevent 自己实现的 evbuffer，它的结构是链表式的内存块，不是固定大小的数组：

当你调用 bufferevent_write() 追加数据时，evbuffer 会先看当前最后一个内存块有没有剩余空间；

如果有剩余空间，直接写进去；

如果没有剩余空间，申请新的内存块，追加到链表后面，再写数据；

只要系统内存够，这个链表可以无限延长，永远不会因为 “缓冲区满” 而阻止你写入。

bufferevent怎么监听socket的读事件？

每个 bufferevent 内部，至少有两个普通的 struct event：

每个 bufferevent 内部都有一个专门监听 socket EV_READ 的普通 struct event；

当输出缓冲区触达高水位线时，libevent 会对这个内部读 event 调用 event_del()；

这会进一步触发内核层面的操作（比如 Linux 下的 epoll_ctl(EPOLL_CTL_DEL)），告诉内核不要再监听这个 socket 的可读事件；

这样，即使对端发了新数据，epoll_wait 也不会返回，数据会留在对端的发送缓冲区，对端会暂停发送，从而实现流量控制。

当数据量降到低水位线时，libevent 会再对这个读 event 调用 event_add()，恢复监听。”

底层流程

因为bufferevent对象内部封装了两个event对象，因此不用维护event与fd之间的关系，可以直接通过API暂停对读/写事件的监听，但对于普通event当移除事件监听表（不释放内存）的时候需要保存event的指针，再次添加进事件循环

一个是 bev_read，绑定同一个 fd，监听 EV_READ，内部回调负责把数据从 socket 读到输入 evbuffer；

一个是 bev_write，绑定同一个 fd，监听 EV_WRITE，内部回调负责把数据从输出 evbuffer 写到 socket。

之所以用两个独立的 event，是因为 TCP 是全双工的，读和写完全独立，可以同时进行，用两个 event 可以独立控制读和写的监听状态，逻辑也更清晰。

比如输出缓冲区触达高水位线时，我们可以只暂停 EV_READ，但继续监听 EV_WRITE 发数据，非常灵活。”

因此，可以绑定两个回调函数。

事件的「待执行池」，存储所有已就绪、等待执行回调的事件

    • 数组下标对应事件优先级，数字越小优先级越高；
    • libevent 会严格按优先级从高到低遍历链表，执行每个事件的回调函数；
    • 回调执行完成后：设置了EV_PERSIST的事件会回到对应的监听列表 / 堆，未设置的事件会被直接移除。

就绪阶段：IO / 信号事件触发、定时器超时，事件会从对应的监听结构中取出，按优先级加入第四列的活跃事件列表；

event_del 是 event_add 的逆操作，同样围绕 event_base 的核心成员展开：

  1. 状态检查：若 event 未注册（ev->ev_flags == EVLIST_INIT），直接返回；

什么是时间轮

时间轮是一种模拟钟表指针转动的高效定时器管理数据结构，核心思想是 “批量处理、按时间槽分组”，把定时器按 “到期时间” 分到不同的 “时间槽” 里，通过指针的周期性转动来处理到期的定时器。

时间轮分类

单层时间轮

  • 结构：只有一个表盘，一个指针；
  • 缺点：如果定时器的超时时间跨度很大（比如 1 秒和 1 小时），表盘需要做的非常大，内存浪费严重；
  • 适用场景：超时时间跨度小、精度要求固定的场景。

多层时间轮（Hierarchical Time Wheel，主流方案）

  • 结构：多个不同精度的表盘嵌套，比如：
    • 第一层（秒轮）：60 个槽，精度 1 秒，处理 0-59 秒的定时器；
    • 第二层（分钟轮）：60 个槽，精度 1 分钟，处理 1-59 分钟的定时器；
    • 第三层（小时轮）：24 个槽，精度 1 小时，处理 1-24 小时的定时器；
  • 工作原理：
    • 秒轮指针每秒转一格，处理秒级定时器；
    • 秒轮转满一圈（60 秒），分钟轮指针转一格，把分钟轮当前槽里的定时器 “降级” 到秒轮的对应槽里；
    • 以此类推，实现大跨度超时时间的高效管理；
  • 优势：内存占用小，能高效处理各种跨度的超时时间，是 Netty、Kafka、Skynet 等主流高性能网络库 / 框架的首选。

最小堆VS时间轮

维度 最小堆（libevent 默认方案） 时间轮（Netty/Kafka 方案）
插入定时器的时间复杂度 O(logN) O (1)（直接放到对应时间槽）
删除定时器的时间复杂度 O(logN) O (1)（如果是双向链表管理时间槽）
处理到期定时器的时间复杂度 O (K*logN)，K 是到期定时器数量 O (K)，K 是到期定时器数量（直接遍历当前时间槽）
内存占用 低，只存储实际的定时器 相对高，需要预分配时间槽（多层时间轮已优化）
通用性 极高，适合所有场景，无特殊限制 较高，适合超时时间相对规律、高频触发的场景
实现复杂度 非常简单，代码量少 相对复杂，多层时间轮的降级逻辑需要仔细处理
典型适用场景 通用网络库（如 libevent）、定时器数量适中的场景 高性能网络库（如 Netty）、消息队列（如 Kafka）、游戏服务器（如 Skynet）、定时器数量巨大且高频触发的场景

为什么 libevent 默认用最小堆，而不用时间轮？

这是面试的高频追问，核心原因是 libevent 的设计目标是 “通用、简单、跨平台”，而不是 “极致性能”：

  1. 通用性优先：
    • libevent 是一个通用的事件驱动库，需要适配各种场景（从嵌入式设备到高性能服务器）；
    • 最小堆没有任何场景限制，不管定时器数量多少、超时时间跨度多大，都能稳定工作；
    • 时间轮虽然在高频场景下性能更好，但在定时器数量极少、超时时间极不规律的场景下，优势不明显，甚至可能有内存浪费。
  2. 实现简单、维护成本低：
    • 最小堆的实现非常简单，代码量少，逻辑清晰，跨平台兼容性好；
    • 多层时间轮的实现相对复杂，需要处理 “降级” 逻辑，维护成本更高；
    • libevent 作为一个开源通用库，简单、可维护是非常重要的设计目标。
  3. 性能足够满足大多数场景：
    • 对于大多数通用网络编程场景，定时器数量不会特别巨大（比如几千、几万），O (logN) 的复杂度完全可以接受；
    • 只有在定时器数量特别巨大（比如几十万、上百万）且高频触发的场景下，时间轮的 O (1) 优势才会明显体现。

补充：libevent 也有时间轮补丁！

虽然 libevent 默认用最小堆，但社区里有基于时间轮的优化补丁（比如 libevent-timerwheel），如果你需要在 libevent 中使用时间轮，可以集成这些补丁，或者自己实现一个简单的时间轮作为补充。

结论

什么是时间轮：时间轮是模拟钟表指针转动的定时器管理结构，把定时器按到期时间分到不同时间槽，通过指针转动批量处理，多层时间轮可以高效处理大跨度超时时间，Netty、Kafka 都用它。

最小堆 vs 时间轮：

  • 最小堆的插入、删除是 O (logN)，处理到期是 O (K*logN)，但实现简单、通用性极高、内存占用小；
  • 时间轮的插入、删除、处理到期都是 O (1)（或 O (K)），高频场景下性能更好，但实现相对复杂、内存占用稍高。

libevent 为什么选最小堆：

  • 因为 libevent 的设计目标是通用、简单、跨平台，最小堆没有场景限制，实现简单，维护成本低，且对于大多数通用场景，O (logN) 的性能完全足够；
  • 时间轮虽然在高频场景下性能好，但不是所有场景都需要，libevent 作为通用库，优先选择了更通用、更简单的方案；
  • 当然，如果有极致性能需求，libevent 也有时间轮的社区补丁可以集成。”

libevent 是怎么实现跨平台的？

libevent 是通过 struct eventop 这个 IO 后端抽象接口，结合策略模式实现跨平台的：

  1. 它把所有 IO 多路复用模型的核心操作，抽象成 init/add/del/dispatch 等统一的函数接口；
  2. 针对不同操作系统的 IO 模型（Linux 的 epoll、macOS 的 kqueue、全平台的 select 等），分别实现这套接口，封装成独立的 struct eventop 实例；
  3. 初始化 event_base 时，libevent 会按优先级自动选择当前系统支持的最优 IO 后端，上层事件循环只需要通过统一接口调用，完全不需要关心底层实现，从而实现了跨平台兼容。

event_base_loop 中 evsel->dispatch 做了什么？

evsel->dispatch 是 IO 后端的事件分发核心函数，对应 event_base_loop 的阻塞等待逻辑：

  1. 它接收一个超时时间参数，这个参数来自定时器最小堆的最近超时时间；
  2. 内部调用对应 IO 模型的阻塞函数（比如 epoll 的 epoll_wait、kqueue 的 kevent），阻塞等待事件就绪；
  3. 当有事件就绪或超时后，它会把所有就绪的 IO 事件，通过 event_io_active 加入 event_base 的活跃事件列表；
  4. 最后返回，由上层的 event_process_active 统一执行所有就绪事件的回调。

十二.时间管理

时间管理模块

libevent的时间管理分为四个模块

核心模块 核心目标 对应 struct event_base 成员
时间获取与缓存 减少 gettimeofday()/clock_gettime() 系统调用次数 tv_cache、tv_clock_diff、last_updated_clock_diff

时间获取与缓存

核心痛点

gettimeofday()/clock_gettime() 是系统调用，频繁调用会有性能开销（虽然单次开销很小，但在高性能网络库中，每秒可能调用成千上万次，积少成多）。

ibevent 的解决方案：时间缓存 tv_cache

  • 核心逻辑：在一次事件循环迭代中，只调用一次 gettimeofday()，把结果缓存到 event_base.tv_cache 中；后续所有需要获取当前时间的地方，直接读缓存，不调用系统调用。

  • 对应 event_base_loop 源码：

  • gettime(base, &base->event_tv); // 【1】调用系统调用，获取当前时间，存入 event_tv（用于时间校正）
clear_time_cache(base); // 【2】清空旧的时间缓存

res = evsel->dispatch(base, tv_p); // 【3】阻塞等待事件，可能耗时较长

update_time_cache(base); // 【4】dispatch 返回后，更新时间缓存 tv_cache

定时器管理（最小堆 + 公共超时队列）

核心数据结构 1：最小堆 timeheap

  • 核心优势：插入、删除定时器 O (logN)，获取最近超时的定时器 O (1)；
  • 对应 struct event_base 成员：struct min_heap timeheap；
  • 对应核心函数：
    • timeout_next(base, &tv_p)：取堆顶的绝对超时时间，计算与当前时间的差值，作为 dispatch() 的阻塞参数；
    • timeout_process(base)：遍历时间堆，把所有 ev_timeout ≤ 当前时间的定时器取出，加入活跃事件列表。

核心数据结构 2：公共超时队列 common_timeout_queues

这是 libevent 的性能优化方案，用于优化大量相同超时时间的定时器场景：

  • 核心痛点：如果有 10000 个定时器，超时时间都是 1 秒，每个都单独插入最小堆，会有 10000 次 O (logN) 操作，性能开销大；
  • libevent 的解决方案：
    • 把相同超时时间的定时器，放到同一个「公共超时队列」里；
    • 每个公共超时队列，只在最小堆里插入一个「代表定时器」；
    • 当「代表定时器」超时时，批量处理这个队列里的所有定时器；
  • 对应 struct event_base 成员：
    • struct common_timeout_list **common_timeout_queues：公共超时队列数组；
    • int n_common_timeouts：当前使用的公共超时队列数量；
    • int n_common_timeouts_allocated：公共超时队列数组的总长度。

超时处理（核心流程）

总结

时间获取与缓存：为了减少 gettimeofday() 系统调用次数，libevent 用 tv_cache 缓存当前时间，一次事件循环迭代中只调用一次系统调用，后续直接读缓存；如果系统支持 CLOCK_MONOTONIC 单调时钟，还会优先用它，避免墙上时间跳变的影响。

时间校正：用 timeout_correct() 处理系统时间跳变，每次事件循环开始时对比当前时间和上次保存的 event_tv，如果时间差超过阈值，就遍历时间堆校正所有定时器的绝对超时时间，保证定时器不会提前触发或永远不触发。

定时器管理：默认用最小堆 timeheap 管理定时器，插入、删除 O (logN)，获取最近超时 O (1)；对于大量相同超时时间的定时器，还提供了公共超时队列 common_timeout_queues 优化性能，把相同超时的定时器放到一个队列，只在堆里插入一个代表定时器，超时后批量处理。

超时处理：通过 timeout_next() 取堆顶的最近超时时间作为 dispatch() 的阻塞参数，dispatch() 返回后通过 timeout_process() 遍历堆，把所有已超时的定时器加入活跃事件列表，和 I/O、信号事件统一调度。

十三.libevent支持多线程

多线程

Libevent 本身不是多线程安全的，在多核的时代，如何能充分利用CPU的能力呢

事件循环是单线程的：一个 event_base 同一时间只能在一个线程中运行 event_base_loop()（对应 struct event_base.running_loop 防止重入的标记）；

提供了完善的线程安全机制：通过互斥锁、条件变量、socketpair 唤醒机制，支持多线程安全地操作 event_base（如 event_add/event_del）、唤醒事件循环；

需要用户显式启用：libevent 默认不启用线程安全（避免单线程场景下的锁开销），需要用户调用 evthread_use_pthreads()（Linux）或 evthread_use_windows_threads()（Windows）显式启用。

libevent 默认不启用线程安全，需要显式调用 evthread_use_pthreads()（Linux）或 evthread_use_windows_threads()（Windows），必须在创建 event_base 之前调用。

一个 event_base 一个线程：这是最推荐的模式，每个线程运行一个独立的 event_base，线程之间通过 socketpair 或管道通信，避免锁竞争，性能最好；

不要多个线程同时运行同一个 event_base 的 event_base_loop()：libevent 用 running_loop 标记防止重入，多个线程同时运行会直接报错；

多线程操作同一个 event_base 时，必须启用线程安全：否则会出现竞态条件，导致程序崩溃；

耗时操作不要放在事件回调中：事件回调是在 event_base_loop() 线程中执行的，如果回调耗时太长，会阻塞整个事件循环，影响其他事件的处理；耗时操作应该放到线程池里执行，执行完后通过 event_add() 唤醒事件循环。

总结

libevent 的事件循环本身是单线程的，但它提供了完善的线程安全机制，完全支持在多线程环境下安全使用：

  1. 核心结论：一个 event_base 同一时间只能在一个线程中运行 event_base_loop()，但可以在其他线程中安全地调用 event_add()/event_del()/event_base_loopbreak() 等函数。
  2. 线程安全机制：
    • 互斥锁 th_base_lock：保护 event_base 的核心数据结构，避免竞态条件；
    • 条件变量 current_event_cond：等待当前事件回调执行完成，避免释放时崩溃；
    • socketpair 唤醒机制 th_notify_fd：跨线程唤醒阻塞的事件循环，和信号事件的处理机制一致。
  3. 启用方式：libevent 默认不启用线程安全（避免单线程锁开销），需要在创建 event_base 之前显式调用 evthread_use_pthreads()（Linux）或 evthread_use_windows_threads()（Windows）。
  4. 最佳实践：推荐一个线程一个 event_base，线程之间通过 socketpair 通信，避免锁竞争；耗时操作不要放在事件回调中，应该放到线程池里执行。

与线程池结合

耗时计算任务放线程池，IO 操作必须回事件循环线程

为什么需要线程池：libevent 的就绪事件回调默认在事件循环线程中执行，如果回调里有耗时操作（比如复杂计算、磁盘 IO、数据库查询），会阻塞整个事件循环，导致其他就绪事件无法及时处理，性能急剧下降；

可以用线程池处理什么：耗时的纯计算任务、非 libevent 的阻塞操作（比如磁盘 IO、数据库查询）；

绝对不能用线程池处理什么：libevent 的 IO 操作（比如bufferevent_write/bufferevent_read）—— 这些操作必须在事件循环线程中执行，否则会出现竞态条件；

线程池处理完后怎么通知事件循环：通过线程安全的event_add() 或 socketpair 唤醒机制，把 “任务完成” 的消息传回事件循环线程，在事件循环线程中处理后续的 IO 操作。

具体流程

初始化阶段：

  • 启用 libevent 线程安全（evthread_use_pthreads()）；
  • 创建event_base；
  • 创建一对socketpair（或使用 libevent 内置的th_notify_fd，但推荐自己创建，更灵活）；
  • 将socketpair[0]（读端）注册到event_base，监听可读事件；
  • 初始化线程池（比如用 C++11 的std::thread + 任务队列，或成熟的线程池库）。

就绪事件回调阶段（事件循环线程）：

  • 收到就绪事件（比如 socket 有数据可读）；
  • 读取数据，把 “数据 + 后续处理逻辑” 封装成一个任务；
  • 把任务丢到线程池的任务队列；
  • 立刻返回，不阻塞事件循环。

线程池处理阶段（线程池线程）：

  • 从任务队列取任务；
  • 执行耗时操作（比如复杂计算、数据库查询）；
  • 处理完成后，向socketpair[1]（写端）写入 1 字节的 “完成标记”（或直接调用event_add()添加一个新事件，event_add()是线程安全的）。

任务完成通知阶段（事件循环线程）：

  • socketpair[0]可读，触发回调；
  • 从回调中获取线程池的处理结果；
  • 在事件循环线程中执行后续的 IO 操作（比如bufferevent_write()把结果写回客户端）。

注意点

必须启用 libevent 线程安全：在创建event_base之前调用evthread_use_pthreads()（Linux），否则多线程调用event_add()会崩溃；

IO 操作必须回事件循环线程：bufferevent_write()/bufferevent_read()/event_add()（除了这个是线程安全的）等 libevent IO 操作，绝对不能在线程池线程中直接调用—— 必须通过 socketpair 或event_add()把消息传回事件循环线程，在事件循环线程中执行；

event_add()是线程安全的：这是 libevent 少数几个可以在其他线程中直接调用的函数之一，线程池处理完任务后，可以直接在池线程中调用event_add()添加一个新的 “任务完成事件”，唤醒事件循环；

注意数据的线程安全：线程池和事件循环线程之间传递的数据，必须保证线程安全（比如用std::shared_ptr管理，或用无锁队列传递）。

总结

核心原则是「耗时计算放线程池，IO 操作回事件循环」：

  1. 为什么需要线程池：libevent 的就绪事件回调默认在事件循环线程执行，如果回调有耗时操作（比如复杂计算、数据库查询），会阻塞整个事件循环，导致其他事件无法及时处理。
  2. 具体实现方案：
    • 初始化时启用 libevent 线程安全，创建 socketpair，把读端注册到 event_base；
    • 就绪事件回调中，把数据和耗时逻辑封装成任务，丢到线程池，立刻返回；
    • 线程池处理完任务后，向 socketpair 写端写入数据，或直接调用线程安全的event_add()，唤醒事件循环；
    • socketpair 读端的回调中，在事件循环线程里执行后续的 IO 操作（比如bufferevent_write）。
  3. 核心注意事项：
    • 必须在创建 event_base 之前启用线程安全；
    • libevent 的 IO 操作（除了event_add）绝对不能在线程池线程中直接调用，必须回事件循环线程；
    • 注意线程间数据传递的线程安全。
  4. 最佳实践：推荐一个线程一个 event_base，线程池处理纯计算任务，IO 操作全在事件循环线程，通过 socketpair 或event_add通信，性能最好。
